1. BIOS Setup (System Setup) Press F2

PowerEdge 서버 부팅 시 F2로 진입하는 System Setup의 System BIOS 메뉴. 실제로는 "BIOS"라기보다 UEFI 펌웨어이며, 하드웨어 초기 동작 방식(전원관리, 부트 방식, 메모리/CPU 동작모드, 보안 정책)을 정의한다. 여기서 바꾼 값은 OS가 아예 인식하지 못하는 하드웨어 레벨 동작을 결정하므로, 잘못 건드리면 OS 설치 자체가 안 되거나 성능이 크게 흔들릴 수 있다. 아래 소분류 탭을 눌러 항목별로 확인하자.

동작 원리 한 줄 요약: BIOS 설정은 부팅 초기 POST 단계에서 하드웨어 초기화 방식을 결정하고, 이후 OS/드라이버는 그 위에서 동작한다. 즉 "그릇의 모양"을 정하는 단계 — RAID 모드, NUMA 노출 여부, 가상화 지원 여부처럼 OS가 나중에 바꿀 수 없는 항목이 많다.

1-3. Processor Settings

17G17세대부터 달라진 점: PowerEdge 17G는 Intel Xeon 6(Sierra Forest=E-core 중심 고밀도, Granite Rapids=P-core 중심 고성능) 또는 AMD EPYC 9005 "Turin"을 탑재하며, PCIe 5.0을 기본 지원한다. 코어 수가 크게 늘면서(E-core 모델은 코어 100개 이상도 가능) x2APIC·NUMA 설계의 중요성이 이전 세대보다 더 커졌다.

1-5. Boot / Network Settings

설정명설명 / 동작 방식✅ 적용(Enable/선택) 시 →⬜ 미적용(Disable·다른 값) 시 →실무 활용
Boot Mode (BIOS/UEFI)부팅 펌웨어 인터페이스 방식.UEFI로 두면 → GPT 파티션(2TB 이상 부트 디스크), Secure Boot, 빠른 부팅을 지원 — 최신 OS의 표준 선택.레거시 BIOS로 두면 → MBR 기반 구형 OS/옵션롬과의 호환성은 좋아지지만 2TB 초과 부트 디스크·Secure Boot는 사용할 수 없음.매우높음
Boot Sequence Retry부팅 가능한 디바이스를 찾지 못했을 때 자동 재시도 여부.Enabled면 → 일시적인 디스크 인식 실패에도 자동으로 재시도해 무인 서버가 스스로 복구 부팅.Disabled면 → 부팅 디바이스를 못 찾으면 그대로 멈춰서 사람이 콘솔로 개입해야 함.높음
Hard-Disk Failover지정한 1번 부트 디스크가 실패하면 목록의 다음 디스크로 자동 전환.Enabled면 → 1번 부트 디스크가 죽어도 목록의 다음 디스크로 자동 부팅되어 물리적으로 분리된 부트 디스크 이중화가 완성.Disabled(기본)면 → 지정된 디스크가 실패하면 다른 디스크가 멀쩡해도 자동으로 넘어가지 않고 부팅이 멈춤.중간
Generic USB Boot표준 USB 대용량저장장치를 부팅 디바이스로 인식할지 여부.Enabled면 → USB 설치 미디어·복구 도구로 즉시 부팅 가능.Disabled면 → USB로 부팅 자체가 안 돼 유지보수 작업에는 불편하지만, USB 부팅을 이용한 무단 시스템 접근을 막는 보안 효과가 있음.중간
Network Settings — PXE Device / UEFI Network Stack네트워크 인터페이스를 통한 PXE 부팅 활성화 여부와 우선순위.Enabled면 → 네트워크로 OS 이미지를 받아 부팅 가능 — 킥스타트/WDS/Ansible+PXE 같은 원격 자동 설치 파이프라인의 전제조건.Disabled면 → 네트워크 부팅이 불가능해 반드시 로컬 미디어(USB/ISO)로만 설치해야 함.높음

1-6. Integrated Devices / Serial Communication

SR-IOV Global Enable물리 NIC 1개를 다수의 가상 기능(VF)으로 분할해 VM이 하드웨어에 직접 접근하도록 지원.Enabled면 → VM이 가상 스위치를 거치지 않고 NIC에 직접 접근해 네트워크 처리량·지연시간이 크게 개선.Disabled(기본)면 → 모든 VM 트래픽이 하이퍼바이저 가상 스위치를 거쳐야 해 오버헤드가 생기지만 구성은 더 단순.중간

User Accessible USB Ports (Front/Rear/Internal 개별 제어)물리 USB 포트를 위치별로 개별 활성화/비활성화.Enabled(기본)면 → 해당 포트로 USB 장치를 자유롭게 연결 가능.Disabled면 → 그 포트로는 USB 인식 자체가 안 돼, 데이터 반출·악성코드 유입 등 물리 보안 위험이 줄어듦.중간
OS Watchdog TimerOS가 일정 시간 응답이 없으면 iDRAC이 이를 감지해 자동으로 재부팅시키는 하드웨어 워치독.Enabled면 → OS 행(hang) 상태가 방치되지 않고 iDRAC이 자동으로 강제 재부팅.Disabled(기본)면 → OS가 멈춰도 아무도 개입하지 않는 한 그 상태 그대로 방치됨.중간
Memory Mapped I/O above 4GB대용량 메모리를 요구하는 PCIe 디바이스(GPU 등)를 위해 4GB 이상 주소공간에 MMIO를 매핑할 수 있게 허용.Enabled면 → 대용량 VRAM GPU나 NVMe 다수 장착처럼 큰 BAR 공간이 필요한 카드가 정상 인식됨.Disabled(기본)면 → 그런 카드가 아예 인식되지 않거나 제한된 기능으로만 동작할 수 있음.중간
Slot Disablement특정 PCIe 슬롯을 개별적으로 비활성화.비활성화하면 → 문제 있는 확장카드를 물리적으로 뽑지 않고도 부팅 과정에서 논리적으로 배제해 부팅 안정성 확보.활성화(기본) 상태면 → 해당 슬롯의 카드가 그대로 열거되어, 카드 자체에 결함이 있으면 부팅 지연/실패의 원인이 될 수 있음.낮음
Serial Communication / Redirection After Boot시리얼 콘솔 포트 사용 여부와, 부팅 완료 후에도 시리얼 리다이렉션(콘솔 출력)을 유지할지 설정.Enabled(Redirection After Boot 포함)면 → OS 부팅 이후에도 시리얼 콘솔로 텍스트 출력을 계속 받아볼 수 있어, 터미널 서버를 통한 대량 서버 관리에 유용.Disabled면 → 시리얼 포트가 관여하지 않아 일반적인 로컬/원격(iDRAC) 콘솔 위주로만 관리 가능.낮음

1-7. System Profile Settings (Power Management)

면접에서 자주 나오는 항목. "성능 vs 전력" 트레이드오프를 이해하는지 확인하는 단골 질문.

프로필설명 / 동작 방식실무 활용언제 사용하는가
PerformanceTurbo Boost 최대 활용, C-States 최소화, 항상 최고 클럭 유지 지향. 전력 소모와 발열이 가장 크다.높음지연시간에 극도로 민감한 워크로드(고빈도 트레이딩, 실시간 렌더링/스트리밍, 벤치마크).
Performance Per Watt (DAPC) — 기본값DAPC(Dell Active Power Controller)가 BIOS 펌웨어 레벨에서 능동적으로 성능과 전력을 실시간 조율. 일반적으로 최대 성능 대비 3~4% 수준의 전력만 절감하면서 성능 저하는 거의 없음.매우높음대부분의 일반 운영 서버(웹, 애플리케이션, 파일서버)에 적합한 기본값. 특별한 이유가 없다면 유지.
Performance Per Watt (OS) / DBPM전력 관리 결정권을 BIOS가 아닌 OS의 ACPI 기반 전원 정책(cpufreq governor 등)에 위임.중간OS 레벨에서 세밀한 전력 정책(예: tuned 프로필)을 직접 제어하고 싶을 때.
CustomTurbo Boost, C1E, C-States, Monitor/Mwait, 메모리 주파수, Uncore 주파수 등 개별 옵션을 수동으로 세팅. Custom 선택 시에만 하위 항목이 활성화됨.중간Dell 공식 튜닝 가이드/벤치마크 기준에 맞춰 세밀하게 최적화해야 하는 HPC·DB 전용 서버.
Workstation Performance단일 스레드 반응성에 가중치를 둔 프로필.낮음서버보다는 워크스테이션 용도로 쓰이는 PowerEdge에서 CAD/시뮬레이션 작업 시.
Custom을 선택하면 무슨 일이 생기나: Performance/DAPC 등 사전 정의 프로필을 고르면 BIOS가 아래 모든 세부 항목을 알아서 채워 넣고 화면에서 회색 처리(수정 불가)한다. System Profile을 Custom으로 바꾸는 순간에만 이 세부 항목들이 하나하나 손으로 조정 가능한 상태로 풀린다 — 즉 Custom은 "프리셋을 버리고 내가 직접 하나씩 맞추겠다"는 선언이다. Dell 공식 BIOS/UEFI Reference Guide 기준 세부 항목은 다음과 같으며, 각 항목마다 켰을 때/껐을 때(또는 다른 값을 골랐을 때) 실제로 무엇이 달라지는지 대비해서 정리했다.

1-7-1. Custom Mode Sub-Options (Full List)

세부 옵션설명 / 동작 방식✅ 적용(Enable/선택) 시 →⬜ 미적용(Disable·다른 값) 시 →실무 활용
CPU Power ManagementCPU 전력 관리 방식 자체를 선택하는 최상위 스위치. 기본값은 System DBPM(Demand-Based Power Management, DAPC) — 부하에 따라 BIOS가 능동적으로 전압/클럭을 조율.System DBPM(기본)이면 → BIOS가 부하를 실시간 감지해 전압·클럭을 자동 조율, 별도 OS 튜닝 없이도 균형 잡힌 성능/전력 확보.OS Control로 바꾸면 → 결정권이 OS의 ACPI 정책(cpufreq governor 등)으로 완전히 넘어가 BIOS는 관여하지 않음 — OS 튜닝을 안 해두면 오히려 전력관리가 허술해질 수 있음.중간
Memory Frequency메모리 클럭 속도를 Maximum Performance(기본) / Maximum Reliability / 특정 고정 속도 중에서 선택.Maximum Performance면 → 메모리가 지원 가능한 최고 클럭으로 동작해 대역폭 최대화.Maximum Reliability나 저속 고정으로 낮추면 → 대역폭은 줄지만 신호 마진이 커져 메모리 관련 오류 가능성이 감소.중간
Turbo BoostCustom 모드에서 개별적으로 Enabled/Disabled 선택 가능(기본 Enabled).Enabled면 → 부하가 몰리는 코어의 클럭이 정격 이상으로 순간 상승해 체감 성능 향상, 대신 전력·발열 변동폭이 커짐.Disabled면 → 클럭이 항상 정격에 고정돼 성능은 낮아지지만 서버마다 클럭 편차가 없어 벤치마크·클러스터 성능 표준화에 유리.높음
C1E프로세서가 유휴 상태일 때 최소 성능 상태로 전환할지 여부(기본 Enabled).Enabled면 → 유휴 시 클럭을 낮춰 전력을 절감하지만, 다시 부하가 걸릴 때 클럭을 올리는 데 짧은 지연(latency)이 발생.Disabled면 → 유휴 상태에서도 클럭을 낮추지 않아 전력은 더 쓰지만 응답속도가 항상 일정하게 유지됨.높음
C States더 깊은 절전 상태(C1E보다 심화) 진입 허용 여부. Performance 프로필 기준 기본값은 Disabled, 그 외 프로필은 Enabled.Enabled면 → 유휴 시 훨씬 깊은 절전모드까지 들어가 전력 절감 효과가 커지지만, 복귀 지연(wake-up latency)도 그만큼 커짐.Disabled면 → 코어가 항상 활성 상태(C0)를 유지해 응답속도는 일정해지지만 유휴 시에도 전력을 계속 소모.높음
Write Data CRC메모리 쓰기 시 CRC(순환중복검사) 값을 추가로 계산해 데이터 무결성을 검증할지 여부(기본 Disabled).Enabled면 → 메모리 쓰기마다 무결성 체크섬을 함께 계산·저장해 데이터 손상을 더 잘 잡아내지만, 그만큼 쓰기 성능이 소폭 저하.Disabled(기본)면 → 성능 손실은 없지만 이 추가 체크섬 검증 없이 동작.낮음
Memory Patrol Scrub백그라운드 메모리 스캔 빈도를 Standard(기본)/Extended 등으로 조정.Extended로 올리면 → 메모리를 더 자주/철저히 스캔해 정정 가능한 오류를 더 빨리 잡아내지만 스캔 자체의 오버헤드가 소폭 증가.Standard(기본)면 → 오버헤드는 최소지만 오류 발견까지 다소 시간이 걸릴 수 있음.낮음

❓ BIOS Interview Questions

C-States를 끄면 성능이 왜 좋아지나?
CPU가 유휴 상태에서 저전력 상태(C1~C6)로 들어갔다가 다시 활성 상태로 복귀하는 데 마이크로초 단위 지연이 발생한다. 초저지연이 중요한 서비스는 이 wake-up latency 자체가 문제가 되므로 C-States를 꺼서 항상 활성 상태(C0)를 유지시켜 응답속도를 일정하게 만든다. 대신 유휴 시에도 전력을 계속 소모한다.
가상화 호스트 구축 전 BIOS에서 가장 먼저 확인할 항목은?
Virtualization Technology(VT-x/AMD-V) Enabled 여부와, 필요 시 IOMMU/VT-d(디바이스 패스스루), SR-IOV Global Enable 여부. 이게 꺼져 있으면 ESXi/Hyper-V/KVM 설치 자체가 실패하거나 중첩 가상화가 동작하지 않는다.
AC Power Recovery 설정을 놓쳐서 생기는 실제 장애 사례는?
데이터센터 정전 후 전원이 복구됐는데 AC Power Recovery가 기본값(Off 등)으로 남아있으면 서버가 자동으로 켜지지 않는다. 무인 원격지 서버라면 담당자가 직접 콘솔로 전원 버튼을 눌러야 하는 상황이 발생 — 실무에서 반드시 On/Last State로 바꿔두는 대표적 체크 항목이다.
온보드 SATA를 AHCI로 둘지 RAID로 둘지 기준은?
별도 PERC 하드웨어 RAID 카드가 있고 그 카드가 디스크를 관리한다면 온보드 SATA는 AHCI(또는 Off)로 둔다. PERC 없이 온보드 컨트롤러만으로 소프트웨어 RAID를 구성해야 한다면 SATA를 RAID 모드로 바꾼다.
GPU 패스스루를 위해 BIOS에서 확인해야 할 3가지는?
① VT-d/IOMMU Enabled, ② Memory Mapped I/O above 4GB Enabled(대용량 VRAM 인식용), ③ SR-IOV Global Enable(가상함수 분할이 필요한 경우). 이 중 하나라도 빠지면 VM에 GPU를 정상적으로 패스스루할 수 없다.
Node Interleaving과 NUMA는 무슨 관계인가?
Node Interleaving이 Disabled(기본값)일 때 NUMA 토폴로지가 그대로 OS에 노출된다. Enabled로 바꾸면 전체 메모리를 인터리빙해 NUMA 구분 자체를 없애버리므로, NUMA를 인식 못 하는 구형 소프트웨어와의 호환성은 좋아지지만 최신 NUMA-aware 애플리케이션에는 오히려 손해다.

2. iDRAC Remote Management (Integrated Dell Remote Access Controller)

서버 메인보드에 내장된 독립 관리 프로세서(BMC)로, 서버 전원이 꺼져 있거나 OS가 다운돼도 별도 네트워크 경로로 원격 접속·전원제어·콘솔접근이 가능하다. 실무에서 "원격지 서버에 물리적으로 가지 않고" 문제를 해결하는 핵심 도구이며, 인프라 엔지니어 면접에서 매우 자주 다뤄지는 주제.

동작 원리: iDRAC은 호스트 OS와 완전히 분리된 자체 CPU·메모리·네트워크 인터페이스를 갖는 "서버 안의 작은 컴퓨터"다. 메인 서버 전원이 내려가 있어도 iDRAC은 보조전원으로 계속 동작하며, Redfish/RACADM/웹UI/IPMI를 통해 하드웨어를 직접 제어한다.
17G17세대부터 달라진 점 — iDRAC10: 17G부터는 새 세대 BMC인 iDRAC10이 탑재된다. TPM·Root-of-Trust·원격 증명(Attestation)·전송 중 암호화를 통합해 "AI 기반" 관리와 Zero Trust 보안 모델을 지원하도록 재설계됐다. UI/API는 iDRAC9와 크게 다르지 않지만 내부 보안 아키텍처와 부팅 시 무결성 검증 방식이 강화됐다.

2-4. RACADM Usage — From Basics to Practice

RACADM은 iDRAC을 GUI 없이 CLI로 조회·설정하는 Dell 전용 도구다. "RAC(Remote Access Controller) ADMin"의 줄임말 — 즉 iDRAC 자체를 다루는 도구이지, OME에 서버를 등록하는 도구가 아니다. (OME 등록/검색 방식은 OME 탭 참고)

3가지 접속 방식
① Local RACADM — 대상 서버의 OS에 iDRAC Service Module(iSM)이 설치돼 있으면 그 OS 안에서 바로 racadm 명령을 실행. IP/계정 입력 없이 로컬 iDRAC을 바로 제어.
② Remote RACADM — 관리자의 PC/배포서버에서 iDRAC의 IP로 원격 접속. 대상 서버 OS 상태와 무관하게 동작하며 스크립트 자동화에 가장 많이 쓰임: racadm -r <iDRAC_IP> -u root -p <비밀번호> --nocertwarn <명령>
③ Firmware RACADM — iDRAC에 SSH/Telnet으로 직접 로그인한 뒤 IP·계정 입력 없이 바로 명령 실행 (이미 인증된 세션이므로).
목적예시 명령설명
시스템 정보 조회racadm getsysinfo서비스 태그, BIOS/펌웨어 버전, 전원 상태 등 시스템 기본 정보를 한 번에 출력. 원격 진단의 첫 명령.
설정값 조회 (get)racadm get iDRAC.Network / racadm get BIOS.SysProfileSettingsFQDD(그룹) 단위로 iDRAC/BIOS 설정 그룹을 통째로 조회. 그룹명.속성명 구조로 계층화돼 있음.
설정값 변경 (set)racadm set iDRAC.NIC.DHCPEnable 0get으로 확인한 속성을 개별적으로 변경. 다수 서버에 동일 설정을 스크립트로 일괄 적용할 때 핵심.
전원 제어racadm serveraction powercycle / hardreset / powerdownOS가 완전히 멈춰 응답이 없을 때 원격으로 강제 전원 재기동. 물리 출동 없이 가능한 최후의 복구 수단.
하드웨어 로그 확인racadm gettracelog / racadm getraclogiDRAC 자체 트레이스 로그와 RAC 로그를 확인. SEL과 함께 하드웨어 장애의 근본 원인을 추적하는 핵심 로그.
RAID/디스크 조회racadm storage get pdisks / racadm storage get vdisks -o -p layout,status물리 디스크(pdisks)와 가상 디스크(vdisks)의 상태·레이아웃을 CLI로 조회. GUI 접속 없이 RAID 상태를 스크립트로 점검할 때.
iDRAC 초기화racadm racreset (iDRAC만 재시작) / racadm racresetcfg (공장 초기화)racreset은 iDRAC 프로세스만 재시작(서버 본체는 영향 없음), racresetcfg는 사용자가 설정한 모든 값을 삭제하고 기본값으로 초기화 — 되돌릴 수 없으므로 신중히 사용.
설정 통째로 복제 (SCP)racadm get -f config.xml -t xml (내보내기) / racadm set -f config.xml -t xml (가져오기)Server Configuration Profile(SCP) 기능으로 BIOS/iDRAC/RAID 설정을 XML/JSON로 통째로 내보내고, 동일 모델의 다른 서버에 그대로 적용. 대량 서버 표준화 배포의 핵심 기법.
💡 실습 팁: 처음이라면 GUI로 값을 먼저 바꿔보고 racadm get으로 해당 속성명이 무엇으로 바뀌는지 확인한 뒤, 그 속성명을 set 스크립트에 그대로 재사용하는 방식으로 익히는 게 가장 빠르다.

❓ iDRAC Interview Questions

IPMI와 Redfish의 실무 차이는?
IPMI는 바이너리 기반 구형 표준으로 기능이 제한적(전원제어, 센서 조회 위주)이고 보안 취약점 이슈도 있었다. Redfish는 REST/JSON 기반으로 BIOS, RAID, 펌웨어, 인벤토리까지 표준화된 스키마로 다룰 수 있어 최근 자동화 도구(Ansible, Terraform)들이 Redfish를 우선 지원한다. 신규 구축 환경이라면 Redfish 우선 검토가 실무 표준.
서버가 켜지지도 않고 OS도 응답이 없을 때 첫 확인 지점은?
iDRAC의 SEL 로그와 Virtual Console. iDRAC은 서버 메인 전원과 무관하게 동작하므로, 전원 상태·부팅 시퀀스·POST 에러 코드를 원격에서 그대로 확인할 수 있다. 물리 출동 전 반드시 거치는 1차 진단 단계.
관리망(iDRAC)을 운영망과 분리하는 이유는?
iDRAC은 OS 레벨 보안 정책과 무관하게 하드웨어를 완전히 제어할 수 있는 강력한 권한을 가지므로, 공격자가 접근하면 서버를 통째로 장악할 수 있다. 별도 VLAN/물리망으로 격리해 외부 노출을 최소화하는 것이 보안 표준.
RACADM과 OME의 역할 차이는?
RACADM은 서버 1대의 iDRAC을 CLI로 직접 조회·설정하는 도구이고, OME는 수십~수천 대의 iDRAC을 중앙에서 검색·모니터링·펌웨어 업데이트하는 관제탑(콘솔) 역할을 한다. racadm으로 서버를 OME에 "수동 등록"하는 절차는 없다 — 등록(Discovery)은 항상 OME 콘솔 쪽에서 IP 대역과 계정으로 검색해 이뤄진다.
Remote RACADM과 Local RACADM의 차이는?
Local RACADM은 대상 서버의 OS 안에서 실행하는 방식(iDRAC Service Module 필요, IP/계정 불필요), Remote RACADM은 관리자의 PC에서 iDRAC IP로 원격 접속하는 방식(-r/-u/-p 옵션 필요)이다. 대량 서버를 스크립트로 관리할 때는 대상 서버 OS 상태와 무관하게 동작하는 Remote RACADM을 주로 사용한다.

3. OpenManage Enterprise (OME) Manage many servers, one screen

iDRAC이 "서버 1대의 관제실"이라면, OME(OpenManage Enterprise)는 그 관제실 수십~수천 개를 한 화면에서 통합 관리하는 중앙 콘솔이다. 가상 어플라이언스(VM) 형태로 배포하며, 다수의 iDRAC을 검색(Discovery)해 인벤토리·모니터링·펌웨어 업데이트·구성 템플릿을 일괄 적용한다.

3-1. Overview — What OME Does

항목설명실무 활용언제 사용하는가
중앙 인벤토리/모니터링등록된 모든 서버의 하드웨어 구성, 상태, 경고를 하나의 대시보드에서 확인. 서버 1대씩 iDRAC에 로그인할 필요가 없어짐.매우높음서버가 10대를 넘어가는 순간부터 iDRAC 개별 접속 관리는 비효율적 — OME 도입이 사실상 표준.
OME vs iDRAC vs OME-ModulariDRAC=서버 1대 단위 관제, OME=랙 서버 다수를 중앙에서 관리하는 콘솔, OME-Modular=MX7000 등 모듈러 섀시 전용 콘솔(섀시에 내장).높음면접에서 셋의 역할 차이를 구분해서 설명할 수 있는지 자주 확인.
라이선스 등급 (기본/Advanced/Advanced+)기본은 인벤토리·모니터링 위주, Advanced부터 구성 템플릿·펌웨어 자동 업데이트, Advanced+에서 내부 iDRAC 자격증명 관리 등 고급 기능 제공.중간대규모 자동화(구성 표준화, 자동 자격증명 로테이션)가 필요한 조직은 Advanced 이상 라이선스 검토.

3-2. Installation / OME Self-Update

단계설명실무 활용언제/왜 하는가
① 배포 방식 선택OME는 가상 어플라이언스(사전 구성된 VM 이미지)로 배포한다. VMware ESXi(OVF/OVA), Microsoft Hyper-V(VHD), KVM 등 주요 하이퍼바이저를 지원.매우높음물리 서버에 직접 설치하는 게 아니라 반드시 가상머신으로 배포 — 사내 가상화 인프라(ESXi 등)가 먼저 준비돼 있어야 한다.
② 리소스 사이징관리 대상 서버 수(예: 수백 대 vs 수천 대)에 따라 vCPU/메모리/디스크 용량을 다르게 할당해야 한다.높음배포 전에 관리 대상 규모를 먼저 파악해 어플라이언스 사이즈를 결정 — 너무 작게 잡으면 나중에 성능 저하.
③ 초기 설정 마법사VM 최초 부팅 후 콘솔/웹UI로 네트워크(IP), 관리자 계정, 시간대, 조직 정보를 설정하는 초기 구성 마법사를 거친다. 이때 "Enable Internal iDRAC credential management" 여부도 선택.매우높음모든 신규 OME 배포에서 반드시 거치는 최초 1회 설정.
④ 웹 콘솔 접속설정한 IP로 HTTPS(443)에 접속해 웹 기반 관리 콘솔을 사용. 이후 모든 작업(검색, 업데이트, 그룹 관리)은 이 웹UI에서 이뤄진다.매우높음설치가 끝나면 바로 이 화면에서 서버 검색(Discovery)으로 넘어간다.
⑤ OME 자체 버전 업데이트OME 어플라이언스 자체의 소프트웨어 버전은 콘솔 내장 "Console Update" 기능(온라인/오프라인 업데이트 파일 업로드)으로 마이너 업그레이드하거나, 메이저 버전은 새 OVA를 배포하고 기존 설정을 백업/복원(Backup & Restore Appliance)하는 방식으로 이전한다.중간보안 패치나 신규 기능(예: 신규 서버 세대 지원) 반영을 위해 정기적으로 콘솔 업데이트 확인이 필요.
⑥ 백업/복원어플라이언스 설정, 인벤토리, 자격증명을 정기적으로 백업해두고, 장애나 마이그레이션 시 복원.중간OME 자체가 다운되면 등록된 모든 서버의 중앙관리 기능이 함께 마비되므로, 어플라이언스 자체의 백업/이중화 계획도 함께 세워야 한다.

3-3. Server Discovery / Registration

racadm으로 등록하는 게 아니다: OME에 서버가 "등록"되는 과정은 racadm 명령을 실행하는 게 아니라, OME 콘솔에서 능동적으로 검색(Discovery Job)하는 방식이다. 검색 대상 IP 범위와 iDRAC 관리자 계정을 OME에 입력하면, OME가 각 IP로 WSMAN/Redfish/SNMP/IPMI 프로토콜을 통해 접속을 시도해 서버를 찾아내고 인벤토리를 수집한다.
단계설명실무 활용언제 사용하는가
Discovery Job 생성Configuration(또는 Devices) > Discovery 메뉴에서 대상 IP 범위(또는 호스트명 목록)와 사용할 프로토콜(주로 Redfish/WSMAN), iDRAC 계정 정보를 입력해 검색 작업을 생성.매우높음새 서버를 랙업한 직후 가장 먼저 하는 작업. 스케줄을 걸어 주기적으로 재검색하면 신규 서버를 자동으로 편입시킬 수도 있다.
내부 iDRAC 자격증명 관리 (Internal iDRAC Credential Management)Advanced 이상 라이선스에서, 최초 검색 성공 후 OME가 각 iDRAC에 "OME_[GUID]" 형식의 전용 서비스 계정을 자동 생성해 이후 통신에 사용. 이렇게 하면 사람이 입력한 최초 계정 정보 없이도 OME가 자체 계정으로 지속 통신 가능.중간다수의 서버 계정 암호를 일일이 관리하기 번거로운 대규모 환경에서, 계정 로테이션까지 OME에 맡기고 싶을 때(초기 구성 마법사에서만 활성화 가능).
비밀번호 정책 주의사항iDRAC에 Password Strength Policy(복잡도 정책)가 걸려 있으면, OME가 자동 생성한 서비스 계정 비밀번호가 정책을 만족하지 못해 계정이 비활성화될 수 있다.중간내부 자격증명 관리 기능을 켜기 전에 대상 iDRAC들의 비밀번호 정책을 먼저 점검해야 하는 실무 체크포인트.
등록 후 상태 확인검색이 끝나면 Devices 목록에 서버가 나타나고, 정상/경고/심각 등 상태 아이콘으로 헬스 상태를 표시.높음검색 직후 반드시 상태를 확인해 인증 실패·프로토콜 미지원 등으로 "빠진" 서버가 없는지 점검.
racadm의 실제 역할racadm은 OME 등록 자체와는 무관하지만, 검색 이전에 iDRAC 네트워크(IP)를 설정하거나, SNMP·계정 정책 같은 "검색이 잘 되기 위한 사전 조건"을 개별 서버마다 스크립트로 미리 맞춰두는 보조 역할로 쓰인다.중간수백 대의 신규 서버를 랙업할 때, racadm 스크립트로 iDRAC IP/계정을 먼저 표준화한 뒤 OME Discovery를 돌리면 검색 성공률이 올라간다.

3-4. Firmware Update (Baseline / Catalog)

단계설명실무 활용언제 사용하는가
카탈로그 소스 선택Dell 온라인 카탈로그(인터넷 연결 필요)를 쓸지, 사내 리포지토리에 미리 받아둔 로컬 카탈로그를 쓸지 선택.높음인터넷이 차단된 폐쇄망(에어갭) 환경에서는 로컬 카탈로그를 주기적으로 갱신해 사용.
Baseline 생성Configuration > Firmware > Create Baseline에서 카탈로그와 대상 장치 그룹을 지정해 "이 그룹은 이 버전을 기준으로 삼는다"는 기준선을 만든다.매우높음동일 모델 서버군의 펌웨어 버전을 표준화하고 싶을 때 — 신규 서버 편입 시에도 같은 Baseline에 묶으면 자동으로 준수 여부가 체크된다.
Compliance Report 확인Baseline과 실제 장치의 현재 펌웨어 버전을 비교해 Upgrade 필요/Critical/Compliant 등의 상태로 보여준다.매우높음패치 관리 정기 점검 시 "어떤 서버가 뒤처져 있는지"를 한눈에 파악하는 용도.
Remediation (Make Compliant)업데이트가 필요한 장치를 선택해 "Make Compliant"를 실행하면 실제 펌웨어 업데이트 작업이 시작된다 — Baseline을 만든다고 자동으로 업데이트되는 것이 아니라 반드시 이 단계를 수동으로 실행해야 한다.매우높음유지보수 창에 맞춰 업데이트를 실제로 적용할 때. 즉시 적용(Update Now) 또는 예약 적용 중 선택 가능.
재부팅 필요 여부BIOS/펌웨어 대부분은 적용을 완료하려면 서버 재부팅이 필요하며, OME가 이를 안내하고 예약된 시간에 재부팅까지 자동 수행할 수 있다.높음운영 중인 서비스에 영향이 없도록 재부팅 시점을 새벽 시간대 등으로 예약할 때.

3-5. Monitoring / Groups / Alerts

항목설명실무 활용언제 사용하는가
정적/동적/쿼리 그룹 (Static/Dynamic/Query Group)서버를 수동으로 묶는 정적 그룹, 특정 조건(모델, 위치 등)에 맞으면 자동으로 편입되는 동적 그룹, 실시간 쿼리 조건으로 필터링하는 쿼리 그룹을 지원.높음부서별/랙별/역할별로 서버를 그룹핑해 Baseline이나 알림 정책을 그룹 단위로 일괄 적용하고 싶을 때.
알림 정책 (Alert Policy)특정 심각도·카테고리의 이벤트가 발생하면 이메일, SNMP trap, Syslog 등으로 전달하도록 정책을 정의.높음수백 대 서버의 경고를 개별적으로 확인하는 대신, 조건에 맞는 이벤트만 담당자에게 자동으로 전달되게 할 때.
구성 템플릿 / 컴플라이언스특정 서버의 BIOS/iDRAC/RAID 설정을 템플릿으로 저장해 다른 서버에 동일하게 배포하고, 이후 설정이 템플릿과 어긋나는지(configuration drift) 정기 검사.높음표준화된 설정을 대량 서버에 일괄 적용하고, 이후 누군가 임의로 설정을 바꾸는 "설정 드리프트"를 감지하고 싶을 때.
리포트인벤토리, 컴플라이언스, 보증기간 만료 임박 등 다양한 사전 정의된/커스텀 리포트를 생성.중간정기 자산관리 보고, 경영진 보고용 데이터 추출이 필요할 때.

❓ OME Interview Questions

OME에 서버를 어떻게 "등록"하나?
racadm으로 서버 쪽에서 등록 명령을 실행하는 방식이 아니라, OME 콘솔에서 Discovery Job을 만들어 대상 IP 범위와 iDRAC 계정 정보로 능동적으로 검색하는 방식이다. OME가 각 iDRAC에 접속을 시도해 성공하면 자동으로 인벤토리에 등록된다.
Baseline을 만들면 자동으로 펌웨어가 업데이트되나?
아니다. Baseline은 "기준 버전"을 정의하는 것일 뿐이고, Compliance Report로 현재 상태와 기준을 비교한 뒤 관리자가 직접 "Make Compliant"를 눌러야 실제 업데이트 작업이 시작된다. 즉시 적용할지 예약할지도 그 시점에 선택한다.
iDRAC, OME, OME-Modular의 차이를 설명하면?
iDRAC은 서버 1대에 내장된 개별 BMC, OME는 다수의 랙 서버를 검색·모니터링·업데이트하는 중앙 콘솔(별도 VM), OME-Modular는 MX7000 같은 모듈러 섀시에 내장돼 그 섀시 안의 슬레드들을 관리하는 콘솔이다.
폐쇄망(에어갭) 환경에서 펌웨어 업데이트는 어떻게 하나?
Dell 온라인 카탈로그에 접근할 수 없으므로, 인터넷이 되는 별도 PC에서 카탈로그와 펌웨어 파일을 미리 다운로드해 사내 로컬 리포지토리(파일 공유 등)에 올려두고, OME의 카탈로그 소스를 그 로컬 경로로 지정해 Baseline을 구성한다.

4. PERC RAID Controller (PowerEdge RAID Controller)

Dell 서버의 하드웨어 RAID 카드(PERC H330~H965i 등)는 디스크의 물리적 배치와 캐시 정책을 통해 성능·가용성을 결정한다. OS는 컨트롤러가 만든 가상 디스크(Virtual Disk)만 하나의 논리 드라이브로 인식하므로, 실제 RAID 동작은 OS와 무관하게 컨트롤러 펌웨어 레벨에서 이뤄진다.

17G17세대부터 달라진 점 — PERC12: 17G에서는 PERC12 세대 컨트롤러(H965i Front/Adapter/MX, H965e 등)가 사용된다. PCIe 5.0 연결과 더 높은 IOPS를 지원하며, 캐시 정책(WT/WB/FWB)이나 Patrol Read·Consistency Check 같은 핵심 개념은 이전 세대(PERC11 이하)와 동일하다 — 즉 세대가 바뀌어도 RAID 운용 개념 자체는 그대로 유지된다.

4-1. RAID Levels

RAID 레벨설명 / 동작 방식실무 활용언제 사용하는가
RAID 0 (스트라이핑)데이터를 여러 디스크에 분산 기록해 속도는 최대이나 이중화 없음 — 디스크 1개만 죽어도 전체 데이터 유실.낮음손실돼도 무방한 임시/캐시성 데이터, 순수 성능 벤치마크 용도.
RAID 1 (미러링)동일 데이터를 2개 디스크에 그대로 복제. 디스크 1개 장애까지 허용, 실사용 용량은 50%.매우높음OS 부트 디스크 등 소용량·고가용성이 필요한 볼륨의 기본 구성.
RAID 5패리티 1개 분산 저장, 디스크 1개 장애까지 허용. 용량 효율 좋으나 쓰기 시 패리티 계산 오버헤드(Write Penalty) 존재.높음읽기 위주 워크로드(파일서버, 백업 저장소)의 비용 대비 효율적인 구성.
RAID 6패리티 2개 분산 저장, 디스크 2개까지 동시 장애 허용. RAID5보다 안전하지만 쓰기 오버헤드가 더 큼.높음대용량 디스크(리빌드 시간이 긴)로 구성된 스토리지에서 이중 장애까지 대비할 때.
RAID 10 (1+0)미러링된 쌍을 스트라이핑. 성능과 가용성을 모두 확보하지만 실사용 용량은 50%.매우높음DB(트랜잭션 로그, 데이터파일) 등 읽기/쓰기 성능과 안정성이 동시에 중요한 워크로드의 표준 선택.
RAID 50 / 60RAID5/6 세트를 여러 개 묶어 스트라이핑한 중첩 RAID. 대용량 + 성능 + 부분 이중화를 함께 노림.중간디스크 수가 많은 대용량 스토리지 어레이에서 RAID5/6 단일 세트의 한계를 넘어서고 싶을 때.

4-2. Cache / Read-Write Policy

설정명설명 / 동작 방식실무 활용언제 사용하는가
① Write Through (WT)디스크에 실제로 쓰기가 완료된 후에만 컨트롤러가 호스트에 "완료" 신호를 보냄. 데이터 안정성이 가장 높지만 쓰기 성능은 느림.높음BBU/CacheVault가 아예 없거나 방전 상태일 때 컨트롤러가 안전을 위해 자동으로 강제 전환하는 정책이기도 함. 데이터 무결성이 최우선인 경우 의도적으로 선택.
② Write Back (WB) — 기본값데이터가 컨트롤러 캐시에 들어오는 즉시 "완료" 신호를 보내고 이후 백그라운드로 디스크에 기록. 단, BBU/CacheVault 상태를 실시간으로 감시해 배터리가 없거나 방전되면 컨트롤러가 자동으로 Write Through로 되돌아가는 "안전한" 버전.매우높음대부분의 프로덕션 워크로드의 기본 선택. 성능과 안전성의 균형이 맞는 표준 옵션 — 배터리 상태가 나빠지면 자동으로 WT로 폴백해 데이터를 보호한다.
③ Force Write Back / Always WB (FWB)배터리(BBU/CacheVault)가 없거나 방전 상태여도 상관없이 항상 강제로 Write Back 모드를 유지. WB처럼 배터리 상태에 따라 자동으로 WT 전환되지 않음.낮음배터리 없는 컨트롤러에서도 최대 쓰기 성능이 반드시 필요한 특수 상황(예: UPS로 전원 안정성이 이미 보장된 랩·테스트 환경)에서만 신중히 사용 — 정전 시 캐시 데이터 유실 위험을 감수하는 옵션이라 프로덕션에서는 권장되지 않음.
Read Ahead / Adaptive Read Ahead / No Read Ahead순차 읽기 패턴을 감지해 다음 섹터를 미리 읽어 캐시에 적재. Adaptive는 최근 읽기 패턴이 순차적일 때만 자동으로 Read Ahead를 적용.높음순차 읽기가 많은 워크로드(백업, 미디어 스트리밍)는 Read Ahead, 랜덤 I/O가 많은 OLTP DB는 No Read Ahead가 유리한 경우가 많음.
Disk Cache Policy (물리 디스크 자체 캐시)HDD/SSD 자체 내장 캐시 사용 여부(Enabled/Disabled/Default). 컨트롤러 캐시와는 별개로 디스크 자체 캐시를 켤지 결정.중간BBU로 보호되지 않는 디스크 자체 캐시까지 켜면 정전 시 유실 위험이 커지므로, 데이터 안정성이 중요하면 Disabled 권장.
Stripe Size데이터를 나눠 쓰는 블록 단위 크기(64KB~1MB 등). I/O 패턴에 따라 성능이 달라짐.중간대용량 순차 I/O(백업, 미디어)는 큰 스트라이프, 소용량 랜덤 I/O(DB)는 작은 스트라이프가 유리한 경향.
BBU / CacheVault (배터리·커패시터 백업)정전 시에도 컨트롤러 캐시의 데이터를 잃지 않도록 배터리(BBU) 또는 커패시터+플래시(CacheVault) 방식으로 캐시를 보존. 방전/고장 상태면 컨트롤러가 자동으로 Write Back → Write Through로 강제 전환하는 안전장치가 동작.매우높음Write Back 캐시를 신뢰성 있게 쓰려면 필수 전제조건. 면접에서 "왜 Write Back인데 갑자기 성능이 떨어졌나"류 질문의 핵심 답이 되는 항목(배터리 방전 → 자동 WT 전환).

4-3. Maintenance / Availability Features

설정명설명 / 동작 방식실무 활용언제 사용하는가
Patrol Read유휴 시간에 백그라운드로 전체 물리 디스크를 스캔해 배드섹터를 조기 발견·복구.높음운영 중인 모든 RAID 어레이에서 기본 활성화 권장 — 실 장애 전에 결함 섹터를 미리 잡아냄.
Consistency Check패리티 기반 RAID(5/6 등)에서 실제 데이터와 패리티 값의 일관성을 정기적으로 검증·복구.높음정기 유지보수 스케줄(주1회 등)로 자동 실행 설정, 사일런트 데이터 오류를 예방.
Rebuild Rate디스크 장애 후 교체 디스크로 데이터를 재구성하는 속도(우선순위). 높이면 리빌드는 빨라지지만 운영 I/O 성능에 영향.높음장애 디스크 교체 직후 리빌드 중에도 서비스 영향을 최소화해야 하면 Rate를 낮추고, 빠른 복구가 우선이면 높임.
Hot Spare (Dedicated / Global)장애 발생 시 자동으로 투입되는 예비 디스크. Dedicated는 특정 어레이 전용, Global은 여러 어레이가 공유.높음무인 운영 서버에서 디스크 장애 시 사람이 개입하기 전에 자동 리빌드를 시작시키고 싶을 때.
CacheCade (SSD 캐시)SSD를 HDD 어레이의 2차 캐시(read/write cache)로 사용해 성능을 끌어올리는 기능.낮음HDD 기반 대용량 스토리지에 소량의 SSD만 추가해 핫데이터 접근속도를 높이고 싶을 때.
JBOD 모드디스크를 RAID로 묶지 않고 컨트롤러가 있는 그대로(개별 디스크)를 OS에 그대로 노출.중간ZFS, Ceph, Ceph OSD처럼 소프트웨어 정의 스토리지가 자체적으로 디스크를 직접 관리해야 하는 구성에서 필수.
Foreign Configuration Import다른 서버/컨트롤러에서 사용하던 디스크를 그대로 장착했을 때, 기존 RAID 구성 정보(메타데이터)를 그대로 가져와 볼륨을 복원.중간서버 장애로 디스크를 통째로 다른 새 서버에 옮겨 데이터를 복구해야 하는 상황.
물리 디스크 상태 (Online/Offline/Rebuilding/Failed/Foreign)RAID를 구성하는 각 물리 디스크의 현재 상태를 나타내는 값. Rebuilding은 리빌드 진행 중, Foreign은 다른 컨트롤러 이력이 남아있는 상태.매우높음장애 조치의 가장 기본이 되는 확인 지점 — 디스크 교체 후 상태가 Online으로 정상 복귀했는지 반드시 확인.
Secure Erase디스크 반납/폐기 전 데이터를 복구 불가능하게 완전 삭제.중간디스크 교체/서버 폐기 시 데이터 유출 방지를 위한 보안 절차(컴플라이언스 요구사항).
T10 PI (Protection Information / DIF)디스크에 데이터를 쓸 때마다 무결성 체크섬을 함께 저장해, 조용한 데이터 손상(silent corruption)을 감지.낮음엔터프라이즈 SAS 디스크 기반의 미션크리티컬 스토리지에서 데이터 무결성을 최우선으로 할 때.

❓ RAID Interview Questions

Write Back인데 갑자기 쓰기 성능이 뚝 떨어졌다면?
일반 Write Back(WB)은 BBU/CacheVault 상태를 실시간 감시하다가 배터리가 방전·고장 상태가 되면 데이터 안정성을 위해 자동으로 Write Through로 강제 전환한다. 이 경우 iDRAC/PERC 상태에서 배터리 상태(Learning cycle 포함)를 먼저 확인하는 것이 정석. 참고로 Force Write Back(FWB)으로 설정돼 있었다면 배터리 상태와 무관하게 계속 WB를 유지하므로 이 증상 자체가 나타나지 않는다 — 대신 정전 시 데이터 유실 위험을 그대로 안고 가는 것이다.
RAID5와 RAID10 중 DB 서버에는 어떤 걸 쓰는가?
RAID10이 일반적으로 선호된다. RAID5는 쓰기 시 패리티 계산 때문에 Write Penalty가 커서 랜덤 쓰기가 많은 OLTP 워크로드에 불리하고, 디스크 1개 장애 후 리빌드 중 2차 장애 위험(특히 대용량 디스크)도 크다. RAID10은 용량 효율은 떨어지지만 쓰기 성능과 리빌드 안정성이 더 우수하다.
WT/WB/FWB 3가지 캐시 정책의 차이를 한 문장으로?
WT는 항상 안전하지만 느리고, WB는 빠르면서도 배터리 상태를 감시해 위험할 때 자동으로 안전모드(WT)로 전환하며, FWB는 배터리 상태와 무관하게 항상 빠른 모드를 강제 유지해 위험을 감수한다.

5. Storage Array (PowerVault ME5 / PowerStore, etc.)

서버 내장 RAID(PERC)와 별개로, 여러 서버가 공유하는 외장형 스토리지 어레이(SAN)는 풀·볼륨·티어링·스냅샷 개념으로 관리한다. 인프라 엔지니어 면접에서는 PowerVault ME 시리즈(엔트리~미드레인지 블록스토리지) 기준 개념을 자주 묻는다.

5-1. Pool / Volume / Mapping

설정명설명 / 동작 방식실무 활용언제 사용하는가
디스크 풀 (Pool) / 디스크 그룹여러 물리 디스크를 묶어 만드는 저장 공간의 최소 관리 단위. Virtual Pool(가상화·티어링 지원)과 Linear Pool(고정 매핑, 예측 가능한 성능) 두 방식이 있음.매우높음스토리지 초기 구성 시 가장 먼저 설계하는 항목. 대부분 최신 환경은 Virtual Pool을 사용.
Thin Provisioning (씬 프로비저닝)볼륨을 실제 사용량만큼만 물리 공간을 할당하며, 논리적으로는 더 큰 용량처럼 보이게 함.높음용량 예측이 어려운 다수 볼륨을 운영할 때 물리 디스크 구매 비용을 아끼기 위해 사용. 단, 과다할당(overcommit) 모니터링 필요.
볼륨(Volume) & 호스트 매핑 (Host/Hostgroup Mapping)풀에서 잘라낸 논리 볼륨을 특정 호스트(서버)의 initiator(iSCSI IQN, FC WWN, SAS 주소)에 매핑해야 해당 서버가 LUN으로 인식.매우높음신규 서버에 스토리지 볼륨을 연결하는 모든 작업의 필수 단계. 매핑이 잘못되면 멀티패싱 이중화가 깨지거나 다른 서버가 볼륨을 잘못 인식하는 사고로 이어짐.
Provisioning Type — Standard vs Virtual(Thin) Volume볼륨 생성 시 처음부터 전체 용량을 물리적으로 예약(Standard/Full)할지, 사용한 만큼만 할당(Thin)할지 선택.중간용량 예측이 정확하고 성능 일관성이 중요한 DB는 Full Provisioning, 유연성이 필요한 범용 파일서버는 Thin.

5-2. Tiering / Snapshot / Replication

설정명설명 / 동작 방식실무 활용언제 사용하는가
Tier Affinity (계층 친화도) — No Affinity / Performance / ArchiveVirtual Pool 내에서 데이터를 SSD 계층(고성능)과 HDD 계층(대용량)에 자동 분산 배치. Performance는 핫데이터를 SSD에 우선 배치, Archive는 콜드데이터를 HDD에 유지.높음비용 효율과 성능을 동시에 잡아야 하는 하이브리드(SSD+HDD) 구성에서, 접근 빈도가 다른 데이터를 자동으로 최적 계층에 배치할 때.
스냅샷 (Snapshot, Redirect-on-Write)특정 시점의 볼륨 상태를 즉시 캡처. Redirect-on-Write 방식은 원본을 건드리지 않고 변경분만 새 위치에 기록해 스냅샷 생성이 매우 빠르고 공간 효율적.높음패치/업그레이드 직전 즉시 롤백 지점 확보, 실수로 인한 데이터 삭제·변경 복구용.
볼륨 복제 (Volume Copy) / 원격 복제 (Remote Replication)볼륨 전체를 다른 풀 또는 원격 사이트로 동기/비동기 복제.중간DR(재해복구) 사이트 구성, 백업 오프로드 목적.
압축 / 중복제거 (Compression / Deduplication)저장되는 데이터의 실제 물리 용량을 줄여 스토리지 효율을 높이는 기능. 스토리지 컨트롤러 CPU 자원을 추가로 소모.중간백업, VDI처럼 중복 데이터 비중이 높은 워크로드에서 스토리지 비용을 절감하고 싶을 때.

5-3. Availability / Cache / Monitoring

설정명설명 / 동작 방식실무 활용언제 사용하는가
캐시 Write Policy (어레이 레벨)컨트롤러 캐시의 Write-back/Write-through 정책. PERC와 개념은 동일하지만 어레이 컨트롤러 이중화(듀얼 컨트롤러) 구조와 결합해 동작.높음Write-back은 성능 위해 기본 사용, 컨트롤러 이중화·배터리 상태가 정상일 때만 안전하게 유지.
멀티패싱 (MPIO) / 호스트 포트 프로토콜 (iSCSI/FC/SAS)서버-스토리지 간 경로를 이중화해 하나의 경로 장애 시에도 자동 failover. 프로토콜에 따라 iSCSI(이더넷 기반), FC(전용 SAN), SAS(직결) 방식 선택.매우높음스토리지 연결의 가용성 확보를 위해 사실상 필수. 경로 하나만 구성하면 컨트롤러/케이블/HBA 장애 시 즉시 서비스 영향.
Global / Dedicated Spare (스토리지 레벨)디스크 장애 시 자동 투입되는 예비 디스크를 풀 전체가 공유(Global)할지 특정 디스크그룹 전용(Dedicated)으로 둘지 결정.높음무인 운영 환경에서 장애 대응 속도를 높이기 위해 최소 1개 이상의 스페어 확보가 표준.
듀얼 컨트롤러 이중화 (Active/Active, Active/Standby)스토리지 컨트롤러 모듈 2개가 각각 볼륨을 소유(Active/Active)하거나 하나가 대기(Standby)하며, 한쪽 장애 시 다른 쪽이 볼륨 소유권을 인계(failover).매우높음스토리지 컨트롤러 장애로 인한 전체 서비스 중단을 막기 위한 필수 이중화 구성.
Pool 용량 임계치 알림 (Threshold Alert)풀의 사용률이 지정한 %(예: 85%, 95%)를 넘으면 관리자에게 경고.높음씬 프로비저닝 환경에서 예상치 못한 용량 고갈(Out of Space) 장애를 사전에 막기 위한 필수 모니터링 설정.

❓ Storage Interview Questions

씬 프로비저닝의 위험 요소는?
여러 볼륨에 논리적으로 할당한 용량의 합이 실제 물리 풀 용량을 초과(overcommit)할 수 있다. 각 볼륨이 예상보다 빨리 채워지면 물리 공간이 실제로 고갈되어 풀 전체가 쓰기 불가 상태(full)에 빠질 수 있으므로 용량 임계치 모니터링과 알림 설정이 필수다.
Redirect-on-Write 스냅샷이 Copy-on-Write보다 빠른 이유는?
Copy-on-Write는 원본 데이터를 변경하기 전에 기존 데이터를 먼저 스냅샷 영역으로 복사(추가 쓰기 발생)해야 하지만, Redirect-on-Write는 원본을 그대로 두고 새로 쓰는 데이터만 다른 위치로 리다이렉트하기 때문에 쓰기 오버헤드가 훨씬 적다.

17G17세대부터 달라진 점 — 산업표준 NIC: 16G까지는 서버에 장착되는 NIC이 Dell 전용으로 커스터마이징된 펌웨어/드라이버를 쓰는 경우가 많았지만, 17G부터는 벤더(Broadcom/Intel/NVIDIA 등)의 네이티브 펌웨어·드라이버를 그대로 사용하는 "산업표준 NIC"으로 전환됐다. 관리·텔레메트리 정보는 표준 프로토콜을 통해 iDRAC에 노출되므로, 드라이버 호환성 확인 시 Dell 전용 다운로드보다 벤더 공식 배포판도 함께 참고해야 하는 경우가 늘었다.

6-1. Basic Switching

설정명설명 / 동작 방식실무 활용언제 사용하는가
VLAN물리적으로 하나의 스위치를 논리적으로 여러 개의 독립된 네트워크로 분리. switchport access vlan / switchport trunk allowed vlan으로 설정.매우높음관리망/운영망/스토리지망 분리, 멀티테넌트 환경에서 트래픽 격리가 필요할 때 — 거의 모든 데이터센터 네트워크의 기본 전제.
Port-Channel / LACP (channel-group ... mode active)여러 물리 포트를 하나의 논리 링크로 묶어 대역폭 합산과 링크 이중화를 동시에 제공. LACP(802.3ad)는 동적 협상 프로토콜로 Active/Passive 모드 지원.매우높음서버 NIC 이중화(팀/본딩)와 스위치 업링크 이중화 모두에서 표준적으로 사용. 서버의 bonding mode 4(802.3ad)와 반드시 짝을 맞춰 구성해야 함.
VLT (Virtual Link Trunking)Dell 고유의 MLAG(Multi-chassis LAG) 기술. 물리적으로 분리된 스위치 2대를 서버/이웃 장비 입장에서 논리적으로 하나의 스위치처럼 보이게 해, 스위치 자체의 장애까지 이중화.높음스위치 한 대가 통째로 다운돼도 서비스 영향 없이 이중화된 업링크를 유지해야 하는 코어/액세스 계층 설계.
Spanning Tree (STP/RSTP/RPVST+)이중화 구성에서 발생할 수 있는 루프를 감지해 특정 경로를 논리적으로 차단, 브로드캐스트 스톰 방지. OS10 기본값은 RPVST+(VLAN별 Rapid STP).높음다중 업링크·이중화 스위치 환경에서 루프 방지가 필요할 때. VLT를 쓰면 루프 자체가 설계상 발생하지 않아 STP 의존도가 줄어드는 경우도 많음.

6-2. QoS / Security

설정명설명 / 동작 방식실무 활용언제 사용하는가
QoS / DCB (PFC, ETS)트래픽 클래스별 우선순위 지정. Data Center Bridging의 PFC(Priority Flow Control)는 특정 트래픽 클래스만 선택적으로 흐름제어해 무손실(lossless) 전송을 보장.중간iSCSI/FCoE 등 스토리지 트래픽처럼 패킷 드랍에 민감한 프로토콜을 이더넷 위에서 안정적으로 돌려야 할 때.
ACL (Access Control List)특정 IP/포트/프로토콜 기준으로 트래픽 허용/차단 규칙을 스위치 레벨에서 적용.중간네트워크 장비 레벨에서 세그먼트 간 접근을 통제해야 하는 보안 요구사항이 있을 때.
Storm Control브로드캐스트/멀티캐스트/유니캐스트 트래픽이 비정상적으로 폭증하면 해당 포트의 트래픽을 자동으로 제한.중간루프나 오작동 장비로 인한 브로드캐스트 스톰이 전체 네트워크를 마비시키는 것을 막는 보호장치.
Port Security특정 포트에 연결이 허용되는 MAC 주소 개수/목록을 제한.중간비인가 장비의 물리적 네트워크 접속을 막아야 하는 보안 요구사항이 있는 액세스 계층.

6-3. Operations / Monitoring

설정명설명 / 동작 방식실무 활용언제 사용하는가
OOB 관리 인터페이스 (전용 관리 포트)스위치 자체 관리 전용 물리 포트로, 데이터 트래픽과 완전히 분리된 대역외(Out-of-Band) 관리망을 구성.높음운영 트래픽 폭주/장애 상황에서도 스위치에 접속해 문제를 진단·복구해야 할 때 필수.
SmartFabric / Spine-Leaf 설계OS10 기반의 자동화된 패브릭 구성(SmartFabric)과, 확장성이 뛰어난 Spine-Leaf 토폴로지(모든 리프가 모든 스파인에 연결).중간대규모 가상화/클라우드 환경에서 동일한 지연시간과 확장성을 보장하는 네트워크 설계가 필요할 때.

Uplink Failure Detection (UFD)액세스 스위치의 업링크(코어 방향) 포트가 모두 죽으면, 하위(서버 방향) 포트도 강제로 down시켜 서버의 NIC 이중화 소프트웨어가 장애를 인지하고 다른 경로로 failover하게 만듦.중간서버는 정상인데 스위치의 업링크만 끊겨 트래픽이 블랙홀에 빠지는 상황을 막기 위해 이중화 설계에서 함께 구성.
sFlow / 트래픽 미러링 (SPAN/Port Mirroring)스위치를 통과하는 트래픽의 샘플(sFlow) 또는 전체 사본(SPAN)을 모니터링 장비로 복제 전송.중간IDS/IPS, 패킷 캡처 분석 등 트래픽 가시성 확보가 필요한 보안·성능 진단 상황.

❓ Network Interview Questions

서버 NIC 본딩과 스위치 설정이 안 맞으면 어떤 문제가 생기나?
서버가 802.3ad(LACP) 모드로 본딩돼 있는데 스위치 포트가 일반 액세스 포트(포트채널 미구성)라면 LACP 협상 자체가 안 돼 링크가 불안정해지거나 특정 방향 트래픽만 유실될 수 있다. 반드시 서버 bonding mode와 스위치 Port-Channel/LACP 모드를 짝지어 구성해야 한다.
VLT와 일반 STP 이중화의 차이는?
일반 STP 이중화는 루프를 막기 위해 한쪽 경로를 논리적으로 차단(blocking)해 실질적으로 대역폭을 절반만 활용한다. VLT(MLAG)는 두 스위치를 논리적으로 하나처럼 만들어 양쪽 링크를 모두 Active로 동시에 사용하면서도 루프 없이 이중화할 수 있어 대역폭 효율이 더 좋다.

net.ipv4.ip_forward서버가 자신에게 온 패킷이 아닌 패킷도 다른 인터페이스로 전달(라우팅)할지 여부. 기본은 0(비활성).높음서버를 라우터/NAT 게이트웨이/VPN 게이트웨이로 쓸 때 반드시 1로 활성화.

CPU Governor (performance/powersave/ondemand/schedutil)OS 레벨에서 CPU 클럭 조정 정책을 결정. performance는 항상 최고 클럭 유지, powersave는 최저 클럭 유지, ondemand/schedutil은 부하에 따라 동적 조정.높음BIOS를 "Performance Per Watt (OS)"로 둔 경우 이 governor 값이 실질적인 전력/성능 트레이드오프를 결정하므로 지연시간 민감 서비스는 performance로 고정.

HugePages기본 4KB 페이지 대신 2MB/1GB 단위의 큰 메모리 페이지를 사용해 TLB 미스를 줄이고 메모리 관리 오버헤드를 낮춤.중간Oracle DB, 대형 가상화 호스트(KVM)처럼 대용량 메모리를 다루는 프로세스의 성능 최적화에 사용.

7-3. Storage / Filesystem / Network Redundancy

설정명설명 / 동작 방식실무 활용언제 사용하는가
I/O 스케줄러 (none/mq-deadline/kyber/bfq)블록 디바이스에 들어오는 I/O 요청의 순서·병합 방식을 결정. NVMe/SSD는 자체적으로 큐 관리가 뛰어나 스케줄러 개입이 적은 none이 유리, 회전형 HDD는 탐색시간을 줄이는 mq-deadline이 흔히 사용됨.높음디스크 타입(SSD/NVMe vs HDD)에 맞지 않는 스케줄러를 쓰면 불필요한 오버헤드가 생기므로 디스크 교체/증설 시 재점검.
마운트 옵션 noatime파일을 읽을 때마다 접근시각(atime)을 갱신하지 않도록 해 불필요한 메타데이터 쓰기를 줄임.높음읽기 위주 워크로드(웹서버 정적 파일, 로그 저장소 등)에서 디스크 I/O를 줄이기 위한 표준적인 튜닝.
Bonding / Teaming — mode 1 (active-backup)한 NIC만 활성으로 사용하고 장애 시 나머지로 failover. 스위치 측 특별 설정(LACP 등) 없이도 동작.매우높음이중화가 목적이고 스위치 간(서로 다른 스위치) 이중화까지 고려할 때 가장 무난하고 안정적인 선택.
Bonding — mode 4 (802.3ad / LACP)여러 NIC을 하나의 논리 링크로 묶어 대역폭을 합산. 스위치 측 Port-Channel/LACP 설정과 반드시 짝을 맞춰야 함.높음NIC 이중화뿐 아니라 대역폭 자체를 늘려야 할 때(스토리지 트래픽, 대용량 백업 구간).
LVM (LVM Volume Group / Logical Volume, Thin Pool)물리 디스크를 볼륨그룹으로 추상화해 논리볼륨을 유연하게 늘리고 줄일 수 있게 함. Thin Pool은 스토리지 어레이의 씬 프로비저닝과 같은 개념을 OS 레벨에서 제공.높음서비스 운영 중 디스크 확장이 잦은 환경에서 다운타임 없이 파티션을 재구성하고 싶을 때.

7-4. Security / Time Sync / Logging

설정명설명 / 동작 방식실무 활용언제 사용하는가

chrony / NTP (시각 동기화)서버 간 시각을 표준시각 서버와 동기화. 최근에는 chrony가 사실상 표준.매우높음클러스터 간 로그 정합성, 인증서(Kerberos, TLS) 유효기간 검증, 분산 트랜잭션 순서 보장 등 시각차가 조금만 나도 장애로 이어지는 모든 환경에서 필수.
journald 로그 볼륨 제한 (SystemMaxUse 등)systemd 저널 로그가 디스크를 무한정 채우지 않도록 최대 사용량/보관기간을 제한.중간루트 파티션 용량이 작은 서버에서 로그 폭증으로 디스크가 가득 차 서비스 장애로 이어지는 사고를 예방할 때.
auditd파일 접근, 시스템콜, 권한 변경 등을 커널 레벨에서 감사(audit) 로그로 기록.중간보안 컴플라이언스(PCI-DSS 등) 감사 대응, 침해사고 발생 시 행위 추적이 필요할 때.
SSH 하드닝 (PermitRootLogin no, 키 기반 인증)root 직접 로그인 차단, 비밀번호 인증 대신 공개키 인증 강제 등 SSH 접근을 강화.매우높음모든 프로덕션 서버에서 기본적으로 적용해야 하는 초기 보안 체크리스트 항목.

🚨 7-6. Troubleshooting Commands — Scenario Checklist

"장애가 터졌을 때 뭐부터 확인하나?"에 대한 답. 하드웨어 레벨(racadm)과 OS 레벨 명령어를 상황별로 함께 정리했다 — 실제 트러블슈팅은 이 둘을 오가며 진행되는 경우가 대부분이다.

① When Disk / Storage Failure Is Suspected

레벨명령어확인 목적
하드웨어 (racadm)racadm storage get pdisks물리 디스크 상태(Online/Failed/Rebuilding)를 컨트롤러 레벨에서 확인. 배드섹터·장애 디스크를 가장 먼저 특정.
하드웨어 (racadm)racadm storage get vdisks -o -p layout,status가상 디스크(RAID 볼륨)가 Degraded/Failed 상태인지, 어떤 물리 디스크로 구성됐는지 확인.
OS (Linux)lsblk / df -hOS가 실제로 인식한 블록 디바이스 목록과 파티션별 용량/사용률 확인 — 디스크가 통째로 안 보이면 컨트롤러 문제, 용량만 부족하면 OS/애플리케이션 문제로 분기.
OS (Linux)dmesg -T | grep -i -E "error|fail"커널 로그에서 디스크 I/O 오류, SCSI/NVMe 타임아웃 등 하드웨어 관련 에러 메시지를 검색.
OS (Linux)smartctl -a /dev/sdX디스크 자체 S.M.A.R.T 상태(배드섹터 카운트, 수명 등)를 직접 조회 — RAID 뒤에 가려진 개별 디스크 건강 상태 확인.
OS (Linux)multipath -ll외장 스토리지 연결 시 경로(path)가 정상적으로 잡혀 있는지, 특정 경로만 죽어 있는지 확인.

② When Server Is Unresponsive / Won’t Power On

레벨명령어확인 목적
하드웨어 (racadm)racadm getsysinfo전원 상태, 서비스 태그, 마지막 부팅 상태 등 시스템 개요를 원격에서 즉시 확인.
하드웨어 (racadm)racadm gettracelog / racadm getraclogPOST 실패, 전원 이상 등 SEL/트레이스 로그에서 원인 단서를 확인.
하드웨어 (racadm)racadm serveraction powerstatus → 필요 시 racadm serveraction powercycle현재 전원 상태를 먼저 확인한 뒤, OS가 완전히 멈춘 게 확실하면 원격으로 강제 재기동.
OS (Linux, 재부팅 후)journalctl -b -1 -p err직전 부팅 세션의 에러 레벨 로그만 필터링해 크래시 직전 상황을 재구성.
OS (Linux)last reboot / uptime서버가 언제 마지막으로 재부팅됐는지, 의도치 않은 재부팅이 반복되는지 이력 확인.

③ When Network Connectivity Issues Are Suspected

레벨명령어확인 목적
하드웨어 (racadm)racadm get iDRAC.NICiDRAC 자체 관리망(OOB) 설정이 정상인지 먼저 확인 — 원격 관리 자체가 끊기면 물리 출동이 필요해짐.
OS (Linux)ip a / ip link인터페이스가 UP 상태인지, 예상한 IP가 잡혀 있는지 확인.
OS (Linux)ss -tuln서비스가 실제로 예상 포트에 바인딩돼 리스닝 중인지 확인 — "포트가 안 열렸다"류 장애의 1차 확인.
OS (Linux)ethtool <iface>링크 속도/듀플렉스 협상 상태, 캐리어(carrier) 감지 여부 확인 — 케이블/스위치 포트 이슈 진단.
OS (Linux)cat /proc/net/bonding/bond0NIC 본딩 구성에서 어느 슬레이브가 Active/Down인지 확인 — 본딩-스위치 LACP 미스매치 진단.
OS (Linux)tcpdump -i <iface>실제로 패킷이 오가는지 인터페이스 레벨에서 직접 캡처해 방화벽/라우팅 문제와 물리 연결 문제를 구분.

④ CPU/Memory Overload or Process Issues

레벨명령어확인 목적

OS (Linux)vmstat 1CPU 대기(wa), 컨텍스트 스위치, 런큐 길이를 초 단위로 관찰해 병목이 CPU인지 I/O인지 구분.
OS (Linux)dmesg | grep -i "Out of memory"OOM Killer가 특정 프로세스를 강제 종료했는지 확인 — 서비스가 "이유 없이" 죽었을 때 1순위 확인 항목.

⑤ When Standardizing iDRAC/OME Settings Across Many Servers

레벨명령어/절차확인 목적
하드웨어 (racadm)racadm get iDRAC.SNMP / racadm set iDRAC.SNMP.AgentEnable 1OME나 Zabbix 등 모니터링 시스템이 SNMP로 알림을 받으려면, 각 서버의 iDRAC에서 SNMP가 활성화되고 트랩 목적지가 올바르게 설정돼 있어야 한다 — 신규 서버 대량 랙업 시 racadm 스크립트로 일괄 표준화.

❓ OS Interview Questions

DB 서버에서 swappiness를 낮추는 이유를 구체적으로 설명하면?
DB는 자체적으로 대용량 버퍼캐시(InnoDB Buffer Pool 등)를 메모리에 올려두고 성능을 낸다. 커널이 이 buffer pool의 일부를 스왑아웃시켜버리면 다음 쿼리에서 디스크 스왑 영역까지 갔다와야 해 순간적으로 응답시간이 수십~수백 배 튀는 지연 스파이크가 발생한다. swappiness를 0~10으로 낮추면 커널이 스왑보다 페이지 캐시 회수를 우선하도록 유도해 이런 스파이크를 최소화한다.
NIC 본딩 모드는 어떤 기준으로 고르나?
단순 이중화(서로 다른 스위치 연결 포함)가 목적이면 mode 1(active-backup) — 스위치 쪽 특별 설정이 필요 없어 안정적이다. 대역폭 합산까지 필요하면 mode 4(802.3ad/LACP)를 쓰되, 반드시 스위치 쪽에 대응하는 Port-Channel/LACP 설정이 되어 있어야 정상 동작한다.
시각 동기화(NTP/chrony)가 어긋나면 실제로 어떤 장애가 나는가?
분산 시스템에서 로그 순서를 시간 기준으로 비교할 수 없게 돼 장애 원인 분석이 꼬이고, Kerberos 인증처럼 시간차 허용범위(clock skew tolerance)가 있는 프로토콜은 인증 자체가 실패한다. 클러스터(예: etcd, Zookeeper)는 합의 알고리즘에서 타임스탬프 기반 판단이 어긋나 스플릿 브레인 등 심각한 장애로 이어질 수 있다.
서버가 다운됐다는 신고를 받으면 어떤 순서로 확인하나?
① racadm getsysinfo/gettracelog로 하드웨어 레벨(전원, POST 에러) 먼저 확인 → ② 정상이면 Virtual Console로 화면을 직접 보고 부팅 단계 확인 → ③ OS가 떴다면 journalctl -b -1, dmesg로 직전 세션 로그 확인 → ④ 디스크/네트워크/메모리 등 리소스 레벨 명령어로 세부 원인을 좁혀간다. 하드웨어(iDRAC/racadm)에서 OS 순으로 "바깥에서 안쪽으로" 좁혀가는 것이 표준 절차다.

📖 용어사전 모르는 단어 여기서 검색

위 탭들에서 당연하다는 듯 등장하는 약어·전문용어를 따로 모았다. NUMA, TPM, LACP처럼 "이게 뭔지부터 모르겠다" 싶은 단어가 나오면 여기서 검색해서 확인하자.

1 Hardware / BIOS Terms

NUMA

Non-Uniform Memory Access

CPU 소켓마다 "가까운(로컬)" 메모리와 "먼(원격)" 메모리가 나뉘어 있는 구조. 로컬 메모리 접근은 빠르고, 다른 소켓에 연결된 원격 메모리 접근은 상대적으로 느리다. 멀티소켓 서버 성능 튜닝의 핵심 개념 — DB나 가상화 프로세스를 특정 NUMA 노드에 고정(numactl)하면 원격 접근을 줄여 지연시간을 낮출 수 있다.

ECC Memory

Error-Correcting Code Memory

메모리에 저장된 데이터에 오류정정 코드를 추가로 저장해, 비트 반전 같은 오류를 감지하고 자동으로 정정하는 메모리. 서버는 안정성을 위해 거의 항상 ECC 메모리를 사용한다.

TPM

Trusted Platform Module

암호화 키를 안전하게 저장·생성하는 전용 보안 칩. 디스크 암호화(BitLocker 등)와 Secure Boot의 신뢰 기반이 된다.

ACPI

Advanced Configuration and Power Interface

하드웨어의 전원 상태(C-State, P-State 등)를 OS가 표준화된 방식으로 제어할 수 있게 하는 인터페이스 규격.

C-State

CPU Idle State

CPU가 유휴 상태일 때 들어가는 저전력 모드 단계(C0=활성, C1~C6로 갈수록 더 깊은 절전). 깊이 들어갈수록 전력은 아끼지만 다시 깨어나는 데 시간(latency)이 더 걸린다.

Turbo Boost

Intel Turbo Boost Technology

발열·전력 여유가 있을 때 CPU 코어 클럭을 정격 이상으로 자동으로 끌어올리는 기능.

Hyper-Threading (SMT)

Simultaneous Multi-Threading

물리 코어 1개를 OS에는 논리 코어 2개처럼 보이게 해 스레드 병렬성을 높이는 기술. Intel은 Hyper-Threading, AMD는 SMT라 부른다.

UEFI

Unified Extensible Firmware Interface

레거시 BIOS를 대체하는 최신 펌웨어 인터페이스. GPT 파티션, Secure Boot, 빠른 부팅을 지원한다.

POST

Power-On Self Test

전원이 켜진 직후 CPU·메모리·디스크 등 하드웨어가 정상인지 자체 점검하는 부팅 초기 단계.

DIMM / Memory Channel

Dual In-line Memory Module

메모리 모듈 자체를 DIMM이라 부르며, CPU-메모리 컨트롤러 간 연결 통로를 채널이라 한다. 채널을 균등하게 채우지 않으면(populate) 메모리 대역폭이 저하된다.

TDP

Thermal Design Power

CPU가 정상 동작 시 방출하는 최대 열량(전력) 설계치. 냉각 시스템 설계와 전력 예산 산정의 기준이 된다.

VT-d / IOMMU

Virtualization Technology for Directed I/O

물리 디바이스를 가상머신에 직접 할당(패스스루)할 수 있게 하는 하드웨어 기능. 디바이스의 DMA 접근을 가상주소로 안전하게 리매핑한다.

SR-IOV

Single Root I/O Virtualization

물리 NIC/디바이스 1개를 여러 개의 가상 기능(Virtual Function)으로 쪼개 여러 VM에 직접 할당하는 기술. 하이퍼바이저의 가상 네트워크 스위치를 거치지 않아 지연시간이 낮다.

PXE

Preboot Execution Environment

디스크 없이도 네트워크를 통해 OS 이미지를 내려받아 부팅하는 방식. 대량 서버 자동 설치(프로비저닝)에 사용.

GPT vs MBR

GUID Partition Table / Master Boot Record

디스크 파티션 정보를 저장하는 두 방식. MBR은 2TB·4개 파티션 한계가 있는 구형 방식, GPT는 대용량 디스크와 UEFI 부팅을 지원하는 최신 방식.

2 RAID / Storage Terms

RAID

Redundant Array of Independent Disks

여러 물리 디스크를 하나의 논리 볼륨으로 묶어 성능(스트라이핑)이나 안정성(미러링/패리티)을 높이는 기술. 레벨(0/1/5/6/10 등)에 따라 특성이 다르다.

Parity

Parity

원본 데이터로부터 계산해낸 검증용 값. RAID5/6에서 디스크 하나(또는 둘)가 죽어도 나머지 데이터와 패리티로 원본을 복원할 수 있게 해준다.

Write-Back / Write-Through

Write Cache Policy

Write-Back은 컨트롤러 캐시에 데이터가 들어오면 즉시 완료 응답(빠름, 정전 시 유실 위험), Write-Through는 디스크에 실제로 써진 후에만 완료 응답(느림, 안전).

BBU / CacheVault

Battery Backup Unit / Cache Vault

정전 시에도 RAID 컨트롤러 캐시의 데이터를 보존해주는 배터리(BBU) 또는 커패시터+플래시(CacheVault) 보호 장치. Write-Back 캐시를 안전하게 쓰기 위한 전제조건.

Patrol Read

Background Disk Scan

유휴 시간에 백그라운드로 전체 디스크를 스캔해 배드섹터를 조기에 발견·복구하는 기능.

Hot Spare

Spare Disk

RAID 구성 디스크 중 하나가 고장 나면 사람 개입 없이 자동으로 투입되는 대기 디스크.

Thin Provisioning

Thin Provisioning

논리적으로는 큰 용량의 볼륨을 만들어주지만, 물리적으로는 실제 사용한 만큼만 공간을 할당하는 방식.

Redirect-on-Write

Redirect-on-Write Snapshot

스냅샷을 찍을 때 원본 데이터는 그대로 두고, 새로 쓰이는 데이터만 다른 위치로 리다이렉트해 저장하는 방식. Copy-on-Write보다 쓰기 오버헤드가 적다.

LUN

Logical Unit Number

스토리지가 서버에게 노출하는 논리적인 디스크 단위 번호. 서버 OS 입장에서는 LUN 하나가 물리 디스크 하나처럼 보인다.

MPIO

Multipath I/O

서버-스토리지 사이의 여러 물리 경로를 하나의 논리 경로로 묶어 이중화·부하분산하는 기술.

HBA

Host Bus Adapter

서버가 SAN 스토리지(FC, SAS 등)에 연결하기 위해 장착하는 전용 어댑터 카드.

iSCSI / FC / SAS

Storage Connectivity Protocol

iSCSI는 일반 이더넷 위에서 SCSI 명령을 전달(저비용), FC(Fibre Channel)는 전용 광케이블 기반 SAN(고성능·고비용), SAS는 서버-스토리지 간 직결에 주로 사용된다.

3 iDRAC / Remote Management Terms

BMC

Baseboard Management Controller

메인보드에 내장된 독립 관리 프로세서의 일반 명칭. Dell에서는 이를 iDRAC이라는 브랜드로 부른다.

iDRAC

Integrated Dell Remote Access Controller

Dell 서버 전용 BMC. 서버 전원과 무관하게 동작하며 원격 콘솔·전원제어·모니터링을 제공.

Redfish

DMTF Redfish API

REST/JSON 기반의 최신 서버관리 표준 API. IPMI를 대체하는 업계 표준으로 자리잡고 있다.

IPMI

Intelligent Platform Management Interface

Redfish 이전 세대의 서버 하드웨어 관리 표준 프로토콜. 기능이 제한적이고 구형 툴체인에서만 주로 사용.

RACADM

Remote Access Controller ADMin

iDRAC 설정을 CLI/스크립트로 다루는 Dell 전용 명령행 도구.

Lifecycle Controller

LC

OS 없이도 BIOS/펌웨어 업데이트, 하드웨어 진단, RAID 구성을 할 수 있게 해주는 Dell의 내장 관리 엔진.

SEL

System Event Log

하드웨어 이벤트(디스크 오류, 전원 이상, 온도 경고 등)를 시간순으로 기록하는 로그. 장애 분석의 1차 확인 지점.

4 Network Terms

VLAN

Virtual LAN

물리적으로 하나인 네트워크를 논리적으로 여러 개의 독립된 네트워크로 분리하는 기술.

LACP / Port-Channel

Link Aggregation Control Protocol

여러 물리 포트를 하나의 논리 링크로 묶어 대역폭 합산과 이중화를 동시에 제공하는 표준(802.3ad) 프로토콜.

VLT / MLAG

Virtual Link Trunking / Multi-Chassis LAG

물리적으로 분리된 스위치 2대를 논리적으로 하나처럼 동작시켜, 스위치 자체 장애까지 이중화하는 기술. Dell은 VLT, 업계 일반 용어는 MLAG.

STP / RSTP

Spanning Tree Protocol

이중화 네트워크에서 루프가 생기지 않도록 특정 경로를 논리적으로 차단하는 프로토콜.

QoS / DCB / PFC

Quality of Service / Data Center Bridging / Priority Flow Control

트래픽 종류별로 우선순위나 대역폭을 차등 적용하는 기술. PFC는 특정 트래픽 클래스만 선택적으로 흐름제어해 무손실 전송을 보장(스토리지 트래픽용).

MTU / Jumbo Frame

Maximum Transmission Unit

이더넷 프레임의 최대 크기. 기본 1500바이트보다 큰 9000바이트 프레임을 Jumbo Frame이라 하며, 대용량 전송의 오버헤드를 줄인다.

OOB Management Network

Out-of-Band Management

실제 서비스 트래픽과 물리적/논리적으로 완전히 분리된 별도의 관리 전용 네트워크 경로.

5 OS / Kernel Terms

sysctl

System Control

Linux 커널 파라미터를 런타임에 조회·변경하는 인터페이스(/proc/sys 기반).

swappiness

Swap Tendency Value

메모리 여유가 있어도 커널이 얼마나 적극적으로 스왑을 쓸지 결정하는 0~100 값. 낮을수록 스왑을 미룬다.

HugePages

Large Memory Pages

기본 4KB 대신 2MB/1GB 단위의 큰 메모리 페이지를 사용해 TLB 미스와 관리 오버헤드를 줄이는 기법.

cgroups

Control Groups

프로세스 그룹별로 CPU·메모리·I/O 자원 사용량을 제한·격리하는 커널 기능. 컨테이너 기술의 근간.

systemd

System and Service Manager

대부분의 최신 Linux 배포판이 사용하는 초기화 및 서비스 관리 데몬. 부팅 프로세스, 서비스 상태, 로그(journald)를 관리.

SELinux

Security-Enhanced Linux

프로세스별로 세밀한 강제접근제어(MAC)를 적용하는 커널 보안 모듈.

ulimit

User Limit

사용자/프로세스가 사용할 수 있는 파일 디스크립터, 프로세스 수 등 리소스의 상한을 지정하는 셸/커널 기능.

I/O Scheduler

I/O Scheduler

블록 디바이스에 들어오는 입출력 요청의 처리 순서·병합 방식을 결정하는 커널 계층(none/mq-deadline/bfq 등).

LVM

Logical Volume Manager

물리 디스크를 볼륨그룹으로 추상화해 논리볼륨을 유연하게 늘리고 줄일 수 있게 하는 스토리지 관리 계층.

chrony / NTP

Time Synchronization Protocol

서버의 시각을 표준시각 서버와 동기화하는 프로토콜/데몬. 최근 Linux 배포판은 chrony가 기본.

6 17th Gen (17G) New Technology Terms

MRDIMM

Multiplexed Rank DIMM

기존 RDIMM보다 랭크를 멀티플렉싱해 같은 핀 수로 더 높은 대역폭(최대 8,800 MT/s)을 내는 신형 DDR5 메모리. Intel Xeon 6(Granite Rapids) 플랫폼부터 지원.

CXL

Compute Express Link

CPU와 메모리/가속기 사이를 PCIe 물리 계층 위에서 캐시 일관성 있게 연결하는 차세대 인터커넥트 표준. CXL 2.0부터는 DIMM 슬롯 한계를 넘는 메모리 확장(메모리 풀링)이 가능해진다.

Intel TDX

Trust Domain Extensions

VM 단위로 메모리를 하드웨어 레벨에서 암호화·격리하는 기밀 컴퓨팅 기능. 하이퍼바이저 관리자조차 VM 내부 메모리를 들여다볼 수 없게 한다.

iDRAC10

New-Gen BMC for 17G

PowerEdge 17G부터 탑재되는 차세대 관리 컨트롤러. TPM·Root-of-Trust·원격 증명·암호화를 통합해 Zero Trust 보안 모델을 지원한다.

PERC12

New RAID Controller Generation for 17G

H965i(Front/Adapter/MX), H965e 등이 속한 PERC 세대. PCIe 5.0과 더 높은 IOPS를 지원하지만 WT/WB/FWB 같은 캐시 정책 개념은 이전 세대와 동일하다.

Industry-Standard NIC

Industry-Standard Network Card

17G부터 Dell 서버에 장착되는 NIC이 벤더(Broadcom/Intel/NVIDIA 등)의 네이티브 펌웨어·드라이버를 그대로 사용하도록 전환된 방식. 이전엔 Dell 전용으로 커스터마이징된 펌웨어를 사용했다.
