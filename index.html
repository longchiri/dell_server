🔰 처음 오셨다면 이 순서로 이해하면 쉽습니다: 서버 전원을 누르면 ① 가장 먼저 BIOS가 켜져서 "이 하드웨어를 어떻게 켤지" 결정하고 → ② 그 위에서 OS가 부팅된다. 서버 전원이 꺼져 있어도 언제든 원격으로 들여다볼 수 있는 별도의 작은 컴퓨터가 iDRAC이고, 그 iDRAC 수백~수천 대를 한 화면에서 관리하는 관제탑이 OME다. 디스크 여러 개를 안전하게 묶는 기술이 RAID(서버 내장)이고, 여러 서버가 공유하는 외장 저장장치가 스토리지 어레이, 그 서버들을 연결하는 게 네트워크 스위치다. 이 모든 하드웨어 위에서 실제로 서비스가 돌아가는 소프트웨어 계층이 OS. 즉 아래 탭 순서(① → ⑦)가 실제로 서버 한 대가 세상에 나오는 순서와 거의 같다고 보면 된다. 모르는 단어가 나오면 언제든 📖 용어사전 탭에서 검색해보자.

컴퓨터(서버) 전원을 눌렀을 때 화면에 OS 로고가 뜨기도 전에 잠깐 지나가는 검은 화면이 있는데, 그 순간 동작하는 게 바로 BIOS다. 컴퓨터를 사람에 비유하면 BIOS는 "눈뜨자마자 팔다리가 제대로 붙어있는지, 숨은 잘 쉬는지" 확인하는 아주 기초적인 반사신경 같은 것 — 이 단계에서 CPU·메모리·디스크 등 하드웨어를 어떻게 켜고 쓸지 미리 정해둔다. PowerEdge 서버는 부팅 중 F2 키를 누르면 이 설정 화면(System Setup)에 들어갈 수 있다. 여기서 바꾼 값은 OS보다 훨씬 아래(더 근본적인) 레벨에서 동작하기 때문에, 잘못 건드리면 아예 OS 설치가 안 되거나 서버가 갑자기 느려지는 등 큰 영향을 줄 수 있어 신중해야 한다. 아래 소분류 탭을 눌러 항목별로 확인하자.

동작 원리 한 줄 요약: BIOS 설정은 부팅 초기 POST 단계에서 하드웨어 초기화 방식을 결정하고, 이후 OS/드라이버는 그 위에서 동작한다. 즉 "그릇의 모양"을 정하는 단계 — RAID 모드, NUMA 노출 여부, 가상화 지원 여부처럼 OS가 나중에 바꿀 수 없는 항목이 많다.

CPU의 하드웨어 가상화 확장 기능을 활성화한다.Enabled면 → 하이퍼바이저(ESXi/Hyper-V/KVM)가 하드웨어 레벨로 VM을 격리·실행 가능.Disabled면 → 하이퍼바이저 설치 자체가 실패하거나, 되더라도 소프트웨어 에뮬레이션으로 떨어져 성능이 크게 저하 — 가상화 호스트에서는 사실상 필수.매우높음
Number of Cores per Processor물리 코어 중 일부만 활성화하고 나머지는 비활성화.🔰 CPU에 코어가 32개 들어있어도 일부러 그중 일부만 켜고 나머지는 꺼두는 옵션이다. 물리 코어 중 일부만 활성화하고 나머지는 비활성화.All(기본)로 두면 → 장착된 모든 코어를 그대로 사용.특정 개수로 제한하면 → 처리 성능은 그만큼 줄지만, 코어 수 기준으로 과금되는 소프트웨어 라이선스(오라클 등) 비용을 절감할 수 있음.낮음
Sub-NUMA Cluster (SNC)소켓 1개를 여러 개의 NUMA 도메인으로 쪼개 메모리 지역성을 세분화.🔰 NUMA란 "내 옆자리 서랍(가까운 메모리)에서 물건을 꺼내는 게, 저 멀리 있는 창고(먼 메모리)까지 가서 꺼내오는 것보다 빠르다"는 구조를 말한다. 이 옵션은 CPU 소켓 1개를 여러 개의 작은 "옆자리 서랍 구역"(NUMA 도메인)으로 더 잘게 쪼개 메모리 지역성을 세분화한다.Enabled면 → 로컬 메모리 접근 지연이 더 줄어들어 NUMA-aware DB/HPC 성능이 향상되지만, NUMA 도메인 수가 늘어 애플리케이션이 이를 제대로 인식 못 하면 오히려 손해를 볼 수 있음.Disabled(기본)면 → 소켓당 NUMA 도메인이 하나로 단순해 예측하기 쉽지만 세밀한 지역성 최적화는 놓침.중간
Turbo Boost코어 부하/발열/전력 여유에 따라 정격 클럭 이상으로 자동 상승.🔰 자동차로 치면 잠깐 필요할 때 밟는 "순간 가속(부스터)" 같은 기능이다. 평소 정해진 속도(정격 클럭)로 달리다가, 여유가 있을 때 잠깐 더 빠르게 밟아준다. 코어 부하/발열/전력 여유에 따라 정격 클럭 이상으로 자동 상승.Enabled(기본)면 → 단일/소수 코어 부하에서 클럭이 순간적으로 크게 올라가 체감 성능 향상, 대신 전력·발열이 상황에 따라 출렁임.Disabled면 → 클럭이 항상 정격에 고정돼 최대 성능은 낮아지지만, 서버마다 클럭 편차가 없어 대규모 클러스터의 성능 표준화·전력 예산 관리에 유리.매우높음
C-States / C1ECPU가 유휴 상태일 때 더 깊은 저전력 상태로 진입.🔰 사람이 가만히 있을 때 숨만 쉬면서 에너지를 아끼듯, CPU도 할 일이 없을 때 "쪽잠"을 자면서 전력을 아끼는 기능이다. 다만 깊이 잠들수록 깨어나는 데 시간이 조금 더 걸린다. CPU가 유휴 상태일 때 더 깊은 저전력 상태로 진입.Enabled(기본)면 → 유휴 시 전력을 크게 절감하지만, 저전력 상태에서 다시 깨어나는 데 지연(latency spike)이 발생.Disabled면 → 코어가 항상 활성 상태를 유지해 응답속도가 일정해지지만 유휴 시에도 전력을 계속 소모 — 금융 트레이딩·실시간 서비스는 Disabled로 응답 일관성을 우선.높음
Execute Disable (NX bit / DEP)데이터 영역에서의 코드 실행을 하드웨어 레벨에서 차단.🔰 컴퓨터 메모리 안에는 "데이터를 저장하는 칸"과 "명령어(프로그램)를 실행하는 칸"이 있는데, 해커들이 종종 데이터 칸에 악성 코드를 몰래 심어 실행시키는 공격을 한다. 이 기능은 데이터 칸에서는 아예 실행 자체를 못 하게 하드웨어 레벨에서 차단해버리는 보안장치다.Enabled(기본)면 → 버퍼 오버플로우 등 데이터 영역 실행 공격이 하드웨어 레벨에서 차단됨.Disabled면 → 이 방어층이 사라져 관련 공격에 취약해짐 — 끄는 경우는 사실상 없음.매우높음
Uncore Frequency코어 외부(캐시, 메모리 컨트롤러 등) 클럭.🔰 CPU 안에서 "계산하는 부분"(코어)이 아니라 "물건을 나르는 통로"(캐시, 메모리 컨트롤러 등)의 속도를 정하는 옵션이다. 코어 외부(캐시, 메모리 컨트롤러 등) 클럭.Dynamic(기본)이면 → 부하에 맞춰 자동 조율돼 대체로 무난.Maximum으로 고정하면 → 메모리 대역폭이 항상 최대로 유지돼 대역폭 병목 워크로드 성능이 오르지만 전력 소모는 늘어남.낮음

x2APIC Mode코어 수가 많은 시스템(보통 128 논리 프로세서 초과)에서 인터럽트 컨트롤러 주소지정 방식을 확장.🔰 학생이 너무 많아 반이 128명을 넘으면 출석부 번호 체계를 새로 짜야 하듯, 코어(논리 프로세서)가 128개를 넘는 초고밀도 서버에서는 각 코어에 "번호"를 매기는 방식 자체를 확장해줘야 한다. 코어 수가 많은 시스템에서 인터럽트 컨트롤러 주소지정 방식을 확장.Enabled면 → 고밀도 코어 서버에서 OS가 모든 논리 프로세서를 누락 없이 인식.Auto(기본)에서 필요한데도 안 켜면 → 일부 논리 프로세서가 OS에 인식되지 않아 실제 코어 수보다 적게 활용될 수 있음.낮음

Configurable TDP (cTDP)CPU의 열설계전력(TDP) 상한을 정격보다 낮추거나(Low) 높여(Nominal/High) 재설정.🔰 TDP란 "이 CPU가 최대로 열을 얼마나 뿜어낼지"를 나타내는 숫자다(전구의 와트 수와 비슷한 개념). 이 옵션은 그 상한을 정격보다 낮추거나(Low, 더 시원하지만 덜 빠름) 높여서(Nominal/High, 더 빠르지만 더 뜨거움) 재설정한다.Low로 낮추면 → 발열·전력 소모가 줄어 랙 밀도를 높일 수 있지만 최대 성능은 낮아짐.High로 올리면 → 성능은 늘지만 냉각·전력 예산이 그만큼 더 필요.낮음
Hardware Prefetcher / Adjacent Cache Line Prefetch / DCU Streamer PrefetcherCPU가 다음에 필요할 것으로 예측되는 데이터를 미리 캐시로 가져오는 기능들.🔰 마트 직원이 손님이 다음에 뭘 살지 미리 예측해서 계산대 옆에 갖다놓는 것처럼, CPU가 다음에 필요할 것 같은 데이터를 미리 예측해서 빠른 저장공간(캐시)에 가져다두는 기능들이다.Enabled(기본)면 → 순차 접근 패턴 워크로드에서 캐시 히트율이 올라가 성능 향상.Disabled면 → 순차 워크로드 성능은 소폭 떨어질 수 있지만, 랜덤 접근 위주 워크로드에서는 불필요한 프리페치로 인한 캐시 오염이 줄어 오히려 성능이 오르는 경우도 있음.낮음

Consistent Device Naming (CDN)온보드/애드인 NIC의 물리적 슬롯 위치를 OS의 네트워크 인터페이스 이름(예: NIC1, NIC.Integrated.1-1)에 일관되게 매핑해주는 기능. 면접에서 자주 나오는 항목. "성능 vs 전력" 트레이드오프를 이해하는지 확인하는 단골 질문.

프로필설명 / 동작 방식실무 활용언제 사용하는가
PerformanceTurbo Boost 최대 활용, C-States 최소화, 항상 최고 클럭 유지 지향. 전력 소모와 발열이 가장 크다.🔰 무조건 최고 속도로만 달리는 모드다. 전기를 가장 많이 먹고 열도 많이 난다.높음지연시간에 극도로 민감한 워크로드(고빈도 트레이딩, 실시간 렌더링/스트리밍, 벤치마크).
Performance Per Watt (DAPC) — 기본값DAPC(Dell Active Power Controller)가 BIOS 펌웨어 레벨에서 능동적으로 성능과 전력을 실시간 조율. 일반적으로 최대 성능 대비 3~4% 수준의 전력만 절감하면서 성능 저하는 거의 없음.🔰 Dell이 알아서 성능과 전력을 균형 있게 조절해주는 기본 모드다. 특별한 이유가 없다면 이 상태로 두면 된다.매우높음대부분의 일반 운영 서버(웹, 애플리케이션, 파일서버)에 적합한 기본값. 특별한 이유가 없다면 유지.
Performance Per Watt (OS) / DBPM전력 관리 결정권을 BIOS가 아닌 OS의 ACPI 기반 전원 정책(cpufreq governor 등)에 위임.🔰 전력 조절을 BIOS가 아니라 OS(Windows/Linux)에게 맡기는 모드다.중간OS 레벨에서 세밀한 전력 정책(예: tuned 프로필)을 직접 제어하고 싶을 때.
CustomTurbo Boost, C1E, C-States, Monitor/Mwait, 메모리 주파수, Uncore 주파수 등 개별 옵션을 수동으로 세팅. Custom 선택 시에만 하위 항목이 활성화됨.🔰 미리 짜인 프리셋을 쓰지 않고, 세부 항목을 하나하나 내가 직접 정하는 모드다.중간Dell 공식 튜닝 가이드/벤치마크 기준에 맞춰 세밀하게 최적화해야 하는 HPC·DB 전용 서버.
Workstation Performance단일 스레드 반응성에 가중치를 둔 프로필.🔰 서버보다는 디자인/설계용 워크스테이션에 어울리는 모드다.낮음서버보다는 워크스테이션 용도로 쓰이는 PowerEdge에서 CAD/시뮬레이션 작업 시.
Custom을 선택하면 무슨 일이 생기나: Performance/DAPC 등 사전 정의 프로필을 고르면 BIOS가 아래 모든 세부 항목을 알아서 채워 넣고 화면에서 회색 처리(수정 불가)한다. System Profile을 Custom으로 바꾸는 순간에만 이 세부 항목들이 하나하나 손으로 조정 가능한 상태로 풀린다 — 즉 Custom은 "프리셋을 버리고 내가 직접 하나씩 맞추겠다"는 선언이다. C1E프로세서가 유휴 상태일 때 최소 성능 상태로 전환할지 여부(기본 Enabled).🔰 CPU가 잠깐 쉴 때 살짝 졸게 할지 정하는 옵션이다.Enabled면 → 유휴 시 클럭을 낮춰 전력을 절감하지만, 다시 부하가 걸릴 때 클럭을 올리는 데 짧은 지연(latency)이 발생.Disabled면 → 유휴 상태에서도 클럭을 낮추지 않아 전력은 더 쓰지만 응답속도가 항상 일정하게 유지됨.높음

Uncore FrequencyDynamic 모드를 선택하면 BIOS가 코어/언코어 간 전력을 실시간 최적화하며, 방향은 Energy Efficient Policy에 좌우됨.🔰 코어가 아닌 나머지 부분(캐시 등)의 속도를 자동으로 둘지, 최대로 고정할지 정하는 옵션이다.Dynamic이면 → 부하 상황에 맞춰 자동으로 조율돼 대체로 무난하지만 특정 벤치마크에서 최고치를 못 찍을 수 있음.Maximum으로 고정하면 → 항상 최대 언코어 클럭을 유지해 메모리 대역폭이 병목인 워크로드에서 성능이 오르지만 전력 소모는 늘어남.낮음
Energy Efficient PolicyCPU가 "성능 우선"과 "절전 우선" 중 어느 쪽으로 행동할지 결정(기본 Balanced Performance).🔰 CPU가 속도와 절전 중 어느 쪽에 더 무게를 둘지 정하는 성향 옵션이다.Performance 쪽으로 두면 → 전력을 더 쓰더라도 처리 속도를 우선시.Energy Efficient 쪽으로 두면 → 성능을 일부 양보하고 전력 소비를 최소화하는 방향으로 CPU 내부 동작이 조정됨.중간

❓ BIOS Interview Questions

C-States를 끄면 성능이 왜 좋아지나?
CPU가 유휴 상태에서 저전력 상태(C1~C6)로 들어갔다가 다시 활성 상태로 복귀하는 데 마이크로초 단위 지연이 발생한다. 초저지연이 중요한 서비스는 이 wake-up latency 자체가 문제가 되므로 C-States를 꺼서 항상 활성 상태(C0)를 유지시켜 응답속도를 일정하게 만든다. 대신 유휴 시에도 전력을 계속 소모한다.
가상화 호스트 구축 전 BIOS에서 가장 먼저 확인할 항목은?
Virtualization Technology(VT-x/AMD-V) Enabled 여부와, 필요 시 IOMMU/VT-d(디바이스 패스스루), SR-IOV Global Enable 여부. 이게 꺼져 있으면 ESXi/Hyper-V/KVM 설치 자체가 실패하거나 중첩 가상화가 동작하지 않는다.
AC Power Recovery 설정을 놓쳐서 생기는 실제 장애 사례는?
데이터센터 정전 후 전원이 복구됐는데 AC Power Recovery가 기본값(Off 등)으로 남아있으면 서버가 자동으로 켜지지 않는다. 무인 원격지 서버라면 담당자가 직접 콘솔로 전원 버튼을 눌러야 하는 상황이 발생 — 실무에서 반드시 On/Last State로 바꿔두는 대표적 체크 항목이다.
온보드 SATA를 AHCI로 둘지 RAID로 둘지 기준은?
별도 PERC 하드웨어 RAID 카드가 있고 그 카드가 디스크를 관리한다면 온보드 SATA는 AHCI(또는 Off)로 둔다. PERC 없이 온보드 컨트롤러만으로 소프트웨어 RAID를 구성해야 한다면 SATA를 RAID 모드로 바꾼다.
GPU 패스스루를 위해 BIOS에서 확인해야 할 3가지는?
① VT-d/IOMMU Enabled, ② Memory Mapped I/O above 4GB Enabled(대용량 VRAM 인식용), ③ SR-IOV Global Enable(가상함수 분할이 필요한 경우). 이 중 하나라도 빠지면 VM에 GPU를 정상적으로 패스스루할 수 없다.
Node Interleaving과 NUMA는 무슨 관계인가?
Node Interleaving이 Disabled(기본값)일 때 NUMA 토폴로지가 그대로 OS에 노출된다. Enabled로 바꾸면 전체 메모리를 인터리빙해 NUMA 구분 자체를 없애버리므로, NUMA를 인식 못 하는 구형 소프트웨어와의 호환성은 좋아지지만 최신 NUMA-aware 애플리케이션에는 오히려 손해다.

2. iDRAC Remote Management (Integrated Dell Remote Access Controller)

서버 안에 들어있는 "리모컨 겸 CCTV"다. 서버 본체의 전원이 꺼져 있거나 OS가 완전히 멈춰버려도, iDRAC이라는 이 별도의 작은 관리 컴퓨터는 계속 살아있어서 인터넷(네트워크)만 연결돼 있으면 사무실 책상에 앉아서도 전원을 켜고, 화면을 보고, 문제를 고칠 수 있게 해준다. 데이터센터가 지방이나 해외에 있어도 직접 가지 않고 대부분의 문제를 해결할 수 있는 게 바로 이 기능 덕분이다. 정식 명칭은 BMC(Baseboard Management Controller)이고, Dell은 이를 iDRAC이라는 이름으로 부른다.

동작 원리: iDRAC은 호스트 OS와 완전히 분리된 자체 CPU·메모리·네트워크 인터페이스를 갖는 "서버 안의 작은 컴퓨터"다. 메인 서버 전원이 내려가 있어도 iDRAC은 보조전원으로 계속 동작하며, Redfish/RACADM/웹UI/IPMI를 통해 하드웨어를 직접 제어한다.
17G17세대부터 달라진 점 — iDRAC10: 17G부터는 새 세대 BMC인 iDRAC10이 탑재된다. TPM·Root-of-Trust·원격 증명(Attestation)·전송 중 암호화를 통합해 "AI 기반" 관리와 Zero Trust 보안 모델을 지원하도록 재설계됐다. UI/API는 iDRAC9와 크게 다르지 않지만 내부 보안 아키텍처와 부팅 시 무결성 검증 방식이 강화됐다.

Quick Sync / LCD 패널NFC/Bluetooth로 모바일 앱에서 서버 정보를 태그하듯 확인하거나, 섀시 전면 LCD로 IP·장애 상태를 바로 확인.🔰 스마트폰을 서버에 갖다 대거나 서버 앞면 화면을 보는 것만으로 IP·상태를 바로 확인할 수 있는 기능이다.낮음랙 전면에서 콘솔 접속 없이 즉시 IP/상태를 확인해야 하는 실사 점검 상황.

2-4. RACADM Usage — From Basics to Practice

RACADM은 iDRAC을 GUI 없이 CLI로 조회·설정하는 Dell 전용 도구다. "RAC(Remote Access Controller) ADMin"의 줄임말 — 즉 iDRAC 자체를 다루는 도구이지, OME에 서버를 등록하는 도구가 아니다. (OME 등록/검색 방식은 OME 탭 참고)

3가지 접속 방식
① Local RACADM — 대상 서버의 OS에 iDRAC Service Module(iSM)이 설치돼 있으면 그 OS 안에서 바로 racadm 명령을 실행. IP/계정 입력 없이 로컬 iDRAC을 바로 제어.
② Remote RACADM — 관리자의 PC/배포서버에서 iDRAC의 IP로 원격 접속. 대상 서버 OS 상태와 무관하게 동작하며 스크립트 자동화에 가장 많이 쓰임: racadm -r <iDRAC_IP> -u root -p <비밀번호> --nocertwarn <명령>
③ Firmware RACADM — iDRAC에 SSH/Telnet으로 직접 로그인한 뒤 IP·계정 입력 없이 바로 명령 실행 (이미 인증된 세션이므로).
목적예시 명령설명
시스템 정보 조회racadm getsysinfo서비스 태그, BIOS/펌웨어 버전, 전원 상태 등 시스템 기본 정보를 한 번에 출력. 원격 진단의 첫 명령.
설정값 조회 (get)racadm get iDRAC.Network / racadm get BIOS.SysProfileSettingsFQDD(그룹) 단위로 iDRAC/BIOS 설정 그룹을 통째로 조회. 그룹명.속성명 구조로 계층화돼 있음.
설정값 변경 (set)racadm set iDRAC.NIC.DHCPEnable 0get으로 확인한 속성을 개별적으로 변경. 다수 서버에 동일 설정을 스크립트로 일괄 적용할 때 핵심.
전원 제어racadm serveraction powercycle / hardreset / powerdownOS가 완전히 멈춰 응답이 없을 때 원격으로 강제 전원 재기동. 물리 출동 없이 가능한 최후의 복구 수단.
하드웨어 로그 확인racadm gettracelog / racadm getraclogiDRAC 자체 트레이스 로그와 RAC 로그를 확인. SEL과 함께 하드웨어 장애의 근본 원인을 추적하는 핵심 로그.
RAID/디스크 조회racadm storage get pdisks / racadm storage get vdisks -o -p layout,status물리 디스크(pdisks)와 가상 디스크(vdisks)의 상태·레이아웃을 CLI로 조회. GUI 접속 없이 RAID 상태를 스크립트로 점검할 때.
iDRAC 초기화racadm racreset (iDRAC만 재시작) / racadm racresetcfg (공장 초기화)racreset은 iDRAC 프로세스만 재시작(서버 본체는 영향 없음), racresetcfg는 사용자가 설정한 모든 값을 삭제하고 기본값으로 초기화 — 되돌릴 수 없으므로 신중히 사용.
설정 통째로 복제 (SCP)racadm get -f config.xml -t xml (내보내기) / racadm set -f config.xml -t xml (가져오기)Server Configuration Profile(SCP) 기능으로 BIOS/iDRAC/RAID 설정을 XML/JSON로 통째로 내보내고, 동일 모델의 다른 서버에 그대로 적용. 대량 서버 표준화 배포의 핵심 기법.
💡 실습 팁: 처음이라면 GUI로 값을 먼저 바꿔보고 racadm get으로 해당 속성명이 무엇으로 바뀌는지 확인한 뒤, 그 속성명을 set 스크립트에 그대로 재사용하는 방식으로 익히는 게 가장 빠르다.

2-5. IPMI / Redfish 명령어 — 초보자를 위한 실습 예시

🔰 이 항목이 왜 헷갈리는지: RACADM은 "Dell 전용" 도구, IPMI와 Redfish는 "업계 표준" 프로토콜이라 다른 회사(HP, Lenovo 등) 서버에서도 문법이 거의 똑같이 통한다. 그래서 여러 제조사 서버를 함께 관리하는 자동화 스크립트에서는 RACADM 대신 IPMI나 Redfish를 쓰는 경우가 많다. 아래는 "전원 상태 확인하기"라는 똑같은 작업을 IPMI(레거시)와 Redfish(최신)로 각각 어떻게 하는지 나란히 보여준다.

사전 준비

항목설명실무 활용
ipmitool 설치yum install ipmitool (RHEL/CentOS) 또는 apt install ipmitool (Ubuntu/Debian)로 설치하는 커맨드라인 도구. 원격 서버 관리 작업을 하는 Linux 서버(배스천 호스트, 자동화 서버 등)에 미리 깔아둔다.이 도구가 없으면 IPMI 명령 자체를 실행할 수 없다 — RACADM처럼 iDRAC 안에 있는 게 아니라, 명령을 "보내는 쪽" 서버에 설치하는 클라이언트 프로그램이라는 점이 초보자가 가장 헷갈리는 부분.
기본 문법 구조ipmitool -I lanplus -H <iDRAC_IP> -U <계정> -P <비밀번호> <하고싶은 명령>

— -U / -P: 로그인 계정/비밀번호		이 한 줄의 구조만 외우면 뒤에 명령어만 바꿔가며 대부분의 IPMI 작업을 할 수 있다.

하고 싶은 일명령어설명
시스템 정보 조회 (GET)curl -k -u root:calvin https://192.168.1.100/redfish/v1/Systems/System.Embedded.1-k는 인증서 검증 생략(사설 인증서라 필요), -u 계정:비밀번호는 로그인 정보. 결과(JSON)에 "PowerState": "On" 같은 필드로 전원 상태가 보인다.
전원 켜기 (POST 액션)curl -k -u root:calvin -X POST -H "Content-Type: application/json" -d '{"ResetType":"On"}' https://192.168.1.100/redfish/v1/Systems/System.Embedded.1/Actions/ComputerSystem.ResetIPMI의 chassis power on과 똑같은 결과를 만드는 Redfish 방식. -X POST는 "무언가를 실행/변경하라"는 뜻, -d 뒤의 JSON이 "어떤 동작을 할지"를 지정.
하드웨어 이벤트 로그 조회curl -k -u root:calvin https://192.168.1.100/redfish/v1/Managers/iDRAC.Embedded.1/LogServices/Sel/EntriesIPMI의 sel list와 같은 정보를 JSON 형식으로 받아온다 — 자동화 스크립트(Python 등)에서 파싱하기 훨씬 편한 구조.
💡 초보자 팁: IPMI 명령은 "명령어를 외워서 그대로 타이핑"하는 방식이고, Redfish는 "URL 경로 + JSON"으로 원하는 리소스에 접근하는 방식이다. 처음엔 IPMI가 더 직관적으로 느껴지지만, 대량 서버 자동화(Python 스크립트 등)에는 구조화된 데이터(JSON)를 주고받는 Redfish가 훨씬 다루기 쉬워 최근 표준으로 자리잡았다.

❓ iDRAC Interview Questions

IPMI와 Redfish의 실무 차이는?
IPMI는 바이너리 기반 구형 표준으로 기능이 제한적(전원제어, 센서 조회 위주)이고 보안 취약점 이슈도 있었다. Redfish는 REST/JSON 기반으로 BIOS, RAID, 펌웨어, 인벤토리까지 표준화된 스키마로 다룰 수 있어 최근 자동화 도구(Ansible, Terraform)들이 Redfish를 우선 지원한다. 신규 구축 환경이라면 Redfish 우선 검토가 실무 표준.
서버가 켜지지도 않고 OS도 응답이 없을 때 첫 확인 지점은?
iDRAC의 SEL 로그와 Virtual Console. iDRAC은 서버 메인 전원과 무관하게 동작하므로, 전원 상태·부팅 시퀀스·POST 에러 코드를 원격에서 그대로 확인할 수 있다. 물리 출동 전 반드시 거치는 1차 진단 단계.
관리망(iDRAC)을 운영망과 분리하는 이유는?
iDRAC은 OS 레벨 보안 정책과 무관하게 하드웨어를 완전히 제어할 수 있는 강력한 권한을 가지므로, 공격자가 접근하면 서버를 통째로 장악할 수 있다. 별도 VLAN/물리망으로 격리해 외부 노출을 최소화하는 것이 보안 표준.
RACADM과 OME의 역할 차이는?
RACADM은 서버 1대의 iDRAC을 CLI로 직접 조회·설정하는 도구이고, OME는 수십~수천 대의 iDRAC을 중앙에서 검색·모니터링·펌웨어 업데이트하는 관제탑(콘솔) 역할을 한다. racadm으로 서버를 OME에 "수동 등록"하는 절차는 없다 — 등록(Discovery)은 항상 OME 콘솔 쪽에서 IP 대역과 계정으로 검색해 이뤄진다.
Remote RACADM과 Local RACADM의 차이는?
Local RACADM은 대상 서버의 OS 안에서 실행하는 방식(iDRAC Service Module 필요, IP/계정 불필요), Remote RACADM은 관리자의 PC에서 iDRAC IP로 원격 접속하는 방식(-r/-u/-p 옵션 필요)이다. 대량 서버를 스크립트로 관리할 때는 대상 서버 OS 상태와 무관하게 동작하는 Remote RACADM을 주로 사용한다.

3. OpenManage Enterprise (OME) Manage many servers, one screen

위에서 배운 iDRAC(리모컨)이 서버 1대짜리라면, OME는 그 리모컨 수십~수천 개를 한 화면에 모아놓은 "관제센터"다. 서버가 5~10대 정도면 iDRAC 화면을 하나씩 열어봐도 되지만, 수백 대가 넘어가면 그건 사실상 불가능하다 — 그래서 OME라는 별도의 소프트웨어(가상머신 형태로 설치)를 하나 띄워두고, 거기서 모든 서버의 상태·경고·펌웨어 버전을 한눈에 보고 한꺼번에 관리한다.

3-1. Overview — What OME Does

항목설명실무 활용언제 사용하는가
중앙 인벤토리/모니터링등록된 모든 서버의 하드웨어 구성, 상태, 경고를 하나의 대시보드에서 확인. 서버 1대씩 iDRAC에 로그인할 필요가 없어짐.🔰 서버 하나하나 로그인하지 않아도 전체 상태를 한 화면에서 볼 수 있게 해주는 기능이다.매우높음서버가 10대를 넘어가는 순간부터 iDRAC 개별 접속 관리는 비효율적 — OME 도입이 사실상 표준.
OME vs iDRAC vs OME-ModulariDRAC=서버 1대 단위 관제, OME=랙 서버 다수를 중앙에서 관리하는 콘솔, OME-Modular=MX7000 등 모듈러 섀시 전용 콘솔(섀시에 내장).🔰 iDRAC은 서버 1대, OME는 서버 여러 대, OME-Modular는 블레이드 섀시 전용 — 관리 범위가 다른 세 가지 도구다.높음면접에서 셋의 역할 차이를 구분해서 설명할 수 있는지 자주 확인.
라이선스 등급 (기본/Advanced/Advanced+)기본은 인벤토리·모니터링 위주, Advanced부터 구성 템플릿·펌웨어 자동 업데이트, Advanced+에서 내부 iDRAC 자격증명 관리 등 고급 기능 제공.🔰 구매한 라이선스 등급에 따라 쓸 수 있는 기능 범위가 달라진다. 등급이 높을수록 자동화 기능이 많아진다.중간대규모 자동화(구성 표준화, 자동 자격증명 로테이션)가 필요한 조직은 Advanced 이상 라이선스 검토.

3-2. Installation / OME Self-Update

단계설명실무 활용언제/왜 하는가
① 배포 방식 선택OME는 가상 어플라이언스(사전 구성된 VM 이미지)로 배포한다. VMware ESXi(OVF/OVA), Microsoft Hyper-V(VHD), KVM 등 주요 하이퍼바이저를 지원.🔰 OME는 직접 설치하는 프로그램이 아니라, 미리 완성된 가상머신 이미지를 통째로 갖다 놓는 방식으로 설치한다.매우높음물리 서버에 직접 설치하는 게 아니라 반드시 가상머신으로 배포 — 사내 가상화 인프라(ESXi 등)가 먼저 준비돼 있어야 한다.
② 리소스 사이징관리 대상 서버 수(예: 수백 대 vs 수천 대)에 따라 vCPU/메모리/디스크 용량을 다르게 할당해야 한다.🔰 관리할 서버가 많을수록 OME 가상머신에도 더 넉넉한 CPU·메모리를 줘야 한다.높음배포 전에 관리 대상 규모를 먼저 파악해 어플라이언스 사이즈를 결정 — 너무 작게 잡으면 나중에 성능 저하.
③ 초기 설정 마법사VM 최초 부팅 후 콘솔/웹UI로 네트워크(IP), 관리자 계정, 시간대, 조직 정보를 설정하는 초기 구성 마법사를 거친다. 이때 "Enable Internal iDRAC credential management" 여부도 선택.🔰 처음 켜면 IP, 관리자 계정 같은 기본 정보를 물어보는 설정 마법사가 뜬다. 처음 한 번만 거치면 된다.매우높음모든 신규 OME 배포에서 반드시 거치는 최초 1회 설정.
④ 웹 콘솔 접속설정한 IP로 HTTPS(443)에 접속해 웹 기반 관리 콘솔을 사용. 이후 모든 작업(검색, 업데이트, 그룹 관리)은 이 웹UI에서 이뤄진다.🔰 설정이 끝나면 그냥 웹브라우저로 그 IP에 접속해서 모든 작업을 하면 된다.매우높음설치가 끝나면 바로 이 화면에서 서버 검색(Discovery)으로 넘어간다.
⑤ OME 자체 버전 업데이트OME 어플라이언스 자체의 소프트웨어 버전은 콘솔 내장 "Console Update" 기능(온라인/오프라인 업데이트 파일 업로드)으로 마이너 업그레이드하거나, 메이저 버전은 새 OVA를 배포하고 기존 설정을 백업/복원(Backup & Restore Appliance)하는 방식으로 이전한다.🔰 OME 자체도 소프트웨어라서 가끔 업데이트가 필요하다. 작은 업데이트는 콘솔 안에서, 큰 버전 업그레이드는 새로 통째로 다시 설치하고 설정을 옮기는 방식으로 한다.중간보안 패치나 신규 기능(예: 신규 서버 세대 지원) 반영을 위해 정기적으로 콘솔 업데이트 확인이 필요.
⑥ 백업/복원어플라이언스 설정, 인벤토리, 자격증명을 정기적으로 백업해두고, 장애나 마이그레이션 시 복원.🔰 OME 설정 자체도 주기적으로 백업해둬야, OME가 고장 나도 복원할 수 있다.중간OME 자체가 다운되면 등록된 모든 서버의 중앙관리 기능이 함께 마비되므로, 어플라이언스 자체의 백업/이중화 계획도 함께 세워야 한다.

📗 신규 설치 — ovftool 무인 배포 명령어

GUI(vSphere Client)로 OVA를 배포해도 되지만, 대량 환경이나 반복 배포가 필요하면 ovftool CLI로 네트워크 설정까지 한 번에 무인 배포할 수 있다.

단계 / 명령어설명실무 활용
① OVA 패키지 내 ovf_properties.config 확인OME OVA 배포 패키지(openmanage_enterprise_ovf_format.zip)를 압축 해제하면 들어있는 설정 파일. 이 안에 IP·호스트명 등 사용 가능한 프로퍼티 키 이름이 정의돼 있다.🔰 자동 배포 스크립트를 짜기 전에 먼저 열어봐야 하는 설정 파일이다. 여기에 어떤 옵션을 쓸 수 있는지 적혀있다.배포 스크립트를 짜기 전에 반드시 먼저 열어봐야 하는 파일 — 버전마다 프로퍼티 키 이름이 조금씩 다를 수 있다.
② 기본 배포 명령ovftool --net:bridged="VM Network" --datastore=datastore1 -dm=thin --name="OME-PROD-01" --noSSLVerify --acceptAllEulas --X:injectOvfEnv --powerOn openmanage_enterprise.ovf vi://administrator@vsphere.local:PASSWORD@esxi-host-or-vcenter/OVA를 지정 데이터스토어에 씬 프로비저닝으로 배포하고 곧바로 전원까지 켜는 기본 명령형. --X:injectOvfEnv --powerOn은 무인 배포 시 OVF 환경변수(IP 등)를 실제로 주입하기 위해 반드시 필요한 플래그다.
③ 고정 IP 주입 (--prop)--prop:vami.ip0.<AppName>=192.168.1.10 --prop:vami.netmask0.<AppName>=255.255.255.0 --prop:vami.gateway.<AppName>=192.168.1.1 --prop:vami.DNS.<AppName>=8.8.8.8초기 설정 마법사에서 수동으로 IP를 입력하는 대신, 배포 명령 한 줄로 고정 IP까지 세팅 — 대량 배포 자동화의 핵심. <AppName> 부분은 ①에서 확인한 실제 프로퍼티 키로 교체해야 한다.
④ 배포 후 최초 접속배포·부팅이 끝나면 설정한 IP로 https://<IP> 접속, 브라우저 기반 초기 설정 마법사(관리자 계정, NTP, 조직 정보)를 완료.🔰 설치가 끝나면 그 IP 주소로 브라우저에서 접속해 첫 설정 마법사를 마치면 된다.ovftool로 네트워크까지 자동 주입했다면 이 단계부터 바로 시작 가능 — IP를 안 넣었다면 VM 콘솔에서 텍스트 UI로 먼저 IP를 설정해야 한다.

📗 업데이트 시 사용하는 메뉴 경로

OME 자체 버전 업데이트는 CLI 명령이 아니라 웹 콘솔의 특정 메뉴에서 진행한다. "어디로 들어가야 하는지"가 핵심이다.

메뉴 경로설명실무 활용
⚙ Application Settings → Console and Extensions → Update Settings업데이트 확인 방식(Automatic/Manual)과 업데이트 소스를 설정하는 첫 화면. Automatic은 매주 월요일 자동으로 새 버전을 확인.🔰 새 버전이 나왔는지 자동으로 확인할지, 내가 직접 확인할지 정하는 화면이다.업데이트를 자동으로 확인만 하고 실제 적용은 수동으로 하고 싶다면 Manual + "다운로드 후 자동 설치" 옵션을 꺼둔다.
업데이트 소스: Online (Dell.com)인터넷이 연결된 환경에서 Dell 공식 서버로부터 직접 최신 버전을 확인·다운로드.🔰 인터넷이 되는 회사라면 그냥 Dell 서버에서 바로 최신 버전을 받아오면 된다.일반적인 사내망(인터넷 아웃바운드 허용) 환경의 기본 선택.
업데이트 소스: Network Share (NFS/HTTP/HTTPS)인터넷이 차단된 폐쇄망에서, 미리 받아둔 업데이트 패키지를 사내 NFS/HTTP/HTTPS 공유 경로에 올려두고 그 경로를 지정.🔰 인터넷이 안 되는 폐쇄망이라면, 다른 곳에서 미리 받아둔 파일을 사내 공유 폴더에 올려두고 그 경로를 알려주면 된다.에어갭 환경에서 업데이트할 때 필수 — 별도 인터넷 PC에서 패키지를 받아 이 공유 경로에 올려두는 절차가 선행돼야 한다.
Console and Extensions → Console and PluginsUpdate Settings에서 소스 설정을 마친 뒤, 실제 업그레이드 작업을 실행하는 화면. 사용 가능한 새 버전이 있으면 여기서 "Update"를 눌러 적용.🔰 새 버전 소스를 정했으면, 이 화면에서 실제로 '업데이트' 버튼을 눌러야 진짜 적용된다.설정(Update Settings)과 실행(Console and Plugins)이 별개 화면으로 나뉘어 있다는 점이 자주 헷갈리는 포인트 — 소스만 설정하고 실제 업데이트 버튼을 못 찾는 경우가 많다.
"다운로드 완료 시 자동으로 업데이트 시작" 옵션패키지 다운로드가 끝나는 즉시 사람 개입 없이 설치·재시작까지 자동 진행할지 여부.🔰 다운로드가 끝나자마자 사람 확인 없이 바로 설치·재시작까지 자동으로 할지 정하는 옵션이다.운영 중인 환경에서는 이 옵션을 꺼두고, 유지보수 창에 맞춰 수동으로 트리거하는 것이 일반적인 실무 관행.

3-3. Server Discovery / Registration

racadm으로 등록하는 게 아니다: OME에 서버가 "등록"되는 과정은 racadm 명령을 실행하는 게 아니라, OME 콘솔에서 능동적으로 검색(Discovery Job)하는 방식이다. 검색 대상 IP 범위와 iDRAC 관리자 계정을 OME에 입력하면, OME가 각 IP로 WSMAN/Redfish/SNMP/IPMI 프로토콜을 통해 접속을 시도해 서버를 찾아내고 인벤토리를 수집한다.
단계설명실무 활용언제 사용하는가
Discovery Job 생성Configuration(또는 Devices) > Discovery 메뉴에서 대상 IP 범위(또는 호스트명 목록)와 사용할 프로토콜(주로 Redfish/WSMAN), iDRAC 계정 정보를 입력해 검색 작업을 생성.🔰 OME 메뉴에서 '이 IP 대역을 검색해줘'라고 요청하는 작업을 등록하는 것이다. 새 서버를 찾는 방법은 이것뿐이다.매우높음새 서버를 랙업한 직후 가장 먼저 하는 작업. 스케줄을 걸어 주기적으로 재검색하면 신규 서버를 자동으로 편입시킬 수도 있다.
내부 iDRAC 자격증명 관리 (Internal iDRAC Credential Management)Advanced 이상 라이선스에서, 최초 검색 성공 후 OME가 각 iDRAC에 "OME_[GUID]" 형식의 전용 서비스 계정을 자동 생성해 이후 통신에 사용. 이렇게 하면 사람이 입력한 최초 계정 정보 없이도 OME가 자체 계정으로 지속 통신 가능.🔰 OME가 서버를 한 번 찾으면, 그 이후엔 자기가 쓸 전용 계정을 자동으로 만들어서 계속 통신하는 고급 기능이다.중간다수의 서버 계정 암호를 일일이 관리하기 번거로운 대규모 환경에서, 계정 로테이션까지 OME에 맡기고 싶을 때(초기 구성 마법사에서만 활성화 가능).
비밀번호 정책 주의사항iDRAC에 Password Strength Policy(복잡도 정책)가 걸려 있으면, OME가 자동 생성한 서비스 계정 비밀번호가 정책을 만족하지 못해 계정이 비활성화될 수 있다.🔰 서버 쪽 비밀번호 규칙이 너무 까다로우면, OME가 자동으로 만든 계정이 오히려 막혀버릴 수 있으니 주의해야 한다.중간내부 자격증명 관리 기능을 켜기 전에 대상 iDRAC들의 비밀번호 정책을 먼저 점검해야 하는 실무 체크포인트.
등록 후 상태 확인검색이 끝나면 Devices 목록에 서버가 나타나고, 정상/경고/심각 등 상태 아이콘으로 헬스 상태를 표시.🔰 검색이 끝나면 목록에 서버가 나타나고, 색깔 아이콘으로 상태(정상/경고/심각)를 바로 확인할 수 있다.높음검색 직후 반드시 상태를 확인해 인증 실패·프로토콜 미지원 등으로 "빠진" 서버가 없는지 점검.
racadm의 실제 역할racadm은 OME 등록 자체와는 무관하지만, 검색 이전에 iDRAC 네트워크(IP)를 설정하거나, SNMP·계정 정책 같은 "검색이 잘 되기 위한 사전 조건"을 개별 서버마다 스크립트로 미리 맞춰두는 보조 역할로 쓰인다.🔰 racadm이 OME에 직접 등록해주는 건 아니지만, 검색이 잘 되도록 서버 쪽 설정을 미리 맞춰두는 보조 도구로는 쓰인다.중간수백 대의 신규 서버를 랙업할 때, racadm 스크립트로 iDRAC IP/계정을 먼저 표준화한 뒤 OME Discovery를 돌리면 검색 성공률이 올라간다.

3-4. Firmware Update (Baseline / Catalog)

단계설명실무 활용언제 사용하는가
카탈로그 소스 선택Dell 온라인 카탈로그(인터넷 연결 필요)를 쓸지, 사내 리포지토리에 미리 받아둔 로컬 카탈로그를 쓸지 선택.🔰 펌웨어 업데이트 파일을 인터넷에서 바로 받을지, 사내에 미리 받아둔 걸 쓸지 정하는 옵션이다.높음인터넷이 차단된 폐쇄망(에어갭) 환경에서는 로컬 카탈로그를 주기적으로 갱신해 사용.
Baseline 생성Configuration > Firmware > Create Baseline에서 카탈로그와 대상 장치 그룹을 지정해 "이 그룹은 이 버전을 기준으로 삼는다"는 기준선을 만든다.🔰 이 서버들은 앞으로 이 펌웨어 버전을 기준으로 삼겠다는 '기준선'을 만드는 작업이다.매우높음동일 모델 서버군의 펌웨어 버전을 표준화하고 싶을 때 — 신규 서버 편입 시에도 같은 Baseline에 묶으면 자동으로 준수 여부가 체크된다.
Compliance Report 확인Baseline과 실제 장치의 현재 펌웨어 버전을 비교해 Upgrade 필요/Critical/Compliant 등의 상태로 보여준다.🔰 정해둔 기준(Baseline)과 실제 서버 버전을 비교해서 '뒤처진 서버'를 한눈에 보여주는 리포트다.매우높음패치 관리 정기 점검 시 "어떤 서버가 뒤처져 있는지"를 한눈에 파악하는 용도.
Remediation (Make Compliant)업데이트가 필요한 장치를 선택해 "Make Compliant"를 실행하면 실제 펌웨어 업데이트 작업이 시작된다 — Baseline을 만든다고 자동으로 업데이트되는 것이 아니라 반드시 이 단계를 수동으로 실행해야 한다.🔰 기준선을 만든다고 자동으로 업데이트되는 게 아니다 — 반드시 이 버튼을 직접 눌러야 실제 업데이트가 시작된다.매우높음유지보수 창에 맞춰 업데이트를 실제로 적용할 때. 즉시 적용(Update Now) 또는 예약 적용 중 선택 가능.
재부팅 필요 여부BIOS/펌웨어 대부분은 적용을 완료하려면 서버 재부팅이 필요하며, OME가 이를 안내하고 예약된 시간에 재부팅까지 자동 수행할 수 있다.🔰 펌웨어 업데이트는 대부분 재부팅이 필요한데, OME가 이 재부팅까지 원하는 시간에 자동으로 해줄 수 있다.높음운영 중인 서비스에 영향이 없도록 재부팅 시점을 새벽 시간대 등으로 예약할 때.

3-5. Monitoring / Groups / Alerts

항목설명실무 활용언제 사용하는가
정적/동적/쿼리 그룹 (Static/Dynamic/Query Group)서버를 수동으로 묶는 정적 그룹, 특정 조건(모델, 위치 등)에 맞으면 자동으로 편입되는 동적 그룹, 실시간 쿼리 조건으로 필터링하는 쿼리 그룹을 지원.🔰 서버를 그룹으로 묶는 방법이 3가지 있다 — 내가 직접 고르거나, 조건에 맞으면 자동으로 들어오거나, 실시간 검색으로 걸러내는 방식이다.높음부서별/랙별/역할별로 서버를 그룹핑해 Baseline이나 알림 정책을 그룹 단위로 일괄 적용하고 싶을 때.
알림 정책 (Alert Policy)특정 심각도·카테고리의 이벤트가 발생하면 이메일, SNMP trap, Syslog 등으로 전달하도록 정책을 정의.🔰 특정 종류의 문제가 생기면 누구에게 어떻게(메일, 알림 등) 알릴지 미리 정해두는 규칙이다.높음수백 대 서버의 경고를 개별적으로 확인하는 대신, 조건에 맞는 이벤트만 담당자에게 자동으로 전달되게 할 때.
구성 템플릿 / 컴플라이언스특정 서버의 BIOS/iDRAC/RAID 설정을 템플릿으로 저장해 다른 서버에 동일하게 배포하고, 이후 설정이 템플릿과 어긋나는지(configuration drift) 정기 검사.🔰 잘 세팅된 서버 하나를 '틀'(템플릿)로 저장해서 다른 서버들에 그대로 복사하고, 나중에 누가 몰래 설정을 바꾸진 않았는지도 계속 확인해준다.높음표준화된 설정을 대량 서버에 일괄 적용하고, 이후 누군가 임의로 설정을 바꾸는 "설정 드리프트"를 감지하고 싶을 때.
리포트인벤토리, 컴플라이언스, 보증기간 만료 임박 등 다양한 사전 정의된/커스텀 리포트를 생성.🔰 자산 현황, 규정 준수 여부, 보증 만료 임박 등 다양한 보고서를 자동으로 뽑아주는 기능이다.중간정기 자산관리 보고, 경영진 보고용 데이터 추출이 필요할 때.

❓ OME Interview Questions

OME에 서버를 어떻게 "등록"하나?
racadm으로 서버 쪽에서 등록 명령을 실행하는 방식이 아니라, OME 콘솔에서 Discovery Job을 만들어 대상 IP 범위와 iDRAC 계정 정보로 능동적으로 검색하는 방식이다. OME가 각 iDRAC에 접속을 시도해 성공하면 자동으로 인벤토리에 등록된다.
Baseline을 만들면 자동으로 펌웨어가 업데이트되나?
아니다. Baseline은 "기준 버전"을 정의하는 것일 뿐이고, Compliance Report로 현재 상태와 기준을 비교한 뒤 관리자가 직접 "Make Compliant"를 눌러야 실제 업데이트 작업이 시작된다. 즉시 적용할지 예약할지도 그 시점에 선택한다.
iDRAC, OME, OME-Modular의 차이를 설명하면?
iDRAC은 서버 1대에 내장된 개별 BMC, OME는 다수의 랙 서버를 검색·모니터링·업데이트하는 중앙 콘솔(별도 VM), OME-Modular는 MX7000 같은 모듈러 섀시에 내장돼 그 섀시 안의 슬레드들을 관리하는 콘솔이다.
폐쇄망(에어갭) 환경에서 펌웨어 업데이트는 어떻게 하나?
Dell 온라인 카탈로그에 접근할 수 없으므로, 인터넷이 되는 별도 PC에서 카탈로그와 펌웨어 파일을 미리 다운로드해 사내 로컬 리포지토리(파일 공유 등)에 올려두고, OME의 카탈로그 소스를 그 로컬 경로로 지정해 Baseline을 구성한다.

4. PERC RAID Controller (PowerEdge RAID Controller)

디스크(하드디스크/SSD) 여러 개를 "안전하게, 또는 빠르게" 묶어서 하나처럼 쓰게 해주는 기술이 RAID다. 예를 들어 디스크 1개만 쓰면 그 디스크가 고장 나는 순간 데이터가 전부 사라지지만, RAID로 여러 개를 묶어두면 디스크 1개가 죽어도 데이터가 안전하게 남아있을 수 있다. 이 작업은 Dell 서버에 꽂힌 PERC라는 전용 카드가 알아서 처리하며, OS(Windows/Linux)는 그 뒤에서 무슨 일이 일어나는지 몰라도 그냥 디스크 하나로 인식하고 쓴다.

17G17세대부터 달라진 점 — PERC12: 17G에서는 PERC12 세대 컨트롤러(H965i Front/Adapter/MX, H965e 등)가 사용된다. PCIe 5.0 연결과 더 높은 IOPS를 지원하며, 캐시 정책(WT/WB/FWB)이나 Patrol Read·Consistency Check 같은 핵심 개념은 이전 세대(PERC11 이하)와 동일하다 — 즉 세대가 바뀌어도 RAID 운용 개념 자체는 그대로 유지된다.

4-1. RAID Levels

RAID 레벨설명 / 동작 방식실무 활용언제 사용하는가
RAID 0 (스트라이핑)데이터를 여러 디스크에 분산 기록해 속도는 최대이나 이중화 없음 — 디스크 1개만 죽어도 전체 데이터 유실.🔰 책 한 권을 여러 명이 한 페이지씩 나눠 베끼면 훨씬 빨리 끝나지만, 그중 한 명이라도 빠지면 책이 완성되지 않는다 — 그게 RAID 0이다. 데이터를 여러 디스크에 나눠 기록해(스트라이핑) 속도는 최대지만 이중화(안전장치)가 전혀 없다 — 디스크 1개만 죽어도 전체 데이터가 통째로 사라진다.낮음없어져도 괜찮은 임시/캐시성 데이터, 순수 성능 벤치마크 용도.
RAID 1 (미러링)동일 데이터를 2개 디스크에 그대로 복제. 디스크 1개 장애까지 허용, 실사용 용량은 50%.🔰 중요한 문서를 프린트 2장 뽑아서 각각 다른 서랍에 보관하는 것과 같다 — 서랍 하나가 불타도 나머지 한 장이 살아남는다. 동일한 데이터를 디스크 2개에 그대로 복제(미러링)해서 저장한다. 디스크 1개가 고장 나도 데이터는 안전하지만, 디스크 2개 값을 다 썼는데 실제 쓸 수 있는 용량은 1개 분량뿐이다(50%).매우높음OS 부트 디스크 등 소용량·고가용성이 필요한 볼륨의 기본 구성.
RAID 5패리티 1개 분산 저장, 디스크 1개 장애까지 허용. 용량 효율 좋으나 쓰기 시 패리티 계산 오버헤드(Write Penalty) 존재.🔰 여러 사람이 각자 원고를 나눠 쓰되, "누가 빠졌을 때 나머지끼리 힌트를 모아 복원할 수 있는 메모"(패리티)를 하나 더 만들어두는 방식이다. 이 패리티 정보를 디스크 여러 개에 나눠 저장해두면, 디스크 1개가 고장 나도 나머지 디스크들의 정보로 원본을 복원할 수 있다. 용량 효율은 좋지만 쓸 때마다 이 패리티를 계산하는 오버헤드(Write Penalty)가 있다.높음읽기 위주 워크로드(파일서버, 백업 저장소)의 비용 대비 효율적인 구성.
RAID 6패리티 2개 분산 저장, 디스크 2개까지 동시 장애 허용. RAID5보다 안전하지만 쓰기 오버헤드가 더 큼.🔰 RAID 5의 "복원용 메모"(패리티)를 한 개가 아니라 두 개 만들어두는 방식이라, 디스크가 동시에 2개까지 고장 나도 데이터를 지킬 수 있다. 대신 그만큼 쓰기 작업의 부담(오버헤드)이 RAID 5보다 더 크다.높음대용량 디스크(고장 난 디스크를 복구하는 데 오래 걸리는)로 구성된 스토리지에서 이중 장애까지 대비할 때.
RAID 10 (1+0)미러링된 쌍을 스트라이핑. 성능과 가용성을 모두 확보하지만 실사용 용량은 50%.🔰 위에서 배운 "복사본 보관"(미러링, RAID 1)과 "나눠서 빨리 쓰기"(스트라이핑, RAID 0)를 합친 방식이다 — 안전하면서도 빠르지만, 미러링을 하는 만큼 실제 쓸 수 있는 용량은 절반(50%)이다.매우높음DB(트랜잭션 로그, 데이터파일) 등 읽기/쓰기 성능과 안정성이 동시에 중요한 워크로드의 표준 선택.
RAID 50 / 60RAID5/6 세트를 여러 개 묶어 스트라이핑한 중첩 RAID. 대용량 + 성능 + 부분 이중화를 함께 노림.🔰 RAID 5나 6으로 묶은 "작은 그룹"을 여러 개 만든 다음, 그 그룹들을 다시 한 번 스트라이핑으로 묶는 "그룹의 그룹" 구조다. 대용량 + 성능 + 어느 정도의 이중화를 동시에 노리는 절충안.중간디스크 수가 많은 대용량 스토리지 어레이에서 RAID5/6 단일 세트의 한계를 넘어서고 싶을 때.

4-2. Cache / Read-Write Policy

설정명설명 / 동작 방식실무 활용언제 사용하는가
① Write Through (WT)디스크에 실제로 쓰기가 완료된 후에만 컨트롤러가 호스트에 "완료" 신호를 보냄. 데이터 안정성이 가장 높지만 쓰기 성능은 느림.🔰 택배를 보낼 때 "상대방이 실제로 받았다는 확인 문자를 받은 뒤에야" 내 할 일이 끝났다고 여기는 방식이다. 디스크에 실제로 쓰기가 완료된 후에만 컨트롤러가 호스트에 "완료" 신호를 보낸다. 데이터 안정성이 가장 높지만(확실하게 확인하니까) 쓰기 성능은 느리다(매번 기다려야 하니까).높음BBU/CacheVault가 아예 없거나 방전 상태일 때 컨트롤러가 안전을 위해 자동으로 강제 전환하는 정책이기도 함. 데이터 무결성이 최우선인 경우 의도적으로 선택.
② Write Back (WB) — 기본값데이터가 컨트롤러 캐시에 들어오는 즉시 "완료" 신호를 보내고 이후 백그라운드로 디스크에 기록. 단, BBU/CacheVault 상태를 실시간으로 감시해 배터리가 없거나 방전되면 컨트롤러가 자동으로 Write Through로 되돌아가는 "안전한" 버전.🔰 택배기사가 "일단 아파트 경비실에 맡겨두고" 바로 다음 배송을 가는 방식이다(경비실=컨트롤러의 임시 저장공간, 캐시). 데이터가 컨트롤러 캐시에 들어오는 즉시 "완료" 신호를 보내고 이후 천천히 디스크에 실제로 기록한다 — 훨씬 빠르다. 단, 정전되면 경비실에 맡겨둔 물건(캐시)이 날아갈 수 있어서, 배터리(BBU/CacheVault)가 없거나 방전되면 자동으로 위의 안전한 방식(Write Through)으로 되돌아간다.매우높음대부분의 프로덕션 워크로드의 기본 선택. 성능과 안전성의 균형이 맞는 표준 옵션 — 배터리 상태가 나빠지면 자동으로 WT로 폴백해 데이터를 보호한다.
③ Force Write Back / Always WB (FWB)배터리(BBU/CacheVault)가 없거나 방전 상태여도 상관없이 항상 강제로 Write Back 모드를 유지. WB처럼 배터리 상태에 따라 자동으로 WT 전환되지 않음.🔰 위의 "경비실에 맡기고 바로 가기" 방식을, 경비실(배터리)이 없거나 고장 나 있어도 무조건 그대로 밀어붙이는 버전이다. 배터리(BBU/CacheVault)가 없거나 방전 상태여도 상관없이 항상 강제로 Write Back 모드를 유지하며, WB처럼 상황에 따라 자동으로 안전 모드(WT)로 돌아가지 않는다 — 그래서 정전 시 데이터를 잃을 위험이 그대로 남는다.낮음배터리 없는 컨트롤러에서도 최대 쓰기 성능이 반드시 필요한 특수 상황(예: UPS로 전원 안정성이 이미 보장된 랩·테스트 환경)에서만 신중히 사용 — 정전 시 캐시 데이터 유실 위험을 감수하는 옵션이라 프로덕션에서는 권장되지 않음.
Read Ahead / Adaptive Read Ahead / No Read Ahead순차 읽기 패턴을 감지해 다음 섹터를 미리 읽어 캐시에 적재. Adaptive는 최근 읽기 패턴이 순차적일 때만 자동으로 Read Ahead를 적용.🔰 책을 순서대로 읽고 있는 걸 눈치채면 다음 페이지를 미리 넘겨서 준비해두는 것과 같다. 순차 읽기 패턴을 감지해 다음 섹터를 미리 읽어 캐시에 적재해둔다. Adaptive는 최근 읽기 패턴이 실제로 순차적일 때만 이 미리읽기를 자동으로 켠다.높음순차 읽기가 많은 워크로드(백업, 미디어 스트리밍)는 Read Ahead, 랜덤 I/O가 많은 OLTP DB는 No Read Ahead가 유리한 경우가 많음.
Disk Cache Policy (물리 디스크 자체 캐시)HDD/SSD 자체 내장 캐시 사용 여부(Enabled/Disabled/Default). 컨트롤러 캐시와는 별개로 디스크 자체 캐시를 켤지 결정.🔰 위에서 배운 RAID 컨트롤러의 캐시(경비실)와는 별개로, 디스크 자체에도 작은 임시 저장공간(캐시)이 내장돼 있다. 이걸 쓸지 말지 정하는 옵션이다. HDD/SSD 자체 내장 캐시 사용 여부(Enabled/Disabled/Default).중간BBU로 보호되지 않는 디스크 자체 캐시까지 켜면 정전 시 유실 위험이 커지므로, 데이터 안정성이 중요하면 Disabled 권장.
Stripe Size데이터를 나눠 쓰는 블록 단위 크기(64KB~1MB 등). I/O 패턴에 따라 성능이 달라짐.🔰 데이터를 여러 디스크에 나눠 쓸 때 "한 번에 몇 조각씩 자를지"를 정하는 값(64KB~1MB 등)이다. 큰 조각으로 자르면 큰 파일 통짜로 옮길 때 유리하고, 작은 조각으로 자르면 자잘한 요청이 많은 작업에 유리하다.중간대용량 순차 I/O(백업, 미디어)는 큰 스트라이프, 소용량 랜덤 I/O(DB)는 작은 스트라이프가 유리한 경향.
BBU / CacheVault (배터리·커패시터 백업)정전 시에도 컨트롤러 캐시의 데이터를 잃지 않도록 배터리(BBU) 또는 커패시터+플래시(CacheVault) 방식으로 캐시를 보존. 방전/고장 상태면 컨트롤러가 자동으로 Write Back → Write Through로 강제 전환하는 안전장치가 동작.🔰 위에서 계속 나온 "경비실"(캐시)이 정전돼도 안에 있는 물건이 안전하도록 지켜주는 비상용 배터리 같은 장치다. 정전 시에도 컨트롤러 캐시의 데이터를 잃지 않도록 배터리(BBU) 또는 커패시터+플래시(CacheVault) 방식으로 캐시를 보존한다. 이 배터리가 방전되거나 고장 나면, 컨트롤러가 알아서 Write Back(빠른 방식) → Write Through(느리지만 안전한 방식)로 전환한다.매우높음Write Back 캐시를 신뢰성 있게 쓰려면 필수 전제조건. 면접에서 "왜 Write Back인데 갑자기 성능이 떨어졌나"류 질문의 핵심 답이 되는 항목(배터리 방전 → 자동 WT 전환).

4-3. Maintenance / Availability Features

설정명설명 / 동작 방식실무 활용언제 사용하는가
Patrol Read유휴 시간에 백그라운드로 전체 물리 디스크를 스캔해 배드섹터를 조기 발견·복구.🔰 디스크가 한가할 때 전체를 미리 훑어보며 불량 섹터를 조기에 찾아 고쳐두는 건강검진 기능이다.높음운영 중인 모든 RAID 어레이에서 기본 활성화 권장 — 실 장애 전에 결함 섹터를 미리 잡아냄.
Consistency Check패리티 기반 RAID(5/6 등)에서 실제 데이터와 패리티 값의 일관성을 정기적으로 검증·복구.🔰 위에서 배운 '복원용 메모'(패리티)가 실제 데이터와 잘 맞는지 정기적으로 대조 검사하는 기능이다.높음정기 유지보수 스케줄(주1회 등)로 자동 실행 설정, 사일런트 데이터 오류를 예방.
Rebuild Rate디스크 장애 후 교체 디스크로 데이터를 재구성하는 속도(우선순위). 높이면 리빌드는 빨라지지만 운영 I/O 성능에 영향.🔰 고장 난 디스크를 새 디스크로 복구할 때 얼마나 빨리(공격적으로) 복구할지 정하는 속도 조절 옵션이다.높음장애 디스크 교체 직후 리빌드 중에도 서비스 영향을 최소화해야 하면 Rate를 낮추고, 빠른 복구가 우선이면 높임.
Hot Spare (Dedicated / Global)장애 발생 시 자동으로 투입되는 예비 디스크. Dedicated는 특정 어레이 전용, Global은 여러 어레이가 공유.🔰 디스크가 고장 나면 사람 없이도 자동으로 교체 투입되는 예비 디스크다.높음무인 운영 서버에서 디스크 장애 시 사람이 개입하기 전에 자동 리빌드를 시작시키고 싶을 때.
CacheCade (SSD 캐시)SSD를 HDD 어레이의 2차 캐시(read/write cache)로 사용해 성능을 끌어올리는 기능.🔰 빠른 SSD를 느린 HDD 앞에 '보조 캐시'로 세워둬서 속도를 끌어올리는 기능이다.낮음HDD 기반 대용량 스토리지에 소량의 SSD만 추가해 핫데이터 접근속도를 높이고 싶을 때.
JBOD 모드디스크를 RAID로 묶지 않고 컨트롤러가 있는 그대로(개별 디스크)를 OS에 그대로 노출.🔰 디스크를 RAID로 묶지 않고 하나씩 그대로 OS에 보여주는 모드다.중간ZFS, Ceph, Ceph OSD처럼 소프트웨어 정의 스토리지가 자체적으로 디스크를 직접 관리해야 하는 구성에서 필수.
Foreign Configuration Import다른 서버/컨트롤러에서 사용하던 디스크를 그대로 장착했을 때, 기존 RAID 구성 정보(메타데이터)를 그대로 가져와 볼륨을 복원.🔰 다른 서버에서 쓰던 디스크를 그대로 옮겨왔을 때, 그 안에 남아있던 RAID 정보를 그대로 읽어와 복원해주는 기능이다.중간서버 장애로 디스크를 통째로 다른 새 서버에 옮겨 데이터를 복구해야 하는 상황.
물리 디스크 상태 (Online/Offline/Rebuilding/Failed/Foreign)RAID를 구성하는 각 물리 디스크의 현재 상태를 나타내는 값. Rebuilding은 리빌드 진행 중, Foreign은 다른 컨트롤러 이력이 남아있는 상태.🔰 각 디스크가 지금 정상인지, 복구 중인지, 다른 서버에서 왔는지를 보여주는 상태 표시다.매우높음장애 조치의 가장 기본이 되는 확인 지점 — 디스크 교체 후 상태가 Online으로 정상 복귀했는지 반드시 확인.
Secure Erase디스크 반납/폐기 전 데이터를 복구 불가능하게 완전 삭제.🔰 디스크를 반납하거나 버리기 전에 안의 데이터를 아예 복구 불가능하게 완전히 지워버리는 기능이다.중간디스크 교체/서버 폐기 시 데이터 유출 방지를 위한 보안 절차(컴플라이언스 요구사항).
T10 PI (Protection Information / DIF)디스크에 데이터를 쓸 때마다 무결성 체크섬을 함께 저장해, 조용한 데이터 손상(silent corruption)을 감지.🔰 데이터를 쓸 때마다 '이게 맞다'는 확인 도장을 함께 찍어둬서, 아무도 모르게 조용히 깨지는 데이터를 잡아내는 기능이다.낮음엔터프라이즈 SAS 디스크 기반의 미션크리티컬 스토리지에서 데이터 무결성을 최우선으로 할 때.

❓ RAID Interview Questions

Write Back인데 갑자기 쓰기 성능이 뚝 떨어졌다면?
일반 Write Back(WB)은 BBU/CacheVault 상태를 실시간 감시하다가 배터리가 방전·고장 상태가 되면 데이터 안정성을 위해 자동으로 Write Through로 강제 전환한다. 이 경우 iDRAC/PERC 상태에서 배터리 상태(Learning cycle 포함)를 먼저 확인하는 것이 정석. 참고로 Force Write Back(FWB)으로 설정돼 있었다면 배터리 상태와 무관하게 계속 WB를 유지하므로 이 증상 자체가 나타나지 않는다 — 대신 정전 시 데이터 유실 위험을 그대로 안고 가는 것이다.
RAID5와 RAID10 중 DB 서버에는 어떤 걸 쓰는가?
RAID10이 일반적으로 선호된다. RAID5는 쓰기 시 패리티 계산 때문에 Write Penalty가 커서 랜덤 쓰기가 많은 OLTP 워크로드에 불리하고, 디스크 1개 장애 후 리빌드 중 2차 장애 위험(특히 대용량 디스크)도 크다. RAID10은 용량 효율은 떨어지지만 쓰기 성능과 리빌드 안정성이 더 우수하다.
WT/WB/FWB 3가지 캐시 정책의 차이를 한 문장으로?
WT는 항상 안전하지만 느리고, WB는 빠르면서도 배터리 상태를 감시해 위험할 때 자동으로 안전모드(WT)로 전환하며, FWB는 배터리 상태와 무관하게 항상 빠른 모드를 강제 유지해 위험을 감수한다.

5. Storage Array (PowerVault ME5 / PowerStore, etc.)

앞서 배운 RAID는 서버 "한 대 안에" 있는 디스크를 묶는 것이고, 여기서 다루는 스토리지 어레이는 서버 밖에 따로 있는 "공용 디스크 창고"다. 서버 여러 대가 케이블로 이 창고에 연결돼서 필요한 만큼 공간을 나눠 쓴다 — 마치 각자 컴퓨터를 쓰면서 같은 공용 하드디스크(NAS)를 나눠 쓰는 것과 비슷한 개념이라고 보면 된다. 여러 서버가 함께 쓰는 만큼, 누구에게 얼마나 나눠줄지(볼륨), 자주 쓰는 데이터는 빠른 디스크에 두는지(티어링) 같은 개념들이 새로 등장한다.

5-1. Pool / Volume / Mapping

설정명설명 / 동작 방식실무 활용언제 사용하는가
디스크 풀 (Pool) / 디스크 그룹여러 물리 디스크를 묶어 만드는 저장 공간의 최소 관리 단위. Virtual Pool(가상화·티어링 지원)과 Linear Pool(고정 매핑, 예측 가능한 성능) 두 방식이 있음.🔰 여러 디스크를 하나의 '큰 저장 창고'로 묶어놓은 단위다. 여기서 필요한 만큼 잘라서 서버에 나눠준다.매우높음스토리지 초기 구성 시 가장 먼저 설계하는 항목. 대부분 최신 환경은 Virtual Pool을 사용.
Thin Provisioning (씬 프로비저닝)볼륨을 실제 사용량만큼만 물리 공간을 할당하며, 논리적으로는 더 큰 용량처럼 보이게 함.🔰 겉보기엔 큰 용량처럼 보이지만, 실제로 쓴 만큼만 진짜 공간을 잡아먹는 절약 방식이다.높음용량 예측이 어려운 다수 볼륨을 운영할 때 물리 디스크 구매 비용을 아끼기 위해 사용. 단, 과다할당(overcommit) 모니터링 필요.
볼륨(Volume) & 호스트 매핑 (Host/Hostgroup Mapping)풀에서 잘라낸 논리 볼륨을 특정 호스트(서버)의 initiator(iSCSI IQN, FC WWN, SAS 주소)에 매핑해야 해당 서버가 LUN으로 인식.🔰 창고에서 잘라낸 저장공간(볼륨)을 특정 서버에게 '이건 네 거야'라고 연결해줘야 그 서버가 디스크로 인식한다.매우높음신규 서버에 스토리지 볼륨을 연결하는 모든 작업의 필수 단계. 매핑이 잘못되면 멀티패싱 이중화가 깨지거나 다른 서버가 볼륨을 잘못 인식하는 사고로 이어짐.
Provisioning Type — Standard vs Virtual(Thin) Volume볼륨 생성 시 처음부터 전체 용량을 물리적으로 예약(Standard/Full)할지, 사용한 만큼만 할당(Thin)할지 선택.🔰 볼륨을 만들 때 처음부터 전체 공간을 확보해둘지, 쓰는 만큼만 나눠줄지 정하는 옵션이다.중간용량 예측이 정확하고 성능 일관성이 중요한 DB는 Full Provisioning, 유연성이 필요한 범용 파일서버는 Thin.

5-2. Tiering / Snapshot / Replication

설정명설명 / 동작 방식실무 활용언제 사용하는가
Tier Affinity (계층 친화도) — No Affinity / Performance / ArchiveVirtual Pool 내에서 데이터를 SSD 계층(고성능)과 HDD 계층(대용량)에 자동 분산 배치. Performance는 핫데이터를 SSD에 우선 배치, Archive는 콜드데이터를 HDD에 유지.🔰 자주 쓰는 데이터는 빠른 SSD에, 잘 안 쓰는 데이터는 저렴한 HDD에 자동으로 나눠 담아주는 똑똑한 기능이다.높음비용 효율과 성능을 동시에 잡아야 하는 하이브리드(SSD+HDD) 구성에서, 접근 빈도가 다른 데이터를 자동으로 최적 계층에 배치할 때.
스냅샷 (Snapshot, Redirect-on-Write)특정 시점의 볼륨 상태를 즉시 캡처. Redirect-on-Write 방식은 원본을 건드리지 않고 변경분만 새 위치에 기록해 스냅샷 생성이 매우 빠르고 공간 효율적.🔰 지금 이 순간의 상태를 그대로 사진 찍어두는 기능이다. 원본은 그대로 두고 바뀐 부분만 따로 기록해서 매우 빠르고 공간도 적게 든다.높음패치/업그레이드 직전 즉시 롤백 지점 확보, 실수로 인한 데이터 삭제·변경 복구용.
볼륨 복제 (Volume Copy) / 원격 복제 (Remote Replication)볼륨 전체를 다른 풀 또는 원격 사이트로 동기/비동기 복제.🔰 저장된 내용 전체를 다른 곳(다른 풀이나 아예 다른 지역)으로 복사해두는 기능이다.중간DR(재해복구) 사이트 구성, 백업 오프로드 목적.
압축 / 중복제거 (Compression / Deduplication)저장되는 데이터의 실제 물리 용량을 줄여 스토리지 효율을 높이는 기능. 스토리지 컨트롤러 CPU 자원을 추가로 소모.🔰 중복되는 데이터를 압축·정리해서 실제로 차지하는 공간을 줄여주는 기능이다. 대신 컨트롤러가 조금 더 일을 해야 한다.중간백업, VDI처럼 중복 데이터 비중이 높은 워크로드에서 스토리지 비용을 절감하고 싶을 때.

5-3. Availability / Cache / Monitoring

설정명설명 / 동작 방식실무 활용언제 사용하는가
캐시 Write Policy (어레이 레벨)컨트롤러 캐시의 Write-back/Write-through 정책. PERC와 개념은 동일하지만 어레이 컨트롤러 이중화(듀얼 컨트롤러) 구조와 결합해 동작.🔰 위에서 배운 RAID 캐시 정책(빠르게 vs 안전하게)과 같은 개념인데, 여기선 컨트롤러가 2개(이중화)로 동작한다는 점이 다르다.높음Write-back은 성능 위해 기본 사용, 컨트롤러 이중화·배터리 상태가 정상일 때만 안전하게 유지.
멀티패싱 (MPIO) / 호스트 포트 프로토콜 (iSCSI/FC/SAS)서버-스토리지 간 경로를 이중화해 하나의 경로 장애 시에도 자동 failover. 프로토콜에 따라 iSCSI(이더넷 기반), FC(전용 SAN), SAS(직결) 방식 선택.🔰 서버와 스토리지 사이 연결 케이블(경로)을 여러 개 만들어둬서, 하나가 끊겨도 나머지로 자동 전환되게 하는 이중화 기능이다.매우높음스토리지 연결의 가용성 확보를 위해 사실상 필수. 경로 하나만 구성하면 컨트롤러/케이블/HBA 장애 시 즉시 서비스 영향.
Global / Dedicated Spare (스토리지 레벨)디스크 장애 시 자동 투입되는 예비 디스크를 풀 전체가 공유(Global)할지 특정 디스크그룹 전용(Dedicated)으로 둘지 결정.🔰 고장 나면 자동으로 투입되는 예비 디스크를, 여러 그룹이 같이 쓸지 특정 그룹만 쓸지 정하는 옵션이다.높음무인 운영 환경에서 장애 대응 속도를 높이기 위해 최소 1개 이상의 스페어 확보가 표준.
듀얼 컨트롤러 이중화 (Active/Active, Active/Standby)스토리지 컨트롤러 모듈 2개가 각각 볼륨을 소유(Active/Active)하거나 하나가 대기(Standby)하며, 한쪽 장애 시 다른 쪽이 볼륨 소유권을 인계(failover).🔰 스토리지 안의 두뇌(컨트롤러)가 2개 있어서, 하나가 죽어도 나머지가 즉시 이어받아 서비스가 안 끊기게 해주는 이중화 구조다.매우높음스토리지 컨트롤러 장애로 인한 전체 서비스 중단을 막기 위한 필수 이중화 구성.
Pool 용량 임계치 알림 (Threshold Alert)풀의 사용률이 지정한 %(예: 85%, 95%)를 넘으면 관리자에게 경고.🔰 저장공간이 거의 다 찼을 때(예: 85%, 95%) 미리 경고해주는 알림 기능이다.높음씬 프로비저닝 환경에서 예상치 못한 용량 고갈(Out of Space) 장애를 사전에 막기 위한 필수 모니터링 설정.

집에서 쓰는 공유기가 컴퓨터·TV·휴대폰을 서로 연결해주듯, 데이터센터의 스위치는 서버 수십~수백 대를 서로, 그리고 인터넷과 연결해주는 훨씬 더 크고 전문적인 "네트워크 허브"다. 다만 일반 공유기와 달리 여러 부서 트래픽을 분리하거나(VLAN), 케이블 하나가 끊겨도 안 끊기게 이중화하는(포트채널) 등 훨씬 정교한 설정이 들어간다. Dell 스위치는 최신 OS10이라는 자체 운영체제로 명령어를 입력해 설정하며, 문법은 Cisco 장비와 비슷한 편이다.

17G17세대부터 달라진 점 — 산업표준 NIC: 16G까지는 서버에 장착되는 NIC이 Dell 전용으로 커스터마이징된 펌웨어/드라이버를 쓰는 경우가 많았지만, 17G부터는 벤더(Broadcom/Intel/NVIDIA 등)의 네이티브 펌웨어·드라이버를 그대로 사용하는 "산업표준 NIC"으로 전환됐다. 관리·텔레메트리 정보는 표준 프로토콜을 통해 iDRAC에 노출되므로, 드라이버 호환성 확인 시 Dell 전용 다운로드보다 벤더 공식 배포판도 함께 참고해야 하는 경우가 늘었다.

6-1. Basic Switching

설정명설명 / 동작 방식실무 활용언제 사용하는가
VLAN물리적으로 하나의 스위치를 논리적으로 여러 개의 독립된 네트워크로 분리. switchport access vlan / switchport trunk allowed vlan으로 설정.🔰 물리적으로 스위치는 하나인데, 논리적으로 여러 개의 분리된 네트워크로 나눠 쓰는 기술이다.매우높음관리망/운영망/스토리지망 분리, 멀티테넌트 환경에서 트래픽 격리가 필요할 때 — 거의 모든 데이터센터 네트워크의 기본 전제.
Port-Channel / LACP (channel-group ... mode active)여러 물리 포트를 하나의 논리 링크로 묶어 대역폭 합산과 링크 이중화를 동시에 제공. LACP(802.3ad)는 동적 협상 프로토콜로 Active/Passive 모드 지원.🔰 여러 케이블(포트)을 하나로 묶어서 속도도 합치고 하나가 끊겨도 안전하게 만드는 기술이다.매우높음서버 NIC 이중화(팀/본딩)와 스위치 업링크 이중화 모두에서 표준적으로 사용. 서버의 bonding mode 4(802.3ad)와 반드시 짝을 맞춰 구성해야 함.
VLT (Virtual Link Trunking)Dell 고유의 MLAG(Multi-chassis LAG) 기술. 물리적으로 분리된 스위치 2대를 서버/이웃 장비 입장에서 논리적으로 하나의 스위치처럼 보이게 해, 스위치 자체의 장애까지 이중화.🔰 물리적으로 다른 스위치 2대를, 서버 입장에서는 마치 하나의 스위치처럼 보이게 만들어 스위치가 통째로 죽어도 안전한 기술이다.높음스위치 한 대가 통째로 다운돼도 서비스 영향 없이 이중화된 업링크를 유지해야 하는 코어/액세스 계층 설계.
Spanning Tree (STP/RSTP/RPVST+)이중화 구성에서 발생할 수 있는 루프를 감지해 특정 경로를 논리적으로 차단, 브로드캐스트 스톰 방지. OS10 기본값은 RPVST+(VLAN별 Rapid STP).🔰 네트워크에 케이블이 뱅글뱅글 도는 루프가 생기지 않게, 한쪽 길을 미리 막아두는 안전장치다.높음다중 업링크·이중화 스위치 환경에서 루프 방지가 필요할 때. VLT를 쓰면 루프 자체가 설계상 발생하지 않아 STP 의존도가 줄어드는 경우도 많음.

6-2. QoS / Security

설정명설명 / 동작 방식실무 활용언제 사용하는가
QoS / DCB (PFC, ETS)트래픽 클래스별 우선순위 지정. Data Center Bridging의 PFC(Priority Flow Control)는 특정 트래픽 클래스만 선택적으로 흐름제어해 무손실(lossless) 전송을 보장.🔰 중요한 트래픽에게 먼저 지나갈 권리를 주는 우선순위 조정 기술이다.중간iSCSI/FCoE 등 스토리지 트래픽처럼 패킷 드랍에 민감한 프로토콜을 이더넷 위에서 안정적으로 돌려야 할 때.
ACL (Access Control List)특정 IP/포트/프로토콜 기준으로 트래픽 허용/차단 규칙을 스위치 레벨에서 적용.🔰 특정 IP나 포트로 가는 트래픽을 허용하거나 막는 출입 규칙이다.중간네트워크 장비 레벨에서 세그먼트 간 접근을 통제해야 하는 보안 요구사항이 있을 때.
Storm Control브로드캐스트/멀티캐스트/유니캐스트 트래픽이 비정상적으로 폭증하면 해당 포트의 트래픽을 자동으로 제한.🔰 특정 트래픽이 갑자기 폭증하면 자동으로 그 포트를 제한해 네트워크 전체가 마비되는 걸 막아준다.중간루프나 오작동 장비로 인한 브로드캐스트 스톰이 전체 네트워크를 마비시키는 것을 막는 보호장치.
Port Security특정 포트에 연결이 허용되는 MAC 주소 개수/목록을 제한.🔰 이 포트에는 정해진 장비(MAC 주소)만 연결되도록 제한하는 보안 기능이다.중간비인가 장비의 물리적 네트워크 접속을 막아야 하는 보안 요구사항이 있는 액세스 계층.

6-3. Operations / Monitoring

설정명설명 / 동작 방식실무 활용언제 사용하는가
OOB 관리 인터페이스 (전용 관리 포트)스위치 자체 관리 전용 물리 포트로, 데이터 트래픽과 완전히 분리된 대역외(Out-of-Band) 관리망을 구성.🔰 스위치 관리 전용으로 따로 빼둔 케이블 구멍이다. 운영 트래픽과 완전히 분리돼 있어 장애 중에도 접속 가능하다.높음운영 트래픽 폭주/장애 상황에서도 스위치에 접속해 문제를 진단·복구해야 할 때 필수.
SmartFabric / Spine-Leaf 설계OS10 기반의 자동화된 패브릭 구성(SmartFabric)과, 확장성이 뛰어난 Spine-Leaf 토폴로지(모든 리프가 모든 스파인에 연결).🔰 대규모 데이터센터에서 스위치를 자동으로, 확장하기 쉬운 구조로 연결하는 설계 방식이다.중간대규모 가상화/클라우드 환경에서 동일한 지연시간과 확장성을 보장하는 네트워크 설계가 필요할 때.
MTU / Jumbo Frame이더넷 프레임 최대 크기를 기본 1500바이트보다 크게(보통 9000바이트) 설정해 헤더 오버헤드를 줄이고 처리량을 높임.🔰 한 번에 보내는 데이터 조각(프레임)을 더 크게 만들어서 전체적으로 더 효율적으로 전송하는 기술이다.높음iSCSI 스토리지 네트워크, vMotion/실시간 마이그레이션 트래픽 등 대용량 전송 구간에서 성능 향상을 위해 사용 — 단, 경로상 모든 장비(서버 NIC·스위치)가 동일하게 설정돼야 함.
Uplink Failure Detection (UFD)액세스 스위치의 업링크(코어 방향) 포트가 모두 죽으면, 하위(서버 방향) 포트도 강제로 down시켜 서버의 NIC 이중화 소프트웨어가 장애를 인지하고 다른 경로로 failover하게 만듦.🔰 스위치의 상위 연결이 다 끊기면, 서버 쪽 포트도 일부러 꺼서 서버가 문제를 눈치채고 다른 길로 갈아타게 해주는 기능이다.중간서버는 정상인데 스위치의 업링크만 끊겨 트래픽이 블랙홀에 빠지는 상황을 막기 위해 이중화 설계에서 함께 구성.
sFlow / 트래픽 미러링 (SPAN/Port Mirroring)스위치를 통과하는 트래픽의 샘플(sFlow) 또는 전체 사본(SPAN)을 모니터링 장비로 복제 전송.🔰 스위치를 지나가는 트래픽의 일부(또는 전체)를 복사해서 모니터링 장비로 보내주는 감시 기능이다.중간IDS/IPS, 패킷 캡처 분석 등 트래픽 가시성 확보가 필요한 보안·성능 진단 상황.

❓ Network Interview Questions

서버 NIC 본딩과 스위치 설정이 안 맞으면 어떤 문제가 생기나?
서버가 802.3ad(LACP) 모드로 본딩돼 있는데 스위치 포트가 일반 액세스 포트(포트채널 미구성)라면 LACP 협상 자체가 안 돼 링크가 불안정해지거나 특정 방향 트래픽만 유실될 수 있다. 반드시 서버 bonding mode와 스위치 Port-Channel/LACP 모드를 짝지어 구성해야 한다.
VLT와 일반 STP 이중화의 차이는?
일반 STP 이중화는 루프를 막기 위해 한쪽 경로를 논리적으로 차단(blocking)해 실질적으로 대역폭을 절반만 활용한다. VLT(MLAG)는 두 스위치를 논리적으로 하나처럼 만들어 양쪽 링크를 모두 Active로 동시에 사용하면서도 루프 없이 이중화할 수 있어 대역폭 효율이 더 좋다.

7. OS-Level Settings (Linux Server Tuning Focus)

지금까지 본 BIOS·iDRAC·RAID·스토리지·네트워크는 전부 "하드웨어를 어떻게 준비할지"에 관한 이야기였다면, 이제부터는 그 하드웨어 위에서 실제로 돌아가는 리눅스(Linux) 서버를 "어떻게 다룰지"에 대한 이야기다. 우리가 흔히 "서버 관리한다"고 할 때 매일 하는 작업(로그 확인, 디스크 정리, 네트워크 점검, 명령어 입력)이 대부분 이 탭에 들어있다. 개념을 줄줄 외우는 것보다 "이 값을 왜 이렇게 바꾸는지" 이유를 이해하는 게 훨씬 중요하다.

파일 / 디렉토리

명령어설명실무 활용 예시
ls -al디렉토리 내 모든 파일(숨김파일 포함)을 권한·소유자·크기·수정시각과 함께 나열.🔰 폴더 안에 있는 파일들을 숨김파일까지 전부, 자세한 정보와 함께 보여주는 명령이다.디스크에 어떤 파일이 있는지, 권한이 잘못 설정된 파일이 없는지 확인하는 가장 기본적인 첫 명령.

du -sh *현재 디렉토리 하위 각 항목의 용량을 사람이 읽기 쉬운 단위로 합산 표시.🔰 지금 폴더 안에서 각 항목이 용량을 얼마나 차지하는지 보여주는 명령이다.디스크 풀(full) 장애 시 "어느 디렉토리가 용량을 잡아먹는지" 좁혀갈 때 1순위 명령.
df -h마운트된 파일시스템별 전체/사용/가용 용량을 퍼센트와 함께 표시.🔰 디스크가 얼마나 찼는지 퍼센트로 보여주는 명령이다. 디스크 꽉 참 장애의 첫 확인 명령이다.디스크 풀 장애의 첫 확인 명령. "/ 파티션이 100%다" 같은 상황을 바로 포착.
tail -f /var/log/messages로그 파일의 끝부분을 실시간으로 계속 출력(follow 모드).🔰 로그 파일에 새로 찍히는 내용을 실시간으로 계속 보여주는 명령이다.서비스를 재시작하거나 배포하는 순간 실시간으로 로그를 지켜보며 문제를 즉시 포착할 때.
chmod 755 script.sh / chown user:group file파일 권한(rwx)과 소유자·그룹을 변경.🔰 파일을 누가 읽고 쓰고 실행할 수 있는지 권한을 바꾸는 명령이다.스크립트 실행 권한 부여, 잘못된 소유자로 생성된 파일 교정 등 배포·운영 중 가장 흔한 권한 문제 해결.
ln -s /data/app /opt/app심볼릭 링크 생성 — 원본을 옮기지 않고 다른 경로에서도 접근 가능하게 함.🔰 원본 파일을 옮기지 않고도 다른 경로에서 접근할 수 있는 '바로가기'를 만드는 명령이다.애플리케이션 배포 경로를 버전별 디렉토리와 분리해 무중단 배포(블루/그린)에 활용.

프로세스 / 리소스

명령어설명실무 활용 예시
ps aux --sort=-%cpu | head전체 프로세스를 CPU 사용률 기준 내림차순 정렬해 상위만 출력.🔰 CPU를 가장 많이 쓰는 프로세스 순서대로 보여주는 명령이다.서버 부하가 급증했을 때 어떤 프로세스가 CPU를 점유하는지 즉시 특정.
top / htopCPU·메모리·프로세스 상태를 실시간으로 갱신하며 보여주는 대화형 모니터.🔰 CPU·메모리·프로세스 상태를 실시간으로 계속 보여주는 모니터링 화면이다.장애 대응 시 가장 먼저 띄워놓고 상황을 지켜보는 필수 도구.
kill -9 <PID>지정한 프로세스를 강제 종료(SIGKILL, 정상 종료 절차 무시).🔰 말 안 듣는 프로세스를 강제로 즉시 종료시키는 명령이다.정상 종료(SIGTERM)로 안 죽는 좀비/행 프로세스를 강제로 정리할 때 최후 수단.
systemctl status/restart/enable <service>systemd 서비스의 상태 확인, 재시작, 부팅 시 자동시작 등록.🔰 서비스가 잘 돌고 있는지 확인하고, 재시작하고, 자동 시작을 켜주는 명령이다.서비스 장애 시 상태 확인 → 재시작이 표준 대응 절차의 시작점.
journalctl -u <service> -f특정 systemd 서비스의 로그만 실시간으로 스트리밍.🔰 특정 서비스 하나의 로그만 콕 집어서 실시간으로 보여주는 명령이다.서비스 하나의 문제만 콕 집어 로그를 봐야 할 때 잡음 없이 확인 가능.
free -h전체/사용/여유/캐시 메모리와 스왑 사용량을 사람이 읽기 쉬운 단위로 표시.🔰 메모리가 얼마나 남았는지, 스왑을 얼마나 쓰고 있는지 보여주는 명령이다.메모리 부족 의심 상황에서 실제 여유 메모리(available)와 스왑 사용 여부를 확인.
nice -n 10 <cmd> / renice -n 5 -p <PID>프로세스의 스케줄링 우선순위(niceness)를 낮추거나(양수) 높임(음수).🔰 특정 작업이 CPU를 덜 뺏어가게(또는 더 우선하게) 순위를 조절하는 명령이다.백업·배치 작업이 실서비스 프로세스의 CPU를 뺏지 않도록 우선순위를 낮춰 실행할 때.
crontab -e / crontab -l사용자별 정기 예약 작업(cron job)을 편집/조회.🔰 정해진 시간에 자동으로 실행될 작업(예약 작업)을 등록·확인하는 명령이다.백업, 로그 로테이션, 정기 점검 스크립트를 특정 시각에 자동 실행시킬 때.

명령어설명실무 활용 예시
ip a / ip route인터페이스별 IP 주소와 라우팅 테이블을 확인(구형 ifconfig/route 대체).🔰 이 서버의 IP 주소와 가는 길(라우팅 경로)을 확인하는 명령이다.네트워크 접속 안 될 때 IP/게이트웨이가 제대로 잡혀있는지 첫 확인.
ss -tuln현재 리스닝 중인 TCP/UDP 포트와 프로세스를 표시(구형 netstat 대체).🔰 지금 어떤 포트가 열려서 서비스를 기다리고 있는지 보여주는 명령이다.서비스가 실제로 예상 포트에서 대기 중인지, 다른 프로세스가 포트를 선점했는지 확인.
ping -c 4 <host>대상 호스트까지 ICMP 패킷을 보내 응답시간·손실률 확인.🔰 상대방 서버까지 신호를 보내서 살아있는지, 응답이 빠른지 확인하는 명령이다.연결 장애 시 가장 먼저 실행하는 기초 연결성 테스트.
traceroute <host> / mtr <host>목적지까지 거치는 각 라우팅 홉과 구간별 지연을 표시.🔰 목적지까지 가는 길에 거치는 각 구간마다 얼마나 걸리는지 보여주는 명령이다."어디서부터 느려지는지/끊기는지"를 구간별로 좁혀갈 때 사용.
curl -I https://hostHTTP 요청을 보내고 응답 헤더만 확인(-v로 상세 로그 확인 가능).🔰 웹사이트에 요청을 보내보고 응답이 오는지 확인하는 명령이다.웹서비스/API가 실제로 응답하는지, 상태 코드가 정상인지 빠르게 점검.
scp file user@host:/path / rsync -avz src/ dst/SSH 기반 파일 복사(scp)와 증분·압축 동기화(rsync).🔰 서버 사이에 파일을 안전하게 복사하거나 동기화하는 명령이다.서버 간 파일 배포, 백업 동기화의 표준 도구 — rsync는 변경분만 전송해 대용량에도 효율적.
ssh -p 22 user@host원격 서버에 암호화된 셸 세션으로 접속.🔰 원격 서버에 안전하게 접속하는 기본적인 명령이다.서버 관리의 가장 기본적인 원격 접속 수단.
tcpdump -i eth0 port 443지정 인터페이스에서 조건에 맞는 패킷을 실시간 캡처.🔰 실제로 오가는 패킷을 눈으로 직접 캡처해서 확인하는 명령이다.애플리케이션 레벨에서 원인을 못 찾을 때 패킷 레벨까지 내려가 실제 통신 여부를 확인.

사용자 / 권한 / 패키지

명령어설명실무 활용 예시
useradd -m -s /bin/bash user / passwd user사용자 계정 생성(홈 디렉토리 포함)과 비밀번호 설정.🔰 새 사용자 계정을 만들고 비밀번호를 설정하는 명령이다.신규 팀원·서비스 계정 온보딩 시 표준 절차.
usermod -aG sudo user기존 사용자를 특정 그룹(sudo 등)에 추가.🔰 이미 있는 사용자에게 특정 권한 그룹(예: sudo)을 추가로 부여하는 명령이다.일반 계정에 관리자 권한을 부여해야 할 때 root 계정 공유 없이 권한을 분배.
sudo -l현재 사용자가 sudo로 실행 가능한 명령 목록을 확인.🔰 지금 계정이 sudo로 뭘 할 수 있는지 확인하는 명령이다.권한 문제 트러블슈팅 시 "이 계정이 뭘 할 수 있는지" 확인하는 첫 단계.
yum install / dnf install / apt install배포판별 패키지 관리자로 소프트웨어를 설치.🔰 리눅스에서 소프트웨어를 설치하는 기본 명령이다.RHEL/CentOS 계열은 yum/dnf, Debian/Ubuntu 계열은 apt — 환경에 맞는 패키지 관리자 사용이 기본.
rpm -qa | grep <pkg> / dpkg -l | grep <pkg>설치된 패키지 목록에서 특정 패키지 버전을 확인.🔰 특정 소프트웨어가 설치돼 있는지, 어떤 버전인지 확인하는 명령이다.보안 취약점 공지가 뜨면 해당 패키지가 설치돼 있는지, 어떤 버전인지 가장 먼저 확인.
tar -czvf archive.tar.gz dir/ / tar -xzvf archive.tar.gz디렉토리를 압축 아카이브로 묶거나(-c) 풀기(-x), gzip 압축(-z) 사용.🔰 여러 파일을 하나로 압축하거나 다시 풀어주는 명령이다.백업 파일 생성, 소프트웨어 배포 패키지 압축/해제의 표준 방식.

디스크 / LVM / 파티션

명령어설명실무 활용 예시
lsblk -f블록 디바이스 트리 구조와 파일시스템 타입·UUID·마운트 위치를 함께 표시.🔰 이 서버에 어떤 디스크·파티션이 있는지 트리 구조로 보여주는 명령이다.새 디스크를 추가했을 때 파티션/파일시스템이 있는지, 어디에 마운트돼 있는지 한 번에 파악.
parted /dev/sdb mklabel gpt / mkpartGPT 파티션 테이블을 생성하고 새 파티션을 추가.🔰 디스크에 파티션(칸막이)을 새로 나누는 명령이다.2TB 이상 대용량 디스크를 파티셔닝할 때 표준 도구(fdisk는 MBR 위주).
mkfs.xfs /dev/sdb1 / mkfs.ext4 /dev/sdb1파티션에 파일시스템을 생성(포맷). RHEL 계열 기본은 XFS.🔰 나눠진 파티션을 실제로 쓸 수 있게 포맷하는 명령이다.새 디스크·볼륨을 서비스에 사용하기 전 반드시 거치는 초기화 단계.
mount /dev/sdb1 /data / umount /data파일시스템을 특정 경로에 연결(마운트)하거나 해제.🔰 포맷한 디스크를 실제로 쓸 폴더 경로에 연결(마운트)하거나 떼어내는 명령이다.스토리지 팀이 매핑한 볼륨을 실제 서비스 경로에 연결하는 마지막 단계 — /etc/fstab에 등록해야 재부팅 후에도 유지.
pvcreate /dev/sdb1 → vgcreate vg_data /dev/sdb1 → lvcreate -L 100G -n lv_data vg_data물리볼륨(PV) 생성 → 볼륨그룹(VG) 생성 → 논리볼륨(LV) 생성의 LVM 3단계.🔰 디스크를 나중에 유연하게 늘릴 수 있는 구조로 만드는 3단계 절차다.디스크를 나중에 유연하게 늘리고 줄일 수 있는 구조로 처음부터 설계할 때(대부분의 프로덕션 서버 표준 구성).
lvextend -L +50G /dev/vg_data/lv_data && xfs_growfs /data논리볼륨 용량을 늘린 뒤(lvextend), 그 위의 파일시스템도 함께 확장(XFS는 xfs_growfs, ext4는 resize2fs).🔰 서비스를 끄지 않고도 디스크 용량을 실시간으로 늘려주는 절차다.서비스 운영 중 디스크 공간이 부족할 때 다운타임 없이 온라인으로 용량을 늘리는 표준 절차.

설정명설명 / 동작 방식실무 활용언제 사용하는가
I/O 스케줄러 (none/mq-deadline/kyber/bfq)블록 디바이스에 들어오는 I/O 요청의 순서·병합 방식을 결정. NVMe/SSD는 자체적으로 큐 관리가 뛰어나 스케줄러 개입이 적은 none이 유리, 회전형 HDD는 탐색시간을 줄이는 mq-deadline이 흔히 사용됨.🔰 디스크로 가는 요청을 어떤 순서로 처리할지 정하는 정책이다. SSD와 HDD는 서로 다른 방식이 유리하다.높음디스크 타입(SSD/NVMe vs HDD)에 맞지 않는 스케줄러를 쓰면 불필요한 오버헤드가 생기므로 디스크 교체/증설 시 재점검.
마운트 옵션 noatime파일을 읽을 때마다 접근시각(atime)을 갱신하지 않도록 해 불필요한 메타데이터 쓰기를 줄임.🔰 파일을 읽을 때마다 '읽은 시각'을 굳이 기록하지 않게 해서 불필요한 쓰기를 줄이는 옵션이다.높음읽기 위주 워크로드(웹서버 정적 파일, 로그 저장소 등)에서 디스크 I/O를 줄이기 위한 표준적인 튜닝.
Bonding / Teaming — mode 1 (active-backup)한 NIC만 활성으로 사용하고 장애 시 나머지로 failover. 스위치 측 특별 설정(LACP 등) 없이도 동작.🔰 랜카드 하나만 쓰다가 문제가 생기면 다른 걸로 자동 전환하는 가장 간단한 이중화 방식이다.매우높음이중화가 목적이고 스위치 간(서로 다른 스위치) 이중화까지 고려할 때 가장 무난하고 안정적인 선택.
Bonding — mode 4 (802.3ad / LACP)여러 NIC을 하나의 논리 링크로 묶어 대역폭을 합산. 스위치 측 Port-Channel/LACP 설정과 반드시 짝을 맞춰야 함.🔰 랜카드 여러 개를 하나로 묶어서 속도까지 합치는 방식이다. 스위치 쪽 설정과 반드시 짝을 맞춰야 한다.높음NIC 이중화뿐 아니라 대역폭 자체를 늘려야 할 때(스토리지 트래픽, 대용량 백업 구간).
LVM (LVM Volume Group / Logical Volume, Thin Pool)물리 디스크를 볼륨그룹으로 추상화해 논리볼륨을 유연하게 늘리고 줄일 수 있게 함. Thin Pool은 스토리지 어레이의 씬 프로비저닝과 같은 개념을 OS 레벨에서 제공.🔰 디스크를 나중에 유연하게 늘리고 줄일 수 있도록 추상화해주는 스토리지 관리 계층이다.높음서비스 운영 중 디스크 확장이 잦은 환경에서 다운타임 없이 파티션을 재구성하고 싶을 때.

journald 로그 볼륨 제한 (SystemMaxUse 등)systemd 저널 로그가 디스크를 무한정 채우지 않도록 최대 사용량/보관기간을 제한.🔰 로그가 디스크를 무한정 채우지 않도록 최대 용량을 미리 제한해두는 설정이다.중간루트 파티션 용량이 작은 서버에서 로그 폭증으로 디스크가 가득 차 서비스 장애로 이어지는 사고를 예방할 때.
auditd파일 접근, 시스템콜, 권한 변경 등을 커널 레벨에서 감사(audit) 로그로 기록.🔰 누가 언제 어떤 파일에 접근했는지 감사 목적으로 기록해두는 기능이다.중간보안 컴플라이언스(PCI-DSS 등) 감사 대응, 침해사고 발생 시 행위 추적이 필요할 때.
SSH 하드닝 (PermitRootLogin no, 키 기반 인증)root 직접 로그인 차단, 비밀번호 인증 대신 공개키 인증 강제 등 SSH 접근을 강화.🔰 SSH 접속을 더 안전하게 만드는 기본 보안 조치들이다.매우높음모든 프로덕션 서버에서 기본적으로 적용해야 하는 초기 보안 체크리스트 항목.

❓ OS Interview Questions

DB 서버에서 swappiness를 낮추는 이유를 구체적으로 설명하면?
DB는 자체적으로 대용량 버퍼캐시(InnoDB Buffer Pool 등)를 메모리에 올려두고 성능을 낸다. 커널이 이 buffer pool의 일부를 스왑아웃시켜버리면 다음 쿼리에서 디스크 스왑 영역까지 갔다와야 해 순간적으로 응답시간이 수십~수백 배 튀는 지연 스파이크가 발생한다. swappiness를 0~10으로 낮추면 커널이 스왑보다 페이지 캐시 회수를 우선하도록 유도해 이런 스파이크를 최소화한다.
NIC 본딩 모드는 어떤 기준으로 고르나?
단순 이중화(서로 다른 스위치 연결 포함)가 목적이면 mode 1(active-backup) — 스위치 쪽 특별 설정이 필요 없어 안정적이다. 대역폭 합산까지 필요하면 mode 4(802.3ad/LACP)를 쓰되, 반드시 스위치 쪽에 대응하는 Port-Channel/LACP 설정이 되어 있어야 정상 동작한다.
시각 동기화(NTP/chrony)가 어긋나면 실제로 어떤 장애가 나는가?
분산 시스템에서 로그 순서를 시간 기준으로 비교할 수 없게 돼 장애 원인 분석이 꼬이고, Kerberos 인증처럼 시간차 허용범위(clock skew tolerance)가 있는 프로토콜은 인증 자체가 실패한다. 클러스터(예: etcd, Zookeeper)는 합의 알고리즘에서 타임스탬프 기반 판단이 어긋나 스플릿 브레인 등 심각한 장애로 이어질 수 있다.
서버가 다운됐다는 신고를 받으면 어떤 순서로 확인하나?
① racadm getsysinfo/gettracelog로 하드웨어 레벨(전원, POST 에러) 먼저 확인 → ② 정상이면 Virtual Console로 화면을 직접 보고 부팅 단계 확인 → ③ OS가 떴다면 journalctl -b -1, dmesg로 직전 세션 로그 확인 → ④ 디스크/네트워크/메모리 등 리소스 레벨 명령어로 세부 원인을 좁혀간다. 하드웨어(iDRAC/racadm)에서 OS 순으로 "바깥에서 안쪽으로" 좁혀가는 것이 표준 절차다.

📖 용어사전 모르는 단어 여기서 검색

위 탭들에서 당연하다는 듯 등장하는 약어·전문용어를 따로 모았다. NUMA, TPM, LACP처럼 "이게 뭔지부터 모르겠다" 싶은 단어가 나오면 여기서 검색해서 확인하자.

1 Hardware / BIOS Terms

NUMA

Non-Uniform Memory Access

CPU 소켓마다 "가까운(로컬)" 메모리와 "먼(원격)" 메모리가 나뉘어 있는 구조. 로컬 메모리 접근은 빠르고, 다른 소켓에 연결된 원격 메모리 접근은 상대적으로 느리다. 멀티소켓 서버 성능 튜닝의 핵심 개념 — DB나 가상화 프로세스를 특정 NUMA 노드에 고정(numactl)하면 원격 접근을 줄여 지연시간을 낮출 수 있다.

ECC Memory

Error-Correcting Code Memory

메모리에 저장된 데이터에 오류정정 코드를 추가로 저장해, 비트 반전 같은 오류를 감지하고 자동으로 정정하는 메모리. 서버는 안정성을 위해 거의 항상 ECC 메모리를 사용한다.

TPM

Trusted Platform Module

암호화 키를 안전하게 저장·생성하는 전용 보안 칩. 디스크 암호화(BitLocker 등)와 Secure Boot의 신뢰 기반이 된다.

ACPI

Advanced Configuration and Power Interface

하드웨어의 전원 상태(C-State, P-State 등)를 OS가 표준화된 방식으로 제어할 수 있게 하는 인터페이스 규격.

C-State

CPU Idle State

CPU가 유휴 상태일 때 들어가는 저전력 모드 단계(C0=활성, C1~C6로 갈수록 더 깊은 절전). 깊이 들어갈수록 전력은 아끼지만 다시 깨어나는 데 시간(latency)이 더 걸린다.

Turbo Boost

Intel Turbo Boost Technology

발열·전력 여유가 있을 때 CPU 코어 클럭을 정격 이상으로 자동으로 끌어올리는 기능.

Hyper-Threading (SMT)

Simultaneous Multi-Threading

물리 코어 1개를 OS에는 논리 코어 2개처럼 보이게 해 스레드 병렬성을 높이는 기술. Intel은 Hyper-Threading, AMD는 SMT라 부른다.

UEFI

Unified Extensible Firmware Interface

레거시 BIOS를 대체하는 최신 펌웨어 인터페이스. GPT 파티션, Secure Boot, 빠른 부팅을 지원한다.

POST

Power-On Self Test

전원이 켜진 직후 CPU·메모리·디스크 등 하드웨어가 정상인지 자체 점검하는 부팅 초기 단계.

DIMM / Memory Channel

Dual In-line Memory Module

메모리 모듈 자체를 DIMM이라 부르며, CPU-메모리 컨트롤러 간 연결 통로를 채널이라 한다. 채널을 균등하게 채우지 않으면(populate) 메모리 대역폭이 저하된다.

TDP

Thermal Design Power

CPU가 정상 동작 시 방출하는 최대 열량(전력) 설계치. 냉각 시스템 설계와 전력 예산 산정의 기준이 된다.

VT-d / IOMMU

Virtualization Technology for Directed I/O

물리 디바이스를 가상머신에 직접 할당(패스스루)할 수 있게 하는 하드웨어 기능. 디바이스의 DMA 접근을 가상주소로 안전하게 리매핑한다.

SR-IOV

Single Root I/O Virtualization

물리 NIC/디바이스 1개를 여러 개의 가상 기능(Virtual Function)으로 쪼개 여러 VM에 직접 할당하는 기술. 하이퍼바이저의 가상 네트워크 스위치를 거치지 않아 지연시간이 낮다.

PXE

Preboot Execution Environment

디스크 없이도 네트워크를 통해 OS 이미지를 내려받아 부팅하는 방식. 대량 서버 자동 설치(프로비저닝)에 사용.

GPT vs MBR

GUID Partition Table / Master Boot Record

디스크 파티션 정보를 저장하는 두 방식. MBR은 2TB·4개 파티션 한계가 있는 구형 방식, GPT는 대용량 디스크와 UEFI 부팅을 지원하는 최신 방식.

DMA

Direct Memory Access

CPU를 거치지 않고 주변장치가 메모리에 직접 데이터를 읽고 쓰는 기술. 성능은 좋지만 인증되지 않은 외부 장치가 이를 악용하면 메모리를 직접 탈취당할 수 있어 Kernel DMA Protection 같은 방어 기능이 필요하다.

ASPM

Active State Power Management

PCIe 링크가 유휴 상태일 때 저전력 상태(L0s/L1)로 들어가는 절전 기능. 전력은 아끼지만 일부 카드에서 복귀 지연·호환성 문제를 일으켜 트러블슈팅 시 끄고 테스트하는 대표적 옵션.

Watchdog Timer

Watchdog Timer

정해진 시간 안에 특정 신호(heartbeat)가 오지 않으면 시스템이 멈췄다고 판단해 자동으로 재시작·전환시키는 감시 타이머. OS Watchdog, Redundant OS Watchdog 등 여러 계층에서 쓰인다.

SED

Self-Encrypting Drive

디스크 자체 컨트롤러가 하드웨어 레벨에서 모든 데이터를 자동 암호화하는 드라이브. 디스크를 분리해가도 암호 키 없이는 데이터를 읽을 수 없어 폐기·교체 시 별도 소거 없이도 안전하다.

2 RAID / Storage Terms

RAID

Redundant Array of Independent Disks

여러 물리 디스크를 하나의 논리 볼륨으로 묶어 성능(스트라이핑)이나 안정성(미러링/패리티)을 높이는 기술. 레벨(0/1/5/6/10 등)에 따라 특성이 다르다.

Parity

Parity

원본 데이터로부터 계산해낸 검증용 값. RAID5/6에서 디스크 하나(또는 둘)가 죽어도 나머지 데이터와 패리티로 원본을 복원할 수 있게 해준다.

Write-Back / Write-Through

Write Cache Policy

Write-Back은 컨트롤러 캐시에 데이터가 들어오면 즉시 완료 응답(빠름, 정전 시 유실 위험), Write-Through는 디스크에 실제로 써진 후에만 완료 응답(느림, 안전).

BBU / CacheVault

Battery Backup Unit / Cache Vault

정전 시에도 RAID 컨트롤러 캐시의 데이터를 보존해주는 배터리(BBU) 또는 커패시터+플래시(CacheVault) 보호 장치. Write-Back 캐시를 안전하게 쓰기 위한 전제조건.

Patrol Read

Background Disk Scan

유휴 시간에 백그라운드로 전체 디스크를 스캔해 배드섹터를 조기에 발견·복구하는 기능.

Hot Spare

Spare Disk

RAID 구성 디스크 중 하나가 고장 나면 사람 개입 없이 자동으로 투입되는 대기 디스크.

Thin Provisioning

Thin Provisioning

논리적으로는 큰 용량의 볼륨을 만들어주지만, 물리적으로는 실제 사용한 만큼만 공간을 할당하는 방식.

Redirect-on-Write

Redirect-on-Write Snapshot

스냅샷을 찍을 때 원본 데이터는 그대로 두고, 새로 쓰이는 데이터만 다른 위치로 리다이렉트해 저장하는 방식. Copy-on-Write보다 쓰기 오버헤드가 적다.

LUN

Logical Unit Number

스토리지가 서버에게 노출하는 논리적인 디스크 단위 번호. 서버 OS 입장에서는 LUN 하나가 물리 디스크 하나처럼 보인다.

MPIO

Multipath I/O

서버-스토리지 사이의 여러 물리 경로를 하나의 논리 경로로 묶어 이중화·부하분산하는 기술.

HBA

Host Bus Adapter

서버가 SAN 스토리지(FC, SAS 등)에 연결하기 위해 장착하는 전용 어댑터 카드.

iSCSI / FC / SAS

Storage Connectivity Protocol

iSCSI는 일반 이더넷 위에서 SCSI 명령을 전달(저비용), FC(Fibre Channel)는 전용 광케이블 기반 SAN(고성능·고비용), SAS는 서버-스토리지 간 직결에 주로 사용된다.

JBOD

Just a Bunch Of Disks

디스크를 RAID로 묶지 않고 개별 디스크 그대로 OS에 노출하는 모드. ZFS, Ceph 등 소프트웨어 정의 스토리지가 디스크를 직접 관리해야 할 때 사용.

Consistency Check

Consistency Check

패리티 기반 RAID(5/6 등)에서 실제 데이터와 패리티 값이 일치하는지 정기적으로 검증·복구하는 유지보수 작업. 사일런트 데이터 손상을 예방한다.

Foreign Configuration

Foreign Configuration Import

다른 서버·컨트롤러에서 쓰던 디스크를 그대로 옮겨 장착했을 때, 그 디스크에 남아있는 기존 RAID 구성 정보를 새 컨트롤러가 인식하는 상태. Import하면 볼륨을 그대로 복원할 수 있다.

3 iDRAC / Remote Management Terms

BMC

Baseboard Management Controller

메인보드에 내장된 독립 관리 프로세서의 일반 명칭. Dell에서는 이를 iDRAC이라는 브랜드로 부른다.

iDRAC

Integrated Dell Remote Access Controller

Dell 서버 전용 BMC. 서버 전원과 무관하게 동작하며 원격 콘솔·전원제어·모니터링을 제공.

Redfish

DMTF Redfish API

REST/JSON 기반의 최신 서버관리 표준 API. IPMI를 대체하는 업계 표준으로 자리잡고 있다.

IPMI

Intelligent Platform Management Interface

Redfish 이전 세대의 서버 하드웨어 관리 표준 프로토콜. 기능이 제한적이고 구형 툴체인에서만 주로 사용.

RACADM

Remote Access Controller ADMin

iDRAC 설정을 CLI/스크립트로 다루는 Dell 전용 명령행 도구.

Lifecycle Controller

LC

OS 없이도 BIOS/펌웨어 업데이트, 하드웨어 진단, RAID 구성을 할 수 있게 해주는 Dell의 내장 관리 엔진.

SEL

System Event Log

하드웨어 이벤트(디스크 오류, 전원 이상, 온도 경고 등)를 시간순으로 기록하는 로그. 장애 분석의 1차 확인 지점.

SNMP

Simple Network Management Protocol

네트워크 장비·서버의 상태를 조회(GET)하거나 이벤트를 알림(Trap)으로 전달하는 표준 관리 프로토콜. iDRAC/스위치/스토리지 모두 SNMP Trap을 모니터링 시스템 연동에 흔히 사용한다.

Syslog

System Logging Protocol

장비의 로그 메시지를 표준화된 형식으로 원격 로그 서버에 전송하는 프로토콜. iDRAC, 스위치, Linux 서버 모두 syslog로 중앙 로그 수집을 구성할 수 있다.

LDAP / AD

Lightweight Directory Access Protocol / Active Directory

중앙 디렉토리 서비스에 사용자 인증을 위임하는 프로토콜(LDAP)과 그 대표 구현체(AD). iDRAC 계정을 로컬로 일일이 만들지 않고 회사 AD 계정으로 로그인하게 할 때 사용.

Firmware Baseline

Firmware Baseline

OME에서 특정 서버군이 유지해야 할 "기준 펌웨어 버전"을 정의한 것. 실제 버전과 비교해 Compliance Report를 만들고, 필요 시 업데이트를 수행한다.

4 Network Terms

VLAN

Virtual LAN

물리적으로 하나인 네트워크를 논리적으로 여러 개의 독립된 네트워크로 분리하는 기술.

LACP / Port-Channel

Link Aggregation Control Protocol

여러 물리 포트를 하나의 논리 링크로 묶어 대역폭 합산과 이중화를 동시에 제공하는 표준(802.3ad) 프로토콜.

VLT / MLAG

Virtual Link Trunking / Multi-Chassis LAG

물리적으로 분리된 스위치 2대를 논리적으로 하나처럼 동작시켜, 스위치 자체 장애까지 이중화하는 기술. Dell은 VLT, 업계 일반 용어는 MLAG.

STP / RSTP

Spanning Tree Protocol

이중화 네트워크에서 루프가 생기지 않도록 특정 경로를 논리적으로 차단하는 프로토콜.

QoS / DCB / PFC

Quality of Service / Data Center Bridging / Priority Flow Control

트래픽 종류별로 우선순위나 대역폭을 차등 적용하는 기술. PFC는 특정 트래픽 클래스만 선택적으로 흐름제어해 무손실 전송을 보장(스토리지 트래픽용).

MTU / Jumbo Frame

Maximum Transmission Unit

이더넷 프레임의 최대 크기. 기본 1500바이트보다 큰 9000바이트 프레임을 Jumbo Frame이라 하며, 대용량 전송의 오버헤드를 줄인다.

OOB Management Network

Out-of-Band Management

실제 서비스 트래픽과 물리적/논리적으로 완전히 분리된 별도의 관리 전용 네트워크 경로.

NIC Bonding / Teaming

Network Interface Bonding

여러 물리 NIC을 논리적으로 하나로 묶어 이중화(active-backup)하거나 대역폭을 합산(802.3ad/LACP)하는 OS 레벨 기술. 반드시 스위치 쪽 Port-Channel 설정과 짝을 맞춰야 한다.

ACL

Access Control List

IP·포트·프로토콜 조건으로 트래픽을 허용하거나 차단하는 규칙 목록. 스위치, 방화벽, 라우터 등 거의 모든 네트워크 장비에서 접근 통제의 기본 수단.

Storm Control

Broadcast/Multicast Storm Control

브로드캐스트·멀티캐스트 트래픽이 비정상적으로 폭증할 때 해당 포트의 트래픽을 자동으로 제한해 네트워크 전체 마비를 막는 보호 기능.

5 OS / Kernel Terms

sysctl

System Control

Linux 커널 파라미터를 런타임에 조회·변경하는 인터페이스(/proc/sys 기반).

swappiness

Swap Tendency Value

메모리 여유가 있어도 커널이 얼마나 적극적으로 스왑을 쓸지 결정하는 0~100 값. 낮을수록 스왑을 미룬다.

HugePages

Large Memory Pages

기본 4KB 대신 2MB/1GB 단위의 큰 메모리 페이지를 사용해 TLB 미스와 관리 오버헤드를 줄이는 기법.

cgroups

Control Groups

프로세스 그룹별로 CPU·메모리·I/O 자원 사용량을 제한·격리하는 커널 기능. 컨테이너 기술의 근간.

systemd

System and Service Manager

대부분의 최신 Linux 배포판이 사용하는 초기화 및 서비스 관리 데몬. 부팅 프로세스, 서비스 상태, 로그(journald)를 관리.

SELinux

Security-Enhanced Linux

프로세스별로 세밀한 강제접근제어(MAC)를 적용하는 커널 보안 모듈.

ulimit

User Limit

사용자/프로세스가 사용할 수 있는 파일 디스크립터, 프로세스 수 등 리소스의 상한을 지정하는 셸/커널 기능.

I/O Scheduler

I/O Scheduler

블록 디바이스에 들어오는 입출력 요청의 처리 순서·병합 방식을 결정하는 커널 계층(none/mq-deadline/bfq 등).

LVM

Logical Volume Manager

물리 디스크를 볼륨그룹으로 추상화해 논리볼륨을 유연하게 늘리고 줄일 수 있게 하는 스토리지 관리 계층.

chrony / NTP

Time Synchronization Protocol

서버의 시각을 표준시각 서버와 동기화하는 프로토콜/데몬. 최근 Linux 배포판은 chrony가 기본.

multipathd

Device Mapper Multipath Daemon

SAN 스토리지로 가는 여러 물리 경로를 하나의 논리 디바이스로 묶어 관리하는 Linux 데몬. 외장 스토리지에 다중 경로로 연결된 서버에서는 필수.

kdump

Kernel Crash Dump

커널 패닉 발생 시 예약된 메모리 영역에서 두 번째 커널을 띄워 크래시 시점의 메모리 덤프(vmcore)를 저장하는 기능. 원인불명 커널 패닉 분석의 핵심 도구.

auditd

Linux Audit Daemon

파일 접근, 시스템콜, 권한 변경 등을 커널 레벨에서 감사(audit) 로그로 남기는 데몬. 보안 컴플라이언스 감사 대응에 사용.

6 17th Gen (17G) New Technology Terms

MRDIMM

Multiplexed Rank DIMM

기존 RDIMM보다 랭크를 멀티플렉싱해 같은 핀 수로 더 높은 대역폭(최대 8,800 MT/s)을 내는 신형 DDR5 메모리. Intel Xeon 6(Granite Rapids) 플랫폼부터 지원.

CXL

Compute Express Link

CPU와 메모리/가속기 사이를 PCIe 물리 계층 위에서 캐시 일관성 있게 연결하는 차세대 인터커넥트 표준. CXL 2.0부터는 DIMM 슬롯 한계를 넘는 메모리 확장(메모리 풀링)이 가능해진다.

Intel TDX

Trust Domain Extensions

VM 단위로 메모리를 하드웨어 레벨에서 암호화·격리하는 기밀 컴퓨팅 기능. 하이퍼바이저 관리자조차 VM 내부 메모리를 들여다볼 수 없게 한다.

iDRAC10

New-Gen BMC for 17G

PowerEdge 17G부터 탑재되는 차세대 관리 컨트롤러. TPM·Root-of-Trust·원격 증명·암호화를 통합해 Zero Trust 보안 모델을 지원한다.

PERC12

New RAID Controller Generation for 17G

H965i(Front/Adapter/MX), H965e 등이 속한 PERC 세대. PCIe 5.0과 더 높은 IOPS를 지원하지만 WT/WB/FWB 같은 캐시 정책 개념은 이전 세대와 동일하다.

Industry-Standard NIC

Industry-Standard Network Card

17G부터 Dell 서버에 장착되는 NIC이 벤더(Broadcom/Intel/NVIDIA 등)의 네이티브 펌웨어·드라이버를 그대로 사용하도록 전환된 방식. 이전엔 Dell 전용으로 커스터마이징된 펌웨어를 사용했다.
